## 针对的问题（Motivation）

RAG（检索增强生成）是企业知识 QA 的主流范式，但存在一个根本性结构缺陷：**LLM 永远只是搜索结果的被动消费者，它从不知道语料库是怎么组织的，也不知道自己还没检索到什么**。

具体表现：
1. **Flat Retrieval（BM25/Dense）**：返回 top-k 相似段落，LLM 收到的是孤立碎片，无法推断语料结构、判断证据是否完整，也无法系统性地往更相关的区域补充检索。
2. **Agentic RAG**：允许 LLM 多轮发起搜索，但每次仍是"黑盒搜索"——agent 没有语料全景图，等于在黑暗中靠猜词来打探情报。
3. **RAPTOR / GraphRAG 等层次方法**：离线构建了文档树/图，但查询时仍用 embedding 相似度检索，层次结构对 LLM 不可见——模型收到的还是搜索结果，而非结构性视角。

核心 insight：**应该把层次结构本身暴露给 LLM**，让 agent 主动"导航"而非被动"接收"。

![Figure 1: Retrieve vs. Navigate 对比](https://arxiv.org/html/2604.14572v1/x1.png)

*Figure 1: 传统 RAG（左）被动喂给 LLM 固定段落；Corpus2Skill（右）将语料蒸馏为可导航的 skill 层次，agent 主动探索、回溯、深钻，像浏览文件系统一样定位证据。*

---

## 核心思路（一句话）

Corpus2Skill 将文档语料库**离线编译**为可浏览的层次 skill 目录树，serving 时让 LLM agent 像翻文件系统一样**主动导航**这棵树，从鸟瞰图出发，逐层下钻直至找到相关文档，彻底替代基于向量索引的被动检索。

## 方法（Framework）

![Figure 2: Corpus2Skill 系统架构](https://arxiv.org/html/2604.14572v1/x2.png)

*Figure 2: Corpus2Skill 系统架构。编译阶段（上）：对文档 embedding → 多级 K-Means 聚类 → LLM 生成摘要，物化为 skill 目录树（根节点 SKILL.md，子节点 INDEX.md）。Serving 阶段（下）：agent 持有预编译 skill 目录 + 两个工具（file navigation / get_document），主动导航定位证据。*

Corpus2Skill 分**编译阶段**和 **serving 阶段**两步：

### 3.1 编译阶段（Compile Phase）

**第一步：文档载入与 Embedding**

- 接受 `.md/.txt/.json/.jsonl` 格式文档，用基于内容哈希的唯一 ID 标识每份文档（确定性、无碰撞）
- 使用句子嵌入模型（Qwen/Qwen3-Embedding-0.6B，1024 维）将每篇文档 embedding 为稠密向量

**第二步：迭代层次聚类**

用两个参数控制：分支比 $p$（每个簇的目标子簇数）和最大顶层簇数 $K$。

迭代过程：

> 当前集合 $n$ 个 embedding → K-Means 分为 $\lceil n/p \rceil$ 组 → 每组用 LLM 生成摘要 → 摘要 re-embedding → 再次 K-Means ... 直到簇数 $\leq K$

层次深度 $O(\log_p N)$：N=6221，p=10 时仅 3 层，约 30 步摘要遍历即可定位单篇文档：

$$N \xrightarrow{\div p} \frac{N}{p} \xrightarrow{\div p} \frac{N}{p^2} \cdots \xrightarrow{\div p} \frac{N}{p^L} \approx 1$$

**第三步：标签生成**

LLM 为每个非叶节点生成 2~5 词的人类可读、文件系统安全标签（例：`wix-commerce-monetization`、`wix-payments-ecosystem`），供 agent 在 routing 时仅凭标签即可判断方向。

**第四步：Skill 树物化**

层次结构物化为文件系统结构：

| 节点 | 文件 | 内容 |
|------|------|------|
| 根簇（顶层技能） | `SKILL.md` | 簇摘要 + 子组列表（通常 <2KB） |
| 子簇（中间层） | `INDEX.md` | 子组列表 or 叶文档 ID + 简短摘要 |
| 叶文档（完整内容） | `documents.json` | 通过 doc_id 检索，与导航文件分离存储 |

WixQA 上的默认配置（p=10, K=7）产出：**6 个顶层 skill**、**665 个导航文件**、**13 MB 文档存储**，编译耗时 6.5 分钟（32-CPU 服务器）。

**核心设计理念 —— 从过程型 Skill 到信息型 Skill：**

Corpus2Skill 把相同的 progressive disclosure 机制（描述 → SKILL.md → INDEX.md → 文档）从"如何执行任务的流程知识"重新诠释为"语料库包含什么内容的信息知识"——这是本文的中心设计洞见。

### 3.2 Serving 阶段

Agent 接收两个工具：
1. `code_execution`：浏览 SKILL.md 和 INDEX.md（导航工具）
2. `get_document(doc_id)`：检索完整文档内容（取证工具）

**典型查询流程（2~3 轮）**：

```
Step 1: 读 SKILL.md → 理解顶层主题分布，选择最相关 skill
Step 2: 读相关子组的 INDEX.md → 查看文档 ID 和摘要标题
Step 3: 调用 get_document(doc_id) → 获取完整内容，合成答案
```

Progressive disclosure 机制：skill 名称和单行描述（约 200 tokens）预加载到 context，完整 SKILL.md / INDEX.md 按需加载。Agent 先"鸟瞰"全局，再逐步下钻，每一步都由摘要引导，知道哪些分支还没探索——flat retrieval 无法提供这种结构性视角。

## 关键设计细节

### Skill 文件格式

每个 SKILL.md 和 INDEX.md 带有 YAML frontmatter（name, description, level, num_documents），用于 Skills API 的 progressive disclosure：

```yaml
---
name: skill-01-wix-commerce-monetization
description: >
  Full Wix commerce and monetization
  ecosystem, from storefronts to payments.
level: 3
num_documents: 1513
---
```

摘要写法面向"路由"：LLM 在生成时被要求明确 topic 域、可回答的问题类型、关键术语，使每个文件都成为 agent 的决策点。文件末尾还内嵌下一步操作指引（如"Use get_document with the doc_id"）。

### 两种导航模式

![Figure 4 & Figure 5: 直接导航 vs 跨分支导航](https://arxiv.org/html/2604.14572v1/x4.png)

*Figure 4: Trace 1 直接导航——agent 沿单一 skill 分支，4 步找到目标文档（"Changing Your Wix Payments Account Type"）并作答。*

1. **直接下钻（Direct Descent）**：skill → subgroup → leaf → document，4 步完成
2. **跨分支探索（Cross-branch Exploration）**：查询涉及多主题时，agent 访问同一 skill 下多个子组，合并来自不同分支的证据

### Compact Mode（文件合并压缩）

当层次较深或语料较大时，叶层 INDEX.md 文件数可能超出 Skills API 的 200 文件/skill 限制。Compact mode 将近叶节点的 INDEX.md 内容合并内联到父节点，减少文件数最多 80%，同时保留所有导航信息。

## 实验与效果

### 实验设置

- **数据集**：WixQA（Cohen et al., 2025）— 企业客服 RAG benchmark，语料库 6,221 篇 Wix 支持文章，评测集 200 道专家撰写的问题 + 标准答案 + 标准文章 ID
- **Baseline**（5 个，横跨 3 种检索范式）：
  - BM25（稀疏关键词检索）
  - Dense（Qwen3-Embedding-0.6B + FAISS 索引，top-5 段落）
  - Hybrid（BM25 + Dense 的 RRF 融合）
  - RAPTOR（UMAP+GMM 层次树检索）
  - Agentic（LLM agent + BM25/Dense/Hybrid 工具，最多 10 轮交互）
- **评估指标**：
  - 词法：Token F1 / BLEU / ROUGE-1 / ROUGE-2
  - LLM 评判：Factuality（1~5 归一化到 0~1）/ Context Recall（同上）
  - 成本：输入 tokens / 每查询美元成本

### 主结果表（Table 1，WixQA 200 道题）

| 方法 | Token F1↑ | BLEU↑ | ROUGE-1↑ | ROUGE-2↑ | Factuality↑ | Context Recall↑ | 每查询成本↓ |
|------|:---------:|:-----:|:--------:|:--------:|:-----------:|:---------------:|:-----------:|
| BM25 | — | — | — | — | 0.470 | 0.386 | — |
| Dense | — | — | — | — | 0.536 | 0.450 | — |
| Hybrid | — | — | — | — | — | — | — |
| RAPTOR | — | — | — | — | 0.675 | 0.616 | ~$0.012 |
| Agentic | 0.388 | — | — | — | 0.724 | 0.481 | ~$0.097 |
| **Corpus2Skill** | **0.460** | — | — | — | **0.729** | **0.652** | $0.172 |

**关键数字摘要**：
- Token F1：Corpus2Skill 0.460，相比 Agentic 0.388 提升 **+19% 相对提升**，相比 Dense 0.363 提升 **+27%**
- Factuality：Corpus2Skill 0.729，超越 Agentic 0.724 和 RAPTOR 0.675
- Context Recall：Corpus2Skill 0.652，明显领先 RAPTOR 0.616 和 Agentic 0.481（高出近 36%），说明导航式方法能取回更多相关内容
- 每查询成本 $0.172，约为 Agentic 的 1.75 倍、RAPTOR 的 14 倍（因 Skills API 每轮都把导航文件内容载入 context）
- 每次查询消耗约 53K 输入 tokens，但仅输出 752 tokens（Agentic 为 1,391），提示导航到正确文档后答案更简洁准确

## 消融与分析

### 簇结构消融（Table 2）

| 配置 | F1↑ | Factuality↑ | Context Recall↑ | 成本/查询↓ |
|------|:---:|:-----------:|:---------------:|:----------:|
| 默认（p=10, K=7，3层） | 0.460 | 0.729 | 0.652 | $0.172 |
| 窄树（p=5，4层，compact） | **0.461** | **0.736** | **0.674** | $0.186 |
| 宽树（p=20，2层） | 0.363 | 0.409 | — | $0.243 |

- **窄树**（p=5）略优于默认（F1 +0.2%，Factuality +1%，Context Recall +3.4%），成本增加 8%。更细粒度的分组 + compact 合并后的密集叶节点视图给 agent 更精准的路由点
- **宽树**（p=20）质量急剧下滑：F1 下降 21%，Factuality 下降 44%，成本反而高 41%。仅 2 个极宽技能、每个覆盖约 3000 文档，SKILL.md 摘要太泛，agent 无法有效 routing

### Agent 探索预算消融（Table 3）

- 预算从 5 轮到 15 轮，质量变化极小：5 轮时 F1=0.453（仅比全预算低 1.5%），Factuality 在 0.721~0.748 之间浮动
- 说明层次组织足够良好，agent 很少需要大量探索轮次——预算不是性能瓶颈

### Serving 模型选择消融（Table 4）

- 换用更便宜的模型：成本减半（$0.088 vs $0.172），F1 降 8%，Factuality 降 12%，但 **Context Recall 反而从 0.652 提升到 0.705**
- 启示：更便宜的模型在 context 内导航更彻底（取回更多相关文档），尽管答案合成稍欠精炼
- 核心驱动力是**编译好的 skill 树质量**，而非 serving 模型的复杂度

## 结论 / 局限 / 启发

### 核心结论

Corpus2Skill 证明：给 LLM agent 一张结构化的语料"地图"（hierarchical skill tree），比给它检索段落获得更高质量答案。Agent 本身就是能干的"导航员"，只要给对工具和地图——关键是**给对地图**。

### 局限性

1. **每查询成本高**：Skills API 每轮都把导航文件内容载入 context，53K 输入 tokens/查询（约 Agentic 的 1.75 倍），不适合低价值高频查询；prompt caching 可大幅缓解
2. **依赖 Anthropic Skills API**：当前受限于 8 个 skill/请求、每 skill 200 文件、30 MB 文件大小，大型语料需要 compact mode 或多级分区
3. **硬单路径聚类**：每篇文档只属于一个聚类路径，跨主题文档（如"账单 + 订阅"）只能归入一个分支，可能在另一路径查询时被遗漏；失败分析确认顶层 routing 错误是主要失败原因
4. **不支持增量更新**：目前是批量编译，新增文档需重新编译整个树
5. **编译成本**：\$5~\$10 LLM API 调用 + 6.5 分钟，对日处理数千查询的企业系统可忽略不计

### 对检索/RAG 方向的启发

1. **Navigability（可导航性）是 RAG 的一个被低估维度**：投入编译期算力让语料可导航，serving 期就不再需要 embedding 索引
2. **Progressive Disclosure 是 token 效率的关键**：仅 200 tokens 的顶层概览让 agent 做出有依据的第一跳，而非黑盒猜测
3. **层次质量决定系统上限**：消融证明服务模型可以降级但层次不能太宽——宽树是性能最大杀手
4. **compile-then-navigate 与 retrieve 互补而非替代**：适合高价值复杂查询（多跳、跨主题），简单高频查询 BM25/Dense 仍有性价比优势
5. **与 RAPTOR 的本质区别**：同样离线聚类 + 摘要，但 Corpus2Skill 把层次物化为 agent 可直接浏览的 skill 文件，而非再次放入 flat vector store——这一步让 LLM 真正"看见"语料结构
