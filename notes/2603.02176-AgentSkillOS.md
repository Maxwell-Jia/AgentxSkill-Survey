## 针对的问题
### 痛点
Agent Skill 生态正急速膨胀（2026年2月已逾28万个），但规模增长带来了三重挑战：
1. **发现困难**：用户难以从数万甚至数十万个 Skill 中找到相关的、合适的工具，缺乏全局视野
2. **组合困境**：LLM Agent 缺乏显式机制来协调多个 Skill 的依赖关系、数据流、执行顺序，导致大量 Skill 被孤立使用
3. **治理成本**：分散的、第三方贡献的 Skill 生态带来质量、安全、可靠性的治理难题

### 核心差距
现有的"平铺式"Skill 调用机制（native flat invocation）本质上无法应对生态规模化：即使提供完整的 Skill 库，Agent 仍然因为信息超载而无法有效利用，更无法进行跨多个 Skill 的结构化编排。

---

## 核心思路
> 两阶段框架：用**层级能力树**组织生态，用**DAG 编排**管理执行

AgentSkillOS 分为两个离线/在线协作的阶段：
1. **Manage Skills**（离线）：构建"能力树"，将海量 Skill 递归地组织成层级分类，支持 coarse-to-fine 的树形检索
2. **Solve Tasks**（在线）：基于能力树进行任务驱动的 Skill 检索、DAG 编排、多 Skill 执行

关键洞察：**结构化组合**而非单纯的 Skill 可用性才是解锁生态潜力的关键；树形检索 + DAG 编排的组合拳才能战胜 flat invocation 的先天局限。

## 方法

### 整体架构
![Figure 1: AgentSkillOS 系统框架](https://arxiv.org/html/2603.02176v1/x1.png)
**Figure 1**: 两阶段框架：Manage Skills 阶段离线构建能力树；Solve Tasks 阶段通过任务驱动的检索、DAG 编排和多 Skill 执行完成任务。

### Skill 的层级组织：能力树构建

**核心思想**：将所有 Skill 递归地分组，形成一棵树。根节点包含所有 Skill，每个非叶节点代表一个"能力类别"，其下的子节点是更细粒度的子类别，最终叶节点对应单个 Skill。

**构造算法**（基于 node-level recursive categorization）：
1. **Group Discovery**：LLM 基于当前节点的 Skill 集合，生成 $B$（branching factor，通常为7-12）个候选类别及其名称和描述
2. **Skill Assignment**：LLM 将每个 Skill 分配到最合适的类别
3. **特殊处理**：
   - 根节点固定为5个大类：content creation, data processing, software development, automation, domain-specific
   - 只含1个Skill的类别被合并到相关的较大类别
   - 小于容量阈值 $C=\lfloor 1.5B \rfloor$ 的类别子节点转为叶节点（不再递归）
4. **动态更新**：新增 Skill 时自顶向下逐层分配，沿路更新每个节点的类别名称和描述

**容量控制**（Usage-Frequency Queue）：当生态规模过大（如200K Skill）时，只选择最高 $K$ 个高频使用 Skill（按装机量）构造树，其余放入"休眠索引"，支持向量相似度检索补充。

### DAG 编排机制

**任务驱动的 Skill 检索**（三步）：
1. **Tree-guided Retrieval**：LLM 遍历能力树，逐层选择相关的类别和 Skill，生成候选集合
2. **Pruning**：LLM 去重、排序、只保留 top-$M$ 个（通常 $M=8$）相关 Skill
3. **用户确认**：用户可在此基础上添加自定义 Skill

**DAG 编排的三种策略**（每种对应不同的任务分解方式）：
- **Quality-First**：最大化每个可交付物的质量，添加准备和精化阶段 → 深且密的 DAG
- **Efficiency-First**：最大化并行度，减少顺序依赖 → 宽且浅的 DAG  
- **Simplicity-First**：最小的本质 DAG，每个节点都必要 → 紧凑的 DAG

给定最终的 Skill 集合 $V$ 和选定的编排策略，LLM 生成相应的 DAG $G=(V,E)$，其中有向边表示 Skill 间的依赖关系（输入输出流）。

**多 Skill 执行**：
- 同层节点无依赖可并行执行
- 不同层节点按拓扑顺序执行
- 每个节点的执行提示包括：任务描述、子任务说明、上游产物列表、预期输出
- 所有生成的文件保存为"制品"供下游使用

### Benchmark 设计

构造了30项任务，分5个类别，每类6项：
- **Data Computation**：数值分析、统计建模
- **Document Creation**：专业格式文档、演示文稿
- **Motion Video**：教学动画、技术可视化
- **Visual Design**：静态视觉设计物件
- **Web Interaction**：交互网页、网页爬虫

**评估方案**（Bradley-Terry 配对对比）：
1. 将任意两个系统的输出转换为 LLM 可理解的格式（文档→截图、视频→帧+元数据、图像→标准分辨率）
2. LLM judge 两个方向都进行配对对比以消除位置偏差；若两方向一致，取偏好；若冲突，记为平局
3. 汇聚所有配对结果入"赢矩阵" $W_{ij}$（系统 $i$ 胜过系统 $j$ 的次数）
4. 拟合 Bradley-Terry 模型得到潜在强度参数 $\beta_i$，再线性缩放到 [0,100] 得最终分数

## 关键设计细节

1. **能力树 vs 纯语义检索**：树形结构迫使 LLM 在每一层做出分类决策，从而发现非显而易见但功能相关的 Skill（"creative skill discovery"），而纯向量检索容易遗漏

2. **DAG 依赖管理**：每个 Skill 的执行上下文明确列出上游产物及其用途，避免歧义和错配

3. **Skill 重用机制**：根据任务描述向量相似度，可以复用历史任务的 DAG 编排，跳过检索和编排两个阶段（recipe pool）

4. **多轮迭代与反馈**：虽然文中重点在一次编排执行，但框架为自动修正失败、逐步改进留有空间

### Benchmark 配置

| 模型配置 | 树构造/检索/编排：Claude Opus 4.5；节点执行：Claude Sonnet 4.5；评估：Opus 4.5 |

### 主结果（Table 1 → Bradley-Terry Score）
![Figure 4: 多类别评分雷达图](https://arxiv.org/html/2603.02176v1/x4.png)

**Figure 4**: 三种 AgentSkillOS 变体（Quality-First/Efficiency-First/Simplicity-First）在全部200、1K、200K三个生态规模上均形成最大的多边形，说明其在各类任务上均表现最优。

**关键数字**：
- 三种 AgentSkillOS 变体在**全部**生态规模上都超越 w/ Full Pool（将整个库提供给原生 Claude Code）
- w/ Full Pool 分数随生态规模增大而急剧下降，揭示了 flat invocation 的根本瓶颈
- Vanilla（无 Skill）最低

这验证了：**生态规模越大，结构化管理的收益越明显**。

### 消融实验与分析
![Figure 5: 消融对比（W/T/L 配对计数）](https://arxiv.org/html/2603.02176v1/x5.png)

**Figure 5**: Quality-First 逐次与下列变体对比：
- vs w/ Full Pool（无检索无编排）：Quality-First 压倒性胜
- vs w/ Retrieval（仅检索无编排）：Quality-First 继续领先  
- vs w/ Oracle Skills（oracle Skill 无编排）：Quality-First 仍有明显优势
- vs Quality-First (Oracle)（oracle Skill + 编排）：Quality-First 接近 oracle 性能

**结论**：
- **DAG 编排是关键** → 即使给予 oracle Skill 集，flat invocation 仍然表现远逊
- **树形检索有效** → Quality-First 与 oracle upper bound 差距在各生态规模上保持小且稳定，验证了树形检索的逼近能力

![Figure 6: DAG 结构特性对比](https://arxiv.org/html/2603.02176v1/x6.png)

**Figure 6**: 三种编排策略生成的 DAG 在节点数、边数、最大深度（最长依赖链）、最大宽度（最大并行度）上的分布：
- Quality-First：最大最深，边最密 → 细粒度多阶段管道
- Efficiency-First：节点数相近，但更宽更浅 → 优化并行
- Simplicity-First：最紧凑 → 本质步骤只

验证了三种策略在编排拓扑上确实体现了设计目标，而非仅是技术变种。

### 定性案例（Figure 7）
三个代表性任务的并排展示（Vanilla vs Quality-First）：
1. **太空探索交互滚动网页**：Vanilla 生成平面、低分辨率页面；AgentSkillOS 调用图像生成 + 前端设计 Skill，生成高质量全屏分层滚动效果
2. **微积分教学包**：Vanilla 仅有静态 matplotlib 图；AgentSkillOS 调用动画 Skill 生成 Manim 极坐标平滑演进，配套精美 PDF 讲义
3. **量子计算演示文稿**：Vanilla 文字堆砌无设计；AgentSkillOS 调用插图 + 演示文稿 Skill，生成 Bloch 球可视化、布局一致、品牌统一的专业幻灯片

**核心观察**：性能差异来自 **Skill 发现与调用的结构化程度**，而非 LLM 底层能力。

## 结论与启发

### 论文结论
1. **能力树 + DAG 是生态规模管理的必要条件**：平铺式调用在小规模（200）时尚可应对，但在千级、万级生态中完全失效
2. **编排比检索更关键**：即使 Skill 选择完美（oracle），缺乏结构化编排也会严重损害质量
3. **三种策略各有所长**：Quality-First 最大化质量但成本高；Efficiency-First 降低延迟；Simplicity-First 便于人理解——用户可根据场景权衡

### 对 Skill 管理与生态发展的启发

1. **层级分类是基础设施**：不应该让每个用户都用纯语义检索；构建全局的、由专家或众包维护的能力树是必要的长期投资

2. **跨 Skill 编排的标准化**：定义清晰的 Skill 输入输出规范、依赖声明格式，让 LLM 能可靠地推理和生成编排计划

3. **质量与规模的权衡**：使用频率队列动态维护"活跃 Skill 集"，将低频/低质 Skill 隔离到休眠索引，既保留了生态的开放性，又避免了噪声干扰

4. **编排计划的复用与演进**：通过向量相似度检索相似任务的历史编排方案，可加速推理同时降低幻觉风险；长期可积累"最佳实践 DAG 模板库"

5. **可解释的评测框架**：Bradley-Terry 配对评估给出了细粒度的排序，避免了任意的绝对评分，为迭代优化提供清晰信号

6. **自进化的入口**：因为 Skill 是可读的 markdown 制品，失败的编排执行日志可被用于自动化地改进 Skill 定义、补充示例、修复边界条件——为 Skill 沉淀与自进化奠定基础
