## 针对的问题（Motivation）

现有 LLM agent 的"自我进化"范式存在三个结构性缺陷：

1. **孤立学习（Isolated Learning）**：每个 agent 独立反复执行同类任务，重复发现相似经验，造成大量冗余探索。
2. **经验泛化弱（Weak Generalization）**：复杂环境中高质量训练数据稀缺，挖掘出的经验往往难以迁移到新任务。
3. **模型能力瓶颈（Model Capability Bottleneck）**：若经验完全由 agent 自身探索和反思获取，提取质量受制于 agent 当前能力上限。

核心问题：**什么形式的经验可以跨越不同能力的 agent、跨越不同环境被广泛复用？**

受 Anthropic Claude Skills 启发，论文认为"技能（skill）"是更合适的抽象——它封装了可复用的能力，直接支持任务执行。但此前 skill 设计依赖长上下文渐进披露（long-context progressive disclosure），对推理和环境沙箱要求过高，限制了鲁棒性和实用性（见下图对比）。

![Figure 1: Claude Skills vs SkillX 表示对比](https://arxiv.org/html/2604.04804v1/x1.png)

*Figure 1: Claude Skills 采用长上下文渐进披露格式，需要复杂沙箱系统和多轮交互；SkillX 采用层次化、条目化表示，通过轻量级检索模块存取，一次性注入 system prompt，更易在不同 base model 间迁移。*

---

## 核心思路（一句话）

SkillX 通过**三层次技能设计（hierarchical skill design）+ 迭代精炼（iterative refinement）+ 探索式扩展（exploratory expansion）**，从 agent 轨迹中全自动构建可插拔的技能知识库（SkillKB），使弱模型能直接使用强模型提炼出的结构化经验。

## 方法

### 整体 Pipeline

![Figure 2: SkillX 自动化迭代 Pipeline](https://arxiv.org/html/2604.04804v1/x2.png)

*Figure 2: SkillX 提供自动化迭代 pipeline，集成技能提取（Skills Extraction）、技能扩展（Skills Expansion）与技能精炼（Skills Refinement）。技能库组织为三层：Planning Skills、Functional Skills、Atomic Skills。*

---

### 3.1 Multi-Level Skills Design（多层次技能设计）

技能库 $\mathcal{D}$ 定义为三层结构的并集：

$$\mathcal{D} = S_{\text{plan}} \oplus S_{\text{func}} \oplus S_{\text{atomic}}$$

| 层次 | 对应粒度 | 作用 |
|------|----------|------|
| **Planning Skills**（规划技能） | 任务级 | 捕获高层任务组织结构（子任务顺序、依赖、分支），指导 functional skills 的组合方式 |
| **Functional Skills**（功能技能） | 子任务级 | 实现可复用的、基于工具的子程序，对应一个子任务的完成步骤；字段：name / document / content |
| **Atomic Skills**（原子技能） | 单工具级 | 单个工具的增强规格说明，包含调用模式、参数配置、常见约束和失败模式；是 functional skills 缺失时的低级补充 |

**提取流程**：
- **Planning skills**：从成功轨迹中压缩出有序的高层步骤集合，过滤掉探索、回溯、试错等非必要转换。
- **Functional skills**：以 planning skill 为引导，逐步骤提取对应子任务的工具调用模式。
- **Atomic skills**：从轨迹中蒸馏单工具的调用惯例、典型参数配置和实际约束注意事项。

---

### 3.2 Rollout & Skills Extraction（轨迹采样与提取）

给定任务 $q$，执行 $m$ 次 rollout 收集轨迹，使用 skill extractor $f$ 从轨迹中提取多层次技能：

$$\tau^{(k)} \sim \pi(\cdot \mid \rho_{\mathcal{D}^{(k)}}(q),\ q), \quad q \in \mathcal{Q}_{\text{train}}$$

每条轨迹产生候选技能集 $\mathcal{S}_i^{(k)} = f(\tau_i^{(k)})$，批次聚合后通过精炼算子 $\phi$ 更新库：

$$\mathcal{D}^{(k+1)} \triangleq \mathcal{D}^{(k)} \cup \phi\!\left(\bigcup_{i=1}^{N_k} \mathcal{S}_i^{(k)}\right)$$

---

### 3.3 Iterative Skills Refinement（迭代精炼）

精炼算子 $\phi$ 包含两个操作：

**Skills Merge（技能合并）**：
- 用余弦相似度检索并聚类语义相近的技能。
- 将多个候选更新方向聚合为最终更新方向：$\delta_{\text{agg}} = \sum_{i \in \mathcal{Z}(s)} \delta_i$
- 合并后若技能过于复杂，进一步分解为更模块化的子技能。

**Skills Filter（技能过滤）**：
- **General Filter**：移除依赖外部 Python 包、函数风格过于特殊、或过度封装的技能。
- **Tool-specific Filter**：对照环境工具 schema 验证每个技能，拒绝引用不存在工具、无效参数或 schema 不兼容的技能。

**库更新**：add（新增）/ modify（修改）/ keep（保留），最多迭代 3 轮。

---

### 3.4 Exploratory Skills Expansion（探索式技能扩展）

当种子训练数据稀缺时（如 AppWorld 暴露数百个 API），仅靠演示数据不足。SkillX 采用**经验引导探索（Experience-Guided Exploration）**：

1. 统计已有 rollout 中各工具的使用情况：已可靠使用的工具、高失败率工具、从未被调用的工具。
2. 优先探索"under-explored 或 failure-prone"工具，提升采样效率。
3. 从探索轨迹中合成新任务 $\mathcal{Q}_{\text{syn}}$，再次运行技能获取与精炼 pipeline。

相比随机探索，经验引导策略发现更多样化的技能。

## 关键设计细节

### Skill 数据结构

Functional Skills 和 Atomic Skills 共用字段格式：

| 字段 | 含义 |
|------|------|
| `name` | 技能名称 |
| `document` | 输入、输出、使用说明描述 |
| `content` | 完成对应子任务的工具调用模式 |

### 使用时的检索策略（SkillX Usage）

1. **Planning Skills 检索与伪计划改写**：
   - 先检索与当前任务语义相似的 planning skills：$\mathcal{P}(q) = \rho(q)$
   - LLM 结合检索结果改写出任务特定的伪计划：$\tilde{p}(q) = \text{LLM}_{\text{rewrite}}(q, \mathcal{P}(q))$
   - 伪计划仅作检索查询，**不注入** system prompt（防止幻觉）

2. **Functional & Atomic Skills 检索**：
   - 以伪计划各步骤为检索查询，逐步检索 $\mathcal{S}_i = \rho(\text{step}_i)$
   - 去重后，LLM 自过滤保留适用技能：$\mathcal{S}_q = \text{LLM\_select}(q, \tilde{p}(q), \mathcal{S}')$
   - 最终技能集 $\mathcal{S}_q$ 注入 system prompt

### 实现参数

- 骨干 agent：GLM-4.6（强 agentic 能力）
- 每个训练任务独立 rollout 4 次
- 最大精炼迭代轮数：3 轮
- 扩展阶段每个训练任务限制 1 次 rollout，温度 1.0
- 检索 embedding 模型：Qwen3-Embedding-8B
- 最小余弦相似度阈值：0.45

## 实验与效果

### Benchmarks

| Benchmark | 特点 | 训练/测试划分 |
|-----------|------|--------------|
| **BFCL-v3** | 多轮函数调用，API 密集 | 50 训练 / 150 测试 |
| **AppWorld** | 长时序、用户交互、数百个 API | 90 训练 / Test Normal |
| **τ²-Bench** | 对话式 agent，双控制环境（retail/airline/telecom） | 各子域独立划分 |

### Baselines

- **No Memory**：无先验经验直接推理
- **A-Mem**：动态管理结构化情节记忆
- **AWM**：复用历史轨迹蒸馏出的模块化工作流
- **ExpeL**：检索历史轨迹作 few-shot 示例 + 蒸馏洞见

### 关键分析发现

**哪种技能最有效？**（AppWorld 实验）
- **Planning skills** 对弱模型（Qwen3-32B、K2）一致减少执行步骤；若直接加 Functional+Atomic Skills 反而可能让弱模型过度模仿、损害性能。
- **Functional skills** 对整体性能提升贡献最大；K2 和 GLM-4.6 仅用 Functional+Atomic 即可观察到明显提升。
- **Atomic skills** 对关键 API 提供重要澄清；缺失时性能大幅下降。
- 最优组合因模型而异：GLM-4.6 = 全部技能；K2 = Functional+Atomic；Qwen3-32B = 仅 Planning。

**迭代精炼**：多轮迭代在训练集和测试集上均进一步提升性能，但训练数据有限时纯文本优化可能过拟合，需选择合适轮数。

**技能扩展**：经验引导策略比随机探索发现更多新颖技能；在复杂环境、稀缺训练数据场景下，技能扩展是经验学习的关键组成。

**执行效率**：SkillX 并非以最少执行步骤或最少输入 token 获胜，而是在效率和性能间取得最佳平衡。

## 结论与启发

### 论文结论

SkillX 证明了：
1. **层次化结构化经验**（planning→functional→atomic）比单一形式（轨迹、工作流、洞见）更有利于跨模型迁移。
2. 强模型（GLM-4.6）提炼的 SkillKB 可直接插拔到弱模型（Qwen3-32B），带来约 10% 的性能提升，同时提升执行效率。
3. 经验表示形式的选择比经验提取模型的强弱更关键。
4. 探索式扩展 + 迭代精炼在数据稀缺的复杂环境中尤为重要。

### 局限性

- 技能与具体工具 schema 绑定，跨域（不同工具生态）直接复用困难。
- 当前主要针对工具调用型 agent，纯对话场景（无函数调用）覆盖不足。

### 对后续研究的启发

1. **技能粒度分层**是 agent 记忆系统设计的有效范式，可推广到更多 agent 框架。
2. **强模型蒸馏 → 弱模型受益**的路径，提供了一种低成本增强弱 agent 的实用方案。
3. **经验引导探索**（而非随机探索）显著提升了技能库的覆盖多样性，值得在其他 self-evolving agent 工作中借鉴。
4. **伪计划改写（Pseudo-Plan Rewriting）**作为检索中间桥梁的设计，是缓解复杂环境中任务与历史经验语义不匹配问题的实用技巧。
