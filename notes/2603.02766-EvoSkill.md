## 针对的问题

### Motivation / 痛点

1. **编码 Agent 缺乏领域专业知识**：以 Claude Code、OpenHands、Codex 为代表的编码 Agent 虽然是通用问题求解器，但其灵活性本身并不赋予特定领域任务所需的专业能力。

2. **技能（Skill）目前依赖人工编写**：现有的 Agent Skill 绝大多数是手工制作的，需要大量领域知识和人工投入，随着目标任务增多，扩展性极差。

3. **已有进化/优化方法层次太低**：AlphaEvolve、GEPA 等进化方法直接优化 prompt 或代码等"低级制件"（low-level artifacts），这些制件与特定模型和任务强耦合，**不能自然产生可在不同场景复用的组件**，迁移性差。

### 现有方法不足小结

| 方法类别 | 代表工作 | 不足之处 |
|---------|---------|---------|
| 手工技能 | 各类 Agent 工具链 | 不可扩展，需大量人力 |
| 进化优化（低层） | AlphaEvolve, GEPA | 优化 prompt/代码，与任务/模型强绑定，不可迁移 |
| 自我细化 | Self-Refine | 只作用于单次输出，不跨迭代积累知识 |

---

## 核心思路

**通过迭代失败分析，自动发现并精炼结构化、可复用的 Agent 技能（Skill Folder），而非优化低层 prompt 或代码。**

---

## 方法

### Framework 总览

EvoSkill 在**技能抽象层**（而非 prompt/代码层）进行进化搜索，底层模型始终保持冻结，只有技能库和 Agent 元数据在迭代中演化。

![EvoSkill loop 总览图](https://arxiv.org/html/2603.02766v1/x2.png)

*Figure 1: EvoSkill 迭代循环总览。*

### 三个协作 Agent

| Agent | 角色 | 权限 |
|-------|------|------|
| **Executor (A)** | 在当前 Agent 程序（含技能库）下执行任务 | 只读代码仓库 |
| **Proposer (P)** | 分析执行 trace 和失败案例，提出创建新技能或编辑已有技能的高层描述 | 只读；维护累积反馈历史 H |
| **Skill-Builder (S)** | 将 Proposer 的提案具体化为完整技能文件夹（SKILL.md + 可选 Python/TS 脚本） | 可写技能目录 |

### Pipeline 详解

1. **数据划分**：将数据集用 LLM 分类到 K 个类别，然后分层切分为训练集（失败检测）、验证集（前沿选择）和测试集（从不参与演化）。

2. **程序表示**：每个"程序"（Agent Program）= 系统提示 + 积累技能库，存储为 git 分支；分支间只有技能文件夹和元数据不同，确保性能差异可归因于技能。

3. **EvoSkill 迭代循环**（Algorithm 1）：
   - 从 Pareto 前沿 G（容量 k=3）以轮询方式选择父程序 p；
   - 对一批训练样本运行 Executor，收集得分低于阈值的失败集合 F；
   - Proposer 接收 F 和历史 H，执行结构化失败分析（trace review → gap analysis → 现有技能审计 → 技能识别），输出提案 π（CREATE 或 EDIT）；
   - Skill-Builder 将 π 物化为候选程序 p̃；
   - 在验证集 V 上评估 p̃；若分数超过前沿最弱成员则替换入 G，否则丢弃；
   - 将提案结果追加到历史 H，供后续迭代参考；
   - T 轮后返回 G 中得分最高的程序。

4. **技能格式**：遵循 Agent Skills 规范（agentskills.io），每个技能为文件系统目录，包含：
   - `SKILL.md`：触发元数据 + 流程指令
   - 可选 helper 脚本（Python / TypeScript）
   - 可选参考材料

## 关键设计细节

### Pareto 前沿治理

- 保留固定容量（k=3）的最高得分程序集合，用 git tag 追踪（前缀 `frontier/`）。
- 候选进入前沿的条件：分数 > 当前最弱成员（若前沿未满则直接加入）。
- 设有早停机制：连续若干迭代无前沿改进则停止。

### 失败检测中的评分

- 训练循环中使用多容忍度加权评分（τ ∈ {0.0, 0.01, 0.025, 0.05, 0.10}，权重 w(τ)=1/(1+20τ) 偏向严格阈值），加权分低于 0.8 的样本被标记为失败。
- 最终评估使用 τ=0 精确匹配。

### Proposer 的反馈历史 H

- 累积记录所有提案、其分数变化、接受/拒绝结果；
- 防止冗余提案，引导 Proposer 对"曾部分奏效"的方向进一步改进；
- 对被丢弃的提案，下次需说明本次提案与之有何不同。

### Skill-Builder 的 Meta-Skill 引导

Skill-Builder 本身通过读取一个 `skill-creator/SKILL.md` 元技能来了解技能最佳实践，实现"技能创造技能"的自举。

### Proposer Prompt 核心流程（简化）

```
分析步骤：
1. 头脑风暴 2-3 种方案（基于 brainstorming skill）
2. 盘点已有技能，检查是否有覆盖
3. 审查反馈历史中被丢弃的提案
4. 确定动作类型：EDIT（已有技能未能覆盖）/ CREATE（无对应技能）

输出：
- action: "create" | "edit"
- target_skill: (EDIT 时必填)
- proposed_skill: 详细描述
- justification: 参考具体 trace、已有技能、历史迭代
- related_iterations: [迭代编号列表]
```

## 实验与效果

### Benchmark 一览

| Benchmark | 类型 | 规模 | 核心挑战 |
|-----------|------|------|---------|
| **OfficeQA** | 落地推理（Grounded Reasoning） | 246 题，~89,000 页美国财政部公报 | 跨文档定位 + 复杂表格 + 数值推理 |
| **SealQA** | 搜索增强 QA（seal-0 split） | 111 题 | 开放网络检索，结果含噪或冲突 |
| **BrowseComp** | 浏览 Agent（零样本迁移目标） | 128 题（分层采样） | 短答案，唯一正确，事实查找 |

所有实验底层模型均为 **Claude Code with Opus 4.5**，模型参数完全冻结。

### OfficeQA 主结果

![OfficeQA 性能对比图（各训练比例 × 各容忍阈值）](https://arxiv.org/html/2603.02766v1/x3.png)

*Figure 2: EvoSkill 在 OfficeQA 上的表现。skill-merge 配置（合并独立运行发现的技能）达到最高精确匹配准确率 67.9%，比基线提升 7.3 个百分点。*

**Table 1：各训练比例 × 各容忍阈值下的准确率（%）**

| 训练划分 / 容忍阈值 | 0.00% | 0.10% | 1.00% | 5.00% | 10.00% |
|-------------------|-------|-------|-------|-------|--------|
| base（基线） | 60.6 | 66.3 | 72.8 | 77.2 | 79.7 |
| 5% 训练 | 63.4 | 67.4 | 74.3 | 77.6 | 80.1 |
| 10% 训练 | **65.8** | **69.9** | **76.4** | **80.5** | **82.5** |
| 15% 训练 | 64.5 | 69.9 | 75.6 | 79.3 | 81.3 |
| merge-unique（技能合并） | **68.1** | **70.8** | **77.1** | **80.5** | **82.4** |

> 蓝色 = 列最优。10% 训练是单次运行最优；15% 出现轻微过拟合/收益递减；merge-unique 整体最优。

**关键数字（OfficeQA）**：
- 精确匹配：60.6% → 67.9%（**+7.3%**，skill-merge）
- 仅用 10% 训练数据（24 例）即可获得 +5.2% 精确匹配提升

### SealQA 结果

| 配置 | 准确率 |
|------|--------|
| 基线（Claude Code Opus 4.5） | 26.6% |
| EvoSkill（10% 训练，1.5 epochs） | **38.7%** |
| 绝对提升 | **+12.1%** |

核心发现技能：**search-persistence-protocol**——强制要求词义扩展、多源验证、完整性检查，防止 Agent 在初始检索结果嘈杂时过早放弃搜索。

### 零样本迁移（BrowseComp）

| 配置 | 准确率 |
|------|--------|
| 基线（Claude Code Opus 4.5） | 43.5% |
| + search-persistence-protocol（直接从 SealQA 迁移，无修改） | **48.8%** |
| 绝对提升 | **+5.3%** |

## 消融与分析

### 训练数据量的影响（OfficeQA）

- 5% → 10%：精确匹配显著提升（+2.4 pp）
- 10% → 15%：性能轻微下滑（64.5% < 65.8%），提示**约 10% 为最优训练规模**，更多数据可能导致过拟合或分布偏移
- 规律在各容忍阈值上一致

### Skill Merge 的互补性

独立运行各自发现不同失败模式，合并后：
- 精确匹配进一步提升至 68.1%（merge-unique），超过任何单次运行
- 表明多次独立运化的技能具有**互补性**，不相互覆盖

### 零样本迁移分析

- **迁移为何有效**：技能是结构化、可解读的文件夹，带有显式触发条件，与训练任务和底层模型解耦
- **与 prompt 迁移的对比**：优化 prompt 迁移往往因模型或任务分布偏移而退化；技能作为更高抽象单位具有更好的迁移性
- **局限**：迁移实验目前仅覆盖一对任务（SealQA → BrowseComp），更广泛的验证留待未来工作

## 结论与启发

### 论文结论

1. EvoSkill 在两个截然不同的 benchmark 上均取得显著提升，且**仅使用极小训练子集**（5%–15%）。
2. 在**技能层（skill level）**进行优化比在 prompt/代码层优化产生更可解读、可组合、可迁移的制件。
3. 技能进化是持续累积的过程——技能库不断扩展，Agent 能力逐步增强，底层模型不需要更新。

### 个人启发

- **抽象层次决定迁移性**：从 prompt-level → code-level → skill-level，优化粒度越粗，可迁移性越强，但发现难度也越高。EvoSkill 用自动化解决了"发现难度"的问题。
- **失败驱动是有效信号**：利用有标签的失败样本做根因分析（类比监督学习中的误分析），比无监督自我进化更高效。
- **Pareto 前沿 + 验证集过滤**：防止技能退化，是保证进化方向正确的关键机制。
- **Skill Merge 作为集成策略**：多次独立运化后合并唯一技能，类似 ensemble，可进一步提升上限。
- **潜在应用**：共享技能库（skill marketplace）——让一个任务上进化出的技能可被其他 Agent 和用户浏览、组合和复用。
- **未来方向**：多模态任务（视觉+代码+语言协同技能）、跨模型和跨 Agent 框架的技能迁移评估。
