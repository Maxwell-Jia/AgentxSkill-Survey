## 针对的问题（Motivation）

现代 Agent 系统中，Skill Library 规模正从数十个快速扩展到数千乃至数万个可复用技能（skill）。主流应对策略存在两大根本缺陷：

1. **Vanilla Skills（全量加载）**：将完整技能库拼入上下文窗口。Token 成本随库规模线性增长，超载的上下文导致模型忽略关键约束（"Lost in the Middle"现象），同时大幅增加幻觉和延迟。

2. **Vector Skills（向量检索）**：按语义相似度检索 top-k 技能。语义相似 ≠ 可执行充分——对于长链任务，顶级语义匹配往往是高层级的求解器技能，而真正执行所需的底层解析器、环境预处理、输出格式化等辅助技能与查询在语义上距离较远，极易被遗漏，形成所谓的 **prerequisite gap（前置依赖缺口）**。

核心 insight：**语义接近不等于可执行完备**。检索系统需要捕捉技能间的依赖结构，才能为 Agent 组装出可以直接执行的最小完整技能包。

![Figure 1: 三种策略对比](https://arxiv.org/html/2604.05333v1/x1.png)

*Figure 1: 概念对比图——Vanilla Skills 把全库塞入 prompt，相关约束和前置技能被淹没在超载上下文中；Vector Skills 提升效率但仍可能遗漏功能性前置技能（prerequisite gap）；Graph of Skills 从混合语义-词法种子出发，通过结构感知检索恢复前置技能，组装出紧凑的执行包。*

---

## 核心思路（一句话）

GoS 离线从本地技能包构建可执行的**类型化有向技能图**（executable skill graph），推理时通过混合语义-词法播种 + 反向感知 Personalized PageRank 图扩散 + 预算约束水化（hydration），输出一个**依赖完备、token 高效的技能执行包**（dependency-aware skill bundle）。

## 方法（Framework）

![Figure 2: GoS 整体框架图](https://arxiv.org/html/2604.05333v1/x2.png)

*Figure 2: GoS 框架总览。左：离线索引将本地技能包转换为规范化技能节点和类型化边。中：类型化有向技能图作为检索底层（边标签含 dependency、workflow、semantic、alternative 四类关系）。右：在线检索将任务查询映射为紧凑查询 schema，通过混合种子生成、反向感知 PPR 扩散和预算化水化，输出执行包。*

### 3.1 问题形式化

设本地技能语料库 C = {d₁, …, dₘ}，GoS 将其转换为类型化有向图：

> **G = (V, E, w, φ)**

- **V**：节点，每个节点是一个规范化的可执行技能记录
- **E**：边，连接两个技能
- **w(e) > 0**：边权重
- **φ(e) ∈ R**：边类型，R = {dep, wf, sem, alt}

给定查询 q 和上下文预算 τ，检索目标是找到一个束（bundle）B(q) ⊆ V，使相关性 + 依赖完备性最大化：

> max\_{B⊆V} Σ_{v∈B} rel(v,q) + β·Σ_{(u,v)∈E\_dep} 𝟙[u∈B ∧ v∈B]  s.t. cost(B) ≤ τ

该优化问题通过三阶段近似求解：混合种子检索 → 反向感知图扩散 → 预算化重排与水化。

---

### 3.2 离线图构建（Offline Graph Construction）

**技能规范化（Skill Normalization）**

每个技能包解析为包含以下字段的规范化节点记录：
- 规范名称、能力摘要
- 输入/输出字段（I/O schema）
- 领域标签、工具依赖
- 脚本入口点（entrypoints）
- 稳定本地源路径（stable local source path）

解析策略优先确定性提取（parser-first）；当文档不完整时，用轻量级 LLM pass 补全检索关键语义字段（能力摘要、输入输出、示例任务），但 LLM 不直接生成图边结构，仅作节点属性的语义补全器。

**类型化关系归纳（Typed Relation Induction）**

四类边：

| 边类型 | 英文 | 含义 |
|--------|------|------|
| dep | dependency | 可执行前置依赖：vᵢ 的输出兼容 vⱼ 的输入，则 vᵢ → vⱼ |
| wf | workflow | 常见多步流程中的顺序关系 |
| sem | semantic | 近似重复或主题相邻的技能 |
| alt | alternative | 针对同一子问题的可互换策略 |

- **dependency 边**：通过 I/O 兼容性匹配确定性归纳（producer-consumer overlap 超阈值即建边），具有明确的可执行方向语义，不可无损降为无向相似图。
- **非依赖边（wf/sem/alt）**：通过**稀疏验证**建立——先用词法相似度 + 语义邻居 + I/O 扩展为每个节点构造候选池，再在池内进行 LLM 关系验证（高精度优先、不确定则不建边），避免全量 pair-wise 推断。

---

### 3.3 在线结构化检索（Online Structural Retrieval）

**混合播种（Hybrid Seeding）**

原始查询映射为轻量检索 schema（目标、关键操作、引用产出物、归一化关键词），然后：
- **语义种子分**：sᵢˢᵉᵐ(q) —— 向量索引最近邻
- **词法种子分**：sᵢˡᵉˣ(q) —— BM25 风格，在规范化节点字段上计算 token 重叠

合并为初始种子分布：

> zᵢ(q) = η·sᵢˢᵉᵐ(q) + (1-η)·sᵢˡᵉˣ(q)

归一化得种子分布向量 **p**。

**反向感知类型扩散（Reverse-Aware Typed Diffusion）**

对每种关系类型 r，构造前向转移算子 Tᵣ→ 和反向转移算子 Tᵣ←，合并为统一转移算子：

> T = RowNorm(Σ_{r∈R} λᵣ·(Tᵣ→ + γᵣ·Tᵣ←))

其中 λᵣ 为关系权重，γᵣ 控制各类型反向强度（**dependency 边的反向系数最大**，以便从高层 solver 向上游 prerequisite 传播相关性）。

执行反向感知 Personalized PageRank（PPR）扩散直至收敛：

> s^(ℓ+1) = α·**p** + (1-α)·T^⊤·s^(ℓ)

重启参数 α ∈ (0,1)。PPR 确保高层 solver 被检索为种子后，语义信号较弱的底层解析器/预处理工具仍能通过反向依赖路径积累分数。

**预算化重排与水化（Budgeted Reranking and Hydration）**

结合扩散分 s★ 和字段级查询证据 mᵢ(q) 重排：

> ρᵢ(q) = sᵢ★ + μ·mᵢ(q)

按 ρᵢ 降序水化（hydration）——将选中技能实例化为 Agent 可直接消费的负载（能力摘要 + 执行注意事项 + 稳定本地源路径），同时遵守单技能与全局 token 预算约束，输出有界执行包。

## 关键设计细节

### 图节点字段

| 字段 | 检索角色 |
|------|---------|
| name / canonical name | 精确匹配、词法检索 |
| capability summary | 语义检索核心字段 |
| inputs / outputs | 依赖边归纳（I/O 兼容性） |
| domain tags | 类目过滤与语义补全 |
| tooling | 工具依赖感知 |
| script entrypoints | 直接执行入口 |
| source path | Agent 水化后的稳定引用路径 |
| rendered snippet | 快速能力预览 |

### 反向权重设计

依赖边（dep）的反向系数 γ\_dep 最大，其余依次降低：

| 边类型 | 反向传播强度 |
|--------|-------------|
| dep（dependency）| 最强（从 solver → prerequisite）|
| wf（workflow）| 中等 |
| sem（semantic）| 较弱 |
| alt（alternative）| 最弱 |

这反映了一个核心原则：**反向遍历最合理的场景，就是沿依赖边反向恢复可执行前置条件**。图本身保持有向，但检索明确支持反向感知。

### 检索伪代码概要

**离线图构建（Algorithm 1）**

1. 解析每个 SKILL.md → 提取确定性字段
2. 不完整时 LLM 补全节点语义字段（不生成边）
3. 序列化为规范化节点 v，加入 V
4. 对候选对 (u,v)：计算 producer-consumer overlap，超阈值加 dependency 边
5. 对每个节点：稀疏候选池 → LLM 验证 → 加 wf/sem/alt 边
6. 构建 HNSW 向量索引，持久化图和元数据

**在线检索（Algorithm 2）**

1. 可选：查询重写为紧凑 schema
2. 语义种子 + 词法种子 → 合并 → 种子分布 p
3. 构建带反向系数的类型化转移矩阵
4. PPR 扩散至收敛 → 图分 s★
5. 图分 + 字段级证据 → 重排
6. 水化（含精确 Source 路径）→ 预算截断 → 输出执行包 B(q)

## 实验与效果

### 实验设置

- **Benchmark**：
  - **SkillsBench**（1000 技能）：11 个领域的真实技术任务（宏观经济去趋势、电网可行性分析、3D 扫描分析、金融建模、地震相位拾取等）
  - **ALFWorld**（140 episodes）：文本化家庭机器人顺序决策任务（二值奖励 = 成功率）
- **Baseline**：
  - Vanilla Skills：全量技能加载
  - Vector Skills：同底座嵌入模型（openai/text-embedding-3-large, 3072维）的 flat top-k 语义检索
- **模型**：Claude Sonnet 4.5、MiniMax M2.7、GPT-5.2 Codex
- **评估指标**：平均奖励（R↑）、平均 token 数（T↓）、Agent 运行时间（S↓）；每个设置运行 2 次取均值

### 主结果表格（Table 1）

| 模型 | 方法 | SkillsBench R↑ | SkillsBench T↓ | SkillsBench S↓ | ALFWorld R↑ | ALFWorld T↓ | ALFWorld S↓ |
|------|------|:---:|:---:|:---:|:---:|:---:|:---:|
| Claude Sonnet 4.5 | Vanilla Skills | 25.0 | 967,791 | 465.8 | 89.3 | 1,524,401 | 53.2 |
| | Vector Skills | 19.3 | 894,640 | 357.3 | 93.6 | 28,407 | 37.8 |
| | **GoS** | **31.0** | **860,315** | 364.9 | **97.9** | **27,215** | 49.2 |
| MiniMax M2.7 | Vanilla Skills | 17.2 | 942,113 | 580.7 | 47.1 | 2,184,823 | 88.6 |
| | Vector Skills | 10.4 | 852,881 | 552.9 | 50.7 | 66,109 | 73.4 |
| | **GoS** | **18.7** | 867,452 | **502.5** | **54.3** | **65,227** | **68.8** |
| GPT-5.2 Codex | Vanilla Skills | 27.4 | 3,187,749 | 686.8 | 89.3 | 1,435,614 | 83.3 |
| | Vector Skills | 21.5 | 1,243,648 | 773.0 | 92.9 | 34,436 | 57.0 |
| | **GoS** | **34.4** | 1,379,773 | **715.6** | **93.6** | 46,462 | **64.7** |

R: 平均奖励（%），T: 总 token 数，S: 运行时间（秒）

**关键数字摘要**：
- 相对 Vanilla Skills 全量加载基线，GoS 在两个 benchmark + 三个模型家族下平均 **reward 提升 43.6%**，输入 **token 减少 37.8%**
- 相对 Vector Skills，GoS 在 SkillsBench 上平均提升 **+10.97 reward points**，在 ALFWorld 上提升 **+2.87 reward points**
- Claude Sonnet 4.5 + ALFWorld：GoS 将成功率从 89.3%（Vanilla）/ 93.6%（Vector）提升至 **97.9%**，同时 token 从 1,524,401 压缩至 **27,215**（减少约 98%）

**核心发现**：
1. **Vanilla Skills 的全量加载代价极高**，且存在 "Lost in the Middle" 效应，使模型遗漏关键依赖技能
2. **Vector Skills 在长链技术任务中大幅退步**（在 Claude/MiniMax 下甚至低于全量加载），验证了 prerequisite gap 假设
3. **GoS 在所有 6 个模型×benchmark 组合中均达到最高奖励**，且 token 成本远低于 Vanilla

|:---:|------|:---:|:---:|:---:|
| 200 | Vanilla Skills | 32.5 | 1.85 | 701.6 |
| | Vector Skills | 21.2 | 1.06 | 833.8 |
| | **GoS** | **32.1** | 1.36 | 731.2 |
| 500 | Vanilla Skills | 26.0 | 1.93 | 756.8 |
| | Vector Skills | 20.7 | 1.10 | 849.5 |
| | **GoS** | **31.4** | 1.16 | 890.3 |
| 1000 | Vanilla Skills | 27.4 | 3.19 | 686.8 |

- Vanilla Skills 的 prompt token 随库规模从 500→2000 增长约 **3×**（1.93M→5.84M），而 GoS 维持在 **1.14M～1.38M**

- 依赖感知检索的优势不随库规模增大而削弱

### 组件消融（Table 3）

| 方法 | R↑（%）| T（M）↓ | S（s）↓ |
|------|:---:|:---:|:---:|
| **Full GoS** | **34.4** | 1.38 | 715.6 |
| w/o 图传播（graph propagation）| 29.3 | 0.89 | 766.2 |
| w/o 词法检索 + 重排（lexical + rerank）| 26.7 | 1.01 | 747.7 |

- **去掉图传播**：reward 下降 5.1 点，token 减少但覆盖不足，证明图扩散是从种子恢复前置依赖的关键
- **去掉词法检索+重排**：reward 下降 7.7 点（更大），说明**种子质量是瓶颈**——初始种子质量差时，图扩散也无从恢复正确的依赖链
- 两组件均不可或缺，且种子质量的影响 > 图扩散的影响

## 结论与启发

### 核心结论

GoS 证明了：**在大规模技能库中，结构化依赖感知检索优于语义相似度检索和全量加载**，且这一优势在跨 benchmark、跨模型家族、跨库规模下均成立。

### 局限性

1. **依赖图质量是上限**：文档不完整、I/O schema 模糊、或执行元数据缺失会降低边质量，从而影响下游检索
2. **图是静态的**：当前系统不从实际执行轨迹、验证器反馈或用户交互中更新图结构

### 未来方向

- 在线边权重自适应（从成功轨迹学习）
- 基于执行反馈的图结构动态更新
- 更强的候选 bundle 重排策略
- 多模态和交互式 Agent 场景下的评估

### 对 Skill 管理与检索方向的启发

1. **Prerequisite gap 是实际问题**：仅靠语义检索对长链技术任务不够，必须显式建模技能间依赖
2. **图结构 + PPR 是可落地的解法**：GoS 的离线图构建成本摊薄到检索时，inference-time 开销可控
3. **混合种子优于单一信号**：词法检索擅长具体文件名/API，语义检索擅长抽象意图，两者互补
4. **水化（Hydration）是最后一公里**：检索结果不是搜索列表，需要物化为 Agent 可直接执行的负载（含源路径和执行注意事项）
5. **Agent Interface 是方法的一部分**：如何将检索结果"强制"传达给 Agent（何时调用、如何解释、如何约束偏离）与检索质量同等重要
6. **与 SkillNet/AgentSkillOS 互补**：GoS 定位于技能库已存在后的高效使用层，与上游技能创建、评估、编排系统形成完整生态链
