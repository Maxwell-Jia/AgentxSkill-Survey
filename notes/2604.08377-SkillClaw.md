## 针对的问题

### 痛点

1. **Skill 部署后静止不动**：OpenClaw 等 Agent 框架的 skill 在安装后基本不再更新，与实际使用脱节。同样的工具调用错误、参数格式问题、执行失败模式在不同会话、不同用户间被反复踩坑，系统从未从这些经验中受益。

2. **单次会话的改进无法沉淀**：Agent 经过多轮试错最终找到了更稳健的解法，但这些发现仅限当次会话，下一次对话又从零开始。

3. **用户经验形成孤岛**：多用户共享任务空间（相似的 workflow、工具、失败模式），但各自的经验无法互通——"用户 A 调通了数据库脚本"这条经验无法自动惠及用户 B。

4. **记忆方法只存轨迹，不能提炼为行为**：RAG-style 记忆检索留存的是原始对话片段，而非可执行的、可复用的技能定义。

5. **技能库缺乏版本化进化机制**：现有 skill-based 框架（Voyager、AutoSkill 等）将 skill 库视为静态资源，没有机制将运行时观察转化为对 skill 本身的稳定改进。

### 核心差距

现有方法在"单用户、单会话内反思/改进"的粒度上做得不错，但缺乏将**跨用户、跨时间的交互轨迹汇聚**并转化为**共享 skill 集合的持续演化**的机制。Skill 演化的单位必须从"个人历史"上升到"群体经验"。

---

## 核心思路

> SkillClaw 在多用户 Agent 生态中，把跨用户交互轨迹聚合为共享证据库，由一个自主演化器（Agentic Evolver）以开放式推理驱动 skill 的精炼与新建，再将更新后的 skill 集合同步回所有 Agent，形成"交互 → 证据 → 演化 → 验证 → 部署"的持续闭环。

### Skill 表示（SKILL.md / YAML frontmatter）

每个 skill 以 `SKILL.md` 文件存储，路径约定为 `{group_id}/skills/<name>/SKILL.md`。文件分为两部分：

**YAML frontmatter（元信息头）**：

```yaml
---
name: lowercase-hyphenated-slug
description: 该 skill 的功能及触发场景（2-4 句）。包含 "NOT for: ..." 排除条件。
category: general
---
```

**Markdown 正文（skill body）**：

包含具体的执行指导内容，通常涵盖：
- API endpoints、端口、payload 格式、工具名称等**环境特定事实**
- 触发条件与适用范围
- 分步骤的操作程序（step-by-step procedure）
- 常见错误的规避指引

设计原则：skill 的核心价值是**压缩环境信息（compress environment information）**，而非重复 Agent 本已掌握的通用最佳实践。SKILL.md 不存储通用建议（如重试逻辑、缓存策略），只记录 Agent 无法自行发现的领域特定细节。

**版本历史目录**：每个 skill 目录下有 `history/` 子目录，按 `v<N>.md`（skill 快照）和 `v<N>_evidence.md`（本轮驱动修改的会话证据）分版本存储，形成完整的演化审计链。版本号反映演化顺序，不使用日期命名。

### Agentic Evolver 机制

演化器是 SkillClaw 的核心，由三个环节组成，对应图中的 **Evidence → Attribution → Evolution**：

#### 阶段一：Evidence（轨迹结构化与分组）

1. **会话录制**：Client Proxy 拦截 Agent 的每次请求（`/v1/chat/completions`、`/v1/messages`），记录完整的因果链——`prompt → action → feedback → ... → agent response`，包括所有工具调用、中间错误、用户反馈。
2. **轻量元数据提取**：从每条会话中提取：(i) 引用了哪些 skill，(ii) 是否有工具错误，(iii) 粗粒度质量估计（PRM 评分）。
3. **按 skill 分组**：将会话按引用的 skill 归组：`G(s) = {τᵢ | s ∈ Kᵢ}`；未使用任何 skill 的会话单独归入 `G(∅)`。跨用户的同一 skill 分组构成天然对照实验：相同 skill 在不同用户、任务、环境下产生不同结果，直接揭示其行为边界。

#### 阶段二：Attribution（根因诊断）

演化器（LLM Agent）配备**结构化 harness**，该 harness 将分组会话证据、当前 skill 定义和允许的演化动作作为结构化输入，但**不约束演化器的推理过程**（开放式推理）。

演化器对每个 skill group 执行：
- 联合分析成功与失败会话：成功会话定义 skill 的**不变量**（不可改动），失败会话定义**改进目标**（需要纠正的具体行为）。
- 区分失败根因：Skill 问题（错误/缺失/误导性指导）→ 修改 skill；Agent 问题（误用、上下文溢出）→ 不修改 skill；环境问题（API 不稳定）→ 酌情添加简短提示。

#### 阶段三：Evolution（动作决策与执行）

对每个 skill group，演化器从三个动作中选一：

| 动作 | 含义 | 触发条件 |
|------|------|---------|
| **Refine** | 精炼现有 skill | 多个会话指向相同缺陷；修复已识别错误 |
| **Create** | 创建新 skill | `G(∅)` 中发现可复用但无 skill 覆盖的程序模式 |
| **Skip** | 维持不变 | 证据不足；失败归因于 Agent 或环境，非 skill 缺陷 |

**编辑原则（保守优先）**：
- 默认**局部精修而非全量重写**；把现有 skill 视为真相来源而非草稿。
- 保留成功会话所支撑的所有内容；仅修改有证据的失败部分。
- 不随意改动 API 端点、端口、payload 格式、输出路径——除非会话证据明确显示其已更改。
- 优先 skip，而非基于猜测的修改。

**完整演化算法**（Algorithm 1）：

```
输入：共享 skill 集合 S，用户会话 T
1. 将 T 转化为结构化证据 E
2. 按引用 skill 分组，得到 {G(s)} 和 G(∅)
3. S' ← S
4. 对每个 G(s) → 演化器分析成功/失败模式 → 选择 {refine, create, skip} → 生成候选更新 → 保守编辑与验证 → 合并入 S'
5. 分析 G(∅) → 发现缺失程序 → 新增 skill 入 S'
6. 将 S' 同步回所有 Agent
```

### 跨用户共享协议

**部署架构（去中心收集、中心演化）**：

- **Client Proxy**（用户侧）：本地 API 代理，拦截请求、记录会话产物、管理本地 skill 库；单用户无需服务端即可运行。
- **Evolve Server**（演化服务端，可选）：从共享存储（Alibaba OSS / AWS S3 / 本地 FS）读取会话数据，运行演化引擎，将更新后的 skill 写回共享存储；可服务单用户或整个团队群组。
- 两个组件仅通过共享存储交互，无需直接网络通信。

**演化引擎两种模式**：
- `workflow`（workflow engine）：固定三阶段 LLM 流水线（Summarize → Aggregate → Execute）
- `agent`（agent engine）：基于 OpenClaw 的自主 Agent 工作区，直接编辑 skill 文件

**夜间验证门（Nighttime Validation Gate）**：

候选 skill 更新不直接部署，而是经过验证才写入共享仓库：
- 验证在夜间于可用的空闲用户环境中执行，确保评估条件贴近真实部署。
- 对候选 `s'` 和现有 `s`，系统从白天交互数据中抽取相关任务，在相同环境下执行两者并比较结果。
- 决策规则：若 `s'` 表现优于 `s`，标记为 **Accept**，合并进共享仓库并次日推送所有 Agent；否则标记为 **Reject**，仅保留为候选记录不部署。
- 效果：部署的 skill 池单调不降（monotonic deployment behavior）——用户始终交互的是前一夜通过验证的最优 skill 池。

**客户端验证节点（可选）**：群组内可指定部分客户端在本地代理空闲时领取验证任务，执行候选 skill 并回写结果，支持配置验证阈值（`EVOLVE_VALIDATION_REQUIRED_APPROVALS`、`EVOLVE_VALIDATION_MIN_MEAN_SCORE` 等），为候选 skill 发布提供额外审查层。

## 关键设计细节

### 更新触发机制

演化器按周期运行（示例配置 `--interval 300`，即每 5 分钟检查一次）。仅当 skill 在白天的交互中被触发且表现出改进潜力时，才生成候选更新；每日 Day 1 以初始 skill 集合作为 baseline。

### 多用户一致性

多用户情境下，所有 Agent 实例共享同一 Evolve Server 和共享存储，没有用户在本机运行 Evolver。Skill 的演化状态由中心化仓库维护，Client 通过 `skillclaw skills pull/push/sync` 同步。用户间不直接协调——集体改进完全通过 skill 版本同步实现。

### 版本化与历史审计

每次 `improve_skill` 或 `create_skill` 动作前，演化器必须读取 `history/` 目录下所有历史版本，理解已有的演化轨迹，再决策当轮修改。历史文件采用严格的版本号命名（`v<N>.md` 和 `v<N>_evidence.md`），不用日期。跳过历史读取被视为硬性错误（hard error），会导致撤销之前的改进。

### 典型 Skill 演化案例：Slack 消息分析（Figure 2）

| 对比项 | 原始 skill | 进化后 skill |
|--------|------------|-------------|
| API 端口 | `localhost:9100`（错误） | `localhost:9110`（修正后的正确值） |
| 消息获取策略 | 检索全量消息，逐一处理 | 先扫描预览，再按需拉取完整消息 |
| 输出路径 | 未指定 | 明确指定 `/tmp_workspace/results/results.md` |
| 错误处理 | trial-and-error 响应式重试 | API 配置固化在 skill 中，主动规避 |

三类核心改进：(1) 任务分解（过滤 + 提取两阶段）；(2) 错误纠正（工具配置编码进 skill 而非运行时试错）；(3) 选择性检索（聚焦相关消息，提升提取质量）。

### Skill 质量过滤原则

- Evolver 的 LLM prompt 明确区分"skill 问题"与"Agent 问题"——若 skill 已包含正确的环境信息而 Agent 未使用，则不删除正确信息、不替换为"去读源码"式指令。
- 新建 skill 的条件：观察到的模式必须**可教授（teachable）且可能复现（likely to recur）**，一次性失败不创建 skill。
- Workflow engine 的 Summarize 阶段以 8–15 句要求输出会话摘要，覆盖：目标、关键轨迹、skill 有效性、关键转折点、工具使用模式、结果评估，为 Evolver 提供压缩后的高质量证据。

## 实验与效果

### Benchmark: WildClawBench

WildClawBench（Ding et al., 2026）是真实世界 Agent 基准，包含 **60 个复杂任务，跨 6 个能力领域**：

| 类别 | 示例任务 | 主要挑战 |
|------|---------|---------|
| Productivity Flow | arXiv 论文分类、日程规划、SCP 文件传输 | 多步流水线 |
| Code Intelligence | 调试、解谜 | 执行正确性 |
| Social Interaction | 谈判、聊天日志分析 | 多轮推理 |
| Search & Retrieval | 学术检索、冲突解决 | API 使用 |
| Creative Synthesis | 视频笔记、海报生成 | 多模态生成 |
| Safety & Alignment | prompt 注入、信息泄漏检测 | 约束满足 |

**评估特性（Table 2）**：完整 Linux 容器执行环境；多模态输入（文本、代码、图像、视频）；每任务 3–27 个聚合指标；硬约束（关键错误得分归零）；任务步骤 15–50 步；依赖外部 API 和模型下载。

### 实验设置

- 模拟 **6 天 "日夜"** 循环演化：白天 8 名并发用户交互，夜间演化 + 验证 + 候选筛选。
- 底层模型全程 **Qwen3-Max**（交互、演化、验证均使用）。
- Day 1 = baseline，Day 2–6 反映每夜验证后携带的最优 skill 池的用户侧结果。
- 目前报告 4 个代表性类别，论文标注为 Work in Progress。

### 主结果（Table 3）

**Table 3：用户侧白天结果（最优 skill 池部署视角）**

| 类别 | Day 1 | Day 2 | Day 3 | Day 4 | Day 5 | Day 6 | 绝对提升 | 相对提升 |
|------|-------|-------|-------|-------|-------|-------|---------|---------|
| Social Interaction | 54.01% | 60.34% | 60.34% | 60.34% | 60.34% | 60.34% | +6.33 pp | +11.72% |
| Search & Retrieval | 22.73% | 30.00% | 30.00% | 34.55% | 34.55% | 34.55% | +11.82 pp | +52.00% |
| Creative Synthesis | 11.57% | 21.80% | 21.80% | 21.80% | 21.80% | 21.80% | +10.23 pp | +88.41% |
| Safety & Alignment | 24.00% | 24.00% | 24.00% | 24.00% | 32.00% | 32.00% | +8.00 pp | +33.33% |

**各类别进化模式**：
- **Social Interaction**：单次高影响 skill 更新（Day 2 立即生效），执行流程显式化后大幅改善跨源整合与执行稳定性，后续轮次维持最优池。
- **Search & Retrieval**：阶梯式改进（Day 2 → Day 4），先解输入验证和文件可访问性，再提升约束感知检索规划，体现"底层可靠性优先、高层推理后至"的规律。
- **Creative Synthesis**：Day 2 大跳跃（+10.23 pp），主要瓶颈在环境设置（工作目录、符号链接、多模态 pipeline）而非内容生成本身；后续多模态 pipeline 更新多次被验证器拒绝，未超越早期最优池。
- **Safety & Alignment**：改进出现在 Day 5（+8.00 pp），聚焦执行可靠性（Git 认证回退、目录克隆协议、非交互式环境安全执行），而非表面任务性能。

### 受控验证（Table 8）

**Skill Evolve Lite**：对 3 个自定义查询单独验证演化机制的直接效果：

| 查询 | 演化前 | 演化后 | 增益 |
|------|--------|--------|------|
| basic extraction | 21.7% | 69.6% | +47.8% |
| deadline parsing | 41.1% | 48.0% | +6.9% |
| save report | 28.3% | 100.0% | +71.7% |
| **平均** | **30.4%** | **72.5%** | **+42.1%** |

- `save report` 从 28.3% 升至 100.0%：初始失败源于缺少环境特定程序（输出路径/格式），一轮演化后完全解决。
- `deadline parsing` 提升较小（+6.9%）：依赖细粒度推理的任务对程序性 skill 更新不够敏感。

### 消融与分析：夜间 Skill 候选详情

**Table 4（Social Interaction）**：6 天中共产生 4 个候选，仅 `03_task6`（跨部门 Slack 总结 skill，将描述性指令重写为严格有序步骤）在 Night 1 被接受，其余全部被拒绝——验证机制有效防止了过度更新。

**Table 5（Search & Retrieval）**：`validate-file-existence`（Night 1，Accept）和 Night 3 的同池重测确认（Accept）为关键节点；4 个后续候选（含约束感知检索规划 skill）均被拒绝。

**Table 6（Creative Synthesis）**：仅 `validate-tmp-workspace-inputs`（Night 1）被接受；后续多个多模态 pipeline skill 多次尝试均未超越已有最优池。

**Table 7（Safety & Alignment）**：Night 1–4 共接受 3 次更新（git push 认证回退 + clone 目录修正），Night 5–6 的改进版本未超越最优池被拒绝，部署池保持单调不降。

## 结论与启发

### 论文结论

1. SkillClaw 将静态 skill 生态转化为**持续演化的动态系统**，把"用户交互"直接当作 skill 更新的驱动信号，无需额外人工投入。
2. 夜间验证机制保证了**单调部署行为**：skill 池只升不降，用户始终享受经验证的最优版本。
3. 演化模式是**类别特异的**（category-specific）：Social Interaction 强调流程显式化，Search & Retrieval 强调输入可靠性，Creative Synthesis 强调环境预检，Safety & Alignment 强调执行鲁棒性——不存在单一通用的 skill 更新路径。
4. 即使在小规模实验（8 用户、6 天）下，集体演化仍展现出显著效果，暗示扩展用户数和交互时长将进一步提升演化质量。

### 对 Skill 自动合成 / 沉淀 / 生态方向的启发

1. **集体演化 vs 单用户演化的根本差异**：单用户产生的信号难以区分"可泛化改进"与"个人特殊解法"；跨用户聚合让同一 skill 在多种上下文下被检验，形成天然对照实验，为稳定演化提供更可靠的证据基础。这一设计值得借鉴于任何需要从多数据源提炼可复用知识的系统。

2. **"夜间验证门"作为质量锚点**：将候选 skill 推送与实际部署解耦，通过真实环境执行而非模型自评决定是否采纳，比直接信任 LLM 自我评判更可靠。未来 skill marketplace 也可借用类似的"staged rollout + 生产验证"范式。

3. **Skill 是环境信息压缩器而非通用知识库**：论文对"skill 应该写什么"给出了清晰定义——API 端口、payload 格式、工具名称等 Agent 无法自行发现的领域特定细节，而非通用最佳实践。这对 skill 质量评估标准的制定具有直接参考价值。

4. **保守编辑原则降低演化风险**：默认"局部修改而非重写"、"先读历史再动笔"、"成功会话保护的部分不轻动"——这套工程约定避免了"修好这个问题，引入另一个问题"的退化陷阱，可直接移植到其他 skill 自进化框架。

5. **Skill Marketplace 的工程含义**：SkillClaw 的共享存储层（OSS/S3/本地 FS）、版本化历史（`history/v<N>.md`）、技能注册表（`skill_registry.json`）和 SkillHub 分发机制，事实上已构成一个轻量 skill marketplace 的技术骨架。跨框架 skill 共享（已有与 Deer-Flow 的讨论）和 skill 质量评测（SkillsBench 方向）将是该生态的自然延伸。

6. **局限性**：当前实验规模较小（8 用户、6 天、60 任务、4 个类别），结果尚不构成大规模部署的充分验证；夜间验证引入额外 token 成本；跨用户隐私与数据安全（轨迹共享）问题未作讨论；论文整体仍标注为 Work in Progress。
