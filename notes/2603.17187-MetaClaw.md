## 针对的问题

**核心矛盾**：LLM agent 在真实部署环境中必须持续服务用户（不能中断），但其能力随任务分布的漂移而逐渐过时（模型训练一次后便静止不动）。

具体来说，在 OpenClaw 这样的平台上（单个 agent 连接 20+ 个消息频道、处理多样化任务），现有方法均存在明显短板：

- **Memory-based 方法**（Reflexion、Mem0 等）：存储原始轨迹，但未提炼可复用的行为知识，轨迹冗长、无法跨任务迁移。
- **Skill-based 方法**（Voyager、ExpeL 等）：将经验压缩为可复用的 skill 指令，但 skill 库是静态的，从不与权重优化相协调。
- **RL-based 方法**（GRPO、DAPO 等）：更新模型权重，但存在"陈旧奖励污染"问题——当 skill 已经演化后，旧 skill 上下文收集的轨迹奖励信号已失效，若不加筛选地复用会损害梯度更新质量；此外也无法解决**何时**训练、**哪批数据**有效的问题。

**本文要解决的核心问题**：如何让 LLM agent 在真实部署中**持续 meta-learn**，同时沉淀**可复用 skill**，且不造成服务中断？

---

## 核心思路

**一句话概括**：MetaClaw 通过 continual meta-learning，将 LLM policy（权重 θ）与 behavioral skill 库（S）**联合演化**，技能驱动的即时 prompt 注入（零停机）与基于空闲时段的 RL 权重更新（opportunistic updates）相互强化，形成持续演化的正向循环。

## 方法

### 框架总览

![MetaClaw Framework Overview](https://arxiv.org/html/2603.17187v1/x5.png)

**Figure 1**：MetaClaw 通过两个互补循环改进 meta-model M = (θ, S)。
- **左侧（快速适应）**：Skill-driven fast adaptation——分析失败轨迹，立即扩展 skill 库 S，无需参数更新，零停机。
- **右侧（慢速优化）**：Opportunistic policy optimization——积累 post-adaptation 轨迹，由 OMLS 检测用户空闲时段（睡眠/键盘不活跃/日历）后触发 RL 权重更新（Cloud LoRA）。

---

### Meta-model 定义

agent 的行为由 **meta-model** 完全决定：

```
M = (θ, S)
```

- θ：基础 LLM policy 的权重参数
- S = {s₁, s₂, ..., sK}：skill 指令库，每条 skill 是简洁、可复用的行为指令

推理时，先通过嵌入检索从 S 中取出与当前任务最相关的 skill 子集，再输入给 policy：

```
a ~ πθ(· | τ, Retrieve(S, τ))
```

---

### 机制一：Skill-Driven Fast Adaptation（技能驱动快速适应）

当失败轨迹积累到阈值后，LLM evolver ε 分析这些失败，合成新的行为指令并并入 skill 库：

```
S_{g+1} = S_g ∪ ε(S_g, D^sup_g)
```

- g 是 **skill generation**（技能代数），每次 skill 库更新时递增
- 完全 **gradient-free**（因为自然语言空间无法做梯度下降）
- **零停机**：通过 prompt 注入生效，不触碰模型权重

**Skill 库的双重角色**：
1. **Meta-parameter**：跨任务流积累行为知识，S_{g+1} ⊇ S_g
2. **Adaptation basis**：推理时按任务检索，提供即时的任务特化

---

### 机制二：Opportunistic Policy Optimization（机会性策略优化）

在用户空闲时段，对积累的 post-adaptation 轨迹（query data）执行 RL 权重更新：

```
θ_{t+1} = θ_t + α ∇θ E_{(τ,ξ,g')~B} [R(πθ(· | τ, S_{g'}))]
```

- R 是 **Process Reward Model (PRM)** 的评分
- 实现方式：**Cloud LoRA fine-tuning + GRPO**，无需本地 GPU
- 训练 buffer B 只接受 query data（post-adaptation 轨迹），support data（失败轨迹）被排除

---

### 机制三：Skill Generation Versioning（技能代数版本控制）

**问题**：触发 skill 演化的轨迹（support data），其奖励反映的是旧 skill 下的行为，如果进入 RL buffer 会"陈旧奖励污染"梯度。

**解决方案**：每条样本打上 skill generation index g_i 的戳记：
- **Support data** D^sup_g：触发 S_g → S_{g+1} 演化的轨迹，被 skill evolver 消费后**从 RL buffer 中丢弃**
- **Query data** D^qry_{g+1}：S_{g+1} 生效后收集的轨迹，才可进入 RL buffer

当 skill generation 从 g 推进到 g+1 时，训练器自动清除 buffer 中所有版本 ≤ g 的样本。

---

### 机制四：Opportunistic Meta-Learning Scheduler（OMLS）

OMLS 是后台 daemon，监控三类空闲信号，只在用户不活跃时触发 RL 训练：

| 信号 | 说明 |
|------|------|
| **Sleep window** | 用户配置的睡眠时段（如 23:00–07:00），最大连续训练块 |
| **System inactivity** | 轮询 OS 的输入设备空闲计时器（macOS: ioreg HIDIdleTime），默认 30 分钟无输入则开启训练窗口 |
| **Calendar-aware** | 查询 Google Calendar API，当前时段在日程会议中则推断用户不可用 |

任意信号指示"用户离开"→ 训练窗口开启；任意信号指示"用户返回"→ 训练暂停（支持 mid-batch checkpointing，断点续训）。

---

### Algorithm 1：MetaClaw 完整流程

```
输入：M = (θ₀, S₀), skill evolver ε, task stream {τᵢ}, PRM R, OMLS
初始化：g ← 0, RL buffer B ← ∅

for 每条任务 τᵢ:
    # 服务任务
    S_τᵢ ← Retrieve(S_g, τᵢ)          // 检索相关技能
    ξᵢ ← Execute(πθ(· | τᵢ, S_τᵢ))   // 执行并收集轨迹
    rᵢ ← R(ξᵢ); 给 (τᵢ, ξᵢ, rᵢ) 打上代数 g

    if ξᵢ 揭示失败:
        加入 support set D^sup_g
    else:
        加入 RL buffer B（query data）

    # Skill-driven fast adaptation
    if |D^sup_g| ≥ 阈值:
        ΔS ← ε(S_g, D^sup_g)          // 从失败中合成新技能
        S_{g+1} ← S_g ∪ ΔS            // 演化技能库
        清除 B 中所有版本 ≤ g 的样本   // 支持-查询分离
        g ← g + 1

    # Opportunistic policy optimization
    if OMLS 检测到空闲窗口 and |B| ≥ batch size:
        θ ← θ + α ∇θ E_{(τ,ξ,r,g')~B}[R(πθ(· | τ, S_{g'}))]
        热替换模型权重
```

## 关键设计细节

### Stateful Prompt 结构

- **Part I**：固定系统 prompt（OpenClaw CLI 任务），长 session 后用压缩 prompt 替换（防止 context overflow）
- **Part II**：启动时注入 workspace context 文件（agent 身份、用户 profile、行为原则）
- **Skill injection format**：检索到的 skill 由 `SkillManager.format_for_conversation` 追加到系统消息末尾

### Skill Library 形式

- **每条 skill**：简洁的自然语言行为指令（behavioral directive），如：
  - "在读文件前先验证路径是否存在"
  - "执行破坏性操作前先备份（.bak）"
  - "遵循日期前缀命名规范（如 20260408_*.json）"
- **检索方式**：sentence embeddings 的 cosine similarity，top-k 检索
- **增量累积**：S_{g+1} ⊇ S_g，技能库只增不减

Part II 实验中观察到 skill evolver 合成的技能主要聚焦三类失败模式：
1. 时间格式规范化（自然语言时间表达 → ISO 8601 + 时区）
2. 修改前备份协议（.bak 文件）
3. 命名规范遵从（日期前缀文件名）

## 实验与效果

### Benchmark 设置

**MetaClaw-Bench**：934 道题，44 个模拟工作日

| 部分 | 工作日数 | 题目数 | 任务类型 | 重点 |
|------|---------|--------|----------|------|
| Part I | 30 天 | 346 题（10-15/天） | file-check + multi-choice | 执行可靠性，难度单调递增 |
| Part II | 14 天 | 588 题（42/天） | multi-choice(434) + file-check(154) | 规则遵从，技能蒸馏友好 |

**AutoResearchClaw**：23 阶段全自动科研 pipeline（从 idea → 论文）

**Baselines**：
- Baseline：无任何适应机制
- MetaClaw (Skills)：仅 skill-driven fast adaptation
- MetaClaw (Full)：技能 + RL 权重更新（仅 Kimi-K2.5）

**Backbone**：GPT-5.2、Kimi-K2.5

---

### 主结果表（Table 1：MetaClaw-Bench）

| 模型 | 方法 | Part I Acc. | Part I Compl. | Part II Acc. | Part II Compl. |
|------|------|------------|--------------|-------------|---------------|
| GPT-5.2 | Baseline | 41.1% | 14.7% | 44.9% | 58.4% |
| GPT-5.2 | MetaClaw (Skills) | **44.0%** | **17.1%** | **49.1%** | **67.5%** |
| Kimi-K2.5 | Baseline | 21.4% | 2.0% | 21.1% | 18.2% |
| Kimi-K2.5 | MetaClaw (Skills) | 28.3% | 2.0% | 26.9% | 33.8% |

*Acc. = 平均每题准确率；Compl. = file-check 完成率（所有 checker 断言同时通过的比例）*

**关键数字**：
- Kimi-K2.5 + Skills：Part I 准确率 +32.2% 相对提升
- Kimi-K2.5 + Full：Part I 任务完成率 **8.25×**（2.0% → 16.5%）；Part II file-check 完成率 **+185%**（18.2% → 51.9%）
- MetaClaw (Full) 将 Kimi-K2.5 从 21.4% 提升至 **40.6%**，几乎追平 GPT-5.2 的 baseline（41.1%）

## 结论与启发

### 论文结论

MetaClaw 建立了一个让 deployed LLM agent 通过**正常使用**自主改进的原则性框架：
- 快速、推理时 skill 注入（从失败中蒸馏可复用行为知识）+ 慢速、空闲时 RL 权重更新（精炼模型），两者相互强化
- 代理架构（proxy-based）无需本地 GPU，与现有 agent 和 LLM provider 透明集成
- **当前局限**：空闲窗口检测依赖用户配置，未必适用所有部署环境

### 对 Agent × Skill 自进化方向的启发

1. **技能库是天然的 meta-parameter**：自然语言指令跨任务可迁移，一次失败蒸馏的 skill 可泛化到结构不同的后续任务——这是传统 memory 方法（存原始轨迹）做不到的。

2. **双时间尺度协同是关键架构决策**：Prompt-level 适应（秒级）与 weight-level 优化（小时级）不是竞争关系，而是**互补**——前者提供即时知识注入，后者将隐性规则内化到参数中。两者缺一，效果均不完整（仅 Skills 无法解锁 file-check 完成；仅 RL 没有结构化失败分析则更新低效）。

3. **版本控制是正确性保证**：在在线、异步的 skill 演化场景下，support/query 分离不是可选设计，而是**必要的正确性保证**——不加版本控制会导致陈旧奖励污染，等价于在已修正错误的前提下仍惩罚 policy。

4. **弱模型 + MetaClaw ≈ 强模型 baseline**：Kimi-K2.5（弱）+ Full Pipeline 几乎追平 GPT-5.2（强）的 baseline——说明 MetaClaw 对于"以经济模型服务生产规模"场景具有实际价值。

5. **零停机 skill 注入具有跨域泛化能力**：AutoResearchClaw 的结果证明，为 CLI 任务设计的 skill evolver 可以直接迁移到多阶段科研 pipeline，无需任何领域特定调整——skill 形式（自然语言行为指令）本身就是领域无关的。
