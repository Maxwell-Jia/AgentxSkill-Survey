## 针对的问题
### 痛点
当前 Agent Skill 增强范式 (inference-time skill augmentation) 存在三大根本性限制：
1. **检索噪声**：不相关或误导性技能污染模型上下文
2. **Token 开销**：注入的 Skill 内容在多轮交互中累积，大幅增加推理成本（如 SkillRL 每步 2.21k token）
3. **未真正学习**：模型只是在执行上下文中的技能描述，技能知识驻留在上下文而非参数中——Agent 永远无法自主行为

### 核心差距
比对人类学习过程（从显式指导 → 内化自主执行），当前方法将 Agent 永远锚定在第一阶段。如何让模型通过强化学习将 Skill 内化成参数，实现**真正的零-shot 自主行为**？

---

## 核心思路
> **"Skills at training, zero at inference."** 通过在-context 强化学习(ICRL) + 动态课程(Dynamic Curriculum)，逐步将上下文注入的 Skill 转移到模型参数中，推断阶段完全不依赖外部技能。

## 方法

### 整体框架
![Figure 1: Skill 增强 vs. Skill 内化范式](https://arxiv.org/html/2604.02268v1/x1.png)
**Figure 1**: (a) 传统 Skill 增强范式在推断时检索并注入技能；(b) SKILL0 范式：训练时提供完整技能上下文，推断时完全无技能，通过课程学习实现过渡。

### Skill 的组织方式
- 两层结构：通用 Skill（探索策略、目标追踪）+ 任务特定 Skill（领域知识、动作序列）
- 目录组织：`skills/{task_name}/{skill_category}.md`，每个 Markdown 文件 𝒮_k 存储一组相关技能
- 完整库：SkillBank = {𝒮_k}_{k=1}^N（共 N 个技能文件）
- 训练时不走语义检索，而是按"在线有用性"排序的 m 个技能文件

### 上下文压缩与渲染
为应对多域扩展时的 Token 成本，采用视觉压缩机制：
- 将文本交互历史 h_t 和检索技能 𝒮 映射为紧凑 RGB 图像
- 视觉编码器 Enc 将其压缩为向量表示：𝒱_t = Enc(h_t, 𝒮; c_t)
- **创新**：压缩比 c_t ∈ (0,1] 由策略自生成（与动作 a_t 一起），而非固定超参
- 效果：ALFWorld 上从 text 的 2.21k token/步降至 0.38k token/步（>5倍压缩）

### In-Context Reinforcement Learning (ICRL)
融合技能提示的样本效率与 RL 的探索能力：

**复合奖励设计**：
- 任务成功 + 压缩效率双目标优化
- r̃_t = r_t + λ · r^comp_t
  - 若成功：r^comp_t = ln(c_t)（对数形式反映边际收益递减）
  - 若失败：r^comp_t = 0
- λ ≥ 0 控制任务性能与压缩的权衡

**训练目标**（基于 PPO 框架）：
- 采样 G 条轨迹，计算优势估计（advantage），进行重要性采样约束更新
- KL 散度项强制策略与参考策略接近，防止过度偏离初始行为

### 动态课程机制
**线性衰减的技能预算**：
- 分 N_S 个阶段，第 s 阶段保留的技能数量：M^(s) = ⌈N · (N_S - s) / (N_S - 1)⌉
- 保证每步减少量约 N / (N_S - 1)，实现分布平稳过渡

**两阶段操作流程**：

(a) **离线相关性分组** (Relevance-Driven Skill Grouping)
- 将验证集 N 个子集 {𝒯_k}_{k=1}^N 与对应技能文件关联
- 基于领域对齐定义相关性，为每个 𝒮_k 分配专属验证任务

(b) **在线有用性驱动课程** (Helpfulness-Driven Dynamic Curriculum)
- 每 d 步评估：有技能 vs. 无技能情况下的子任务性能
- 定义有用性指标 Δ_k（两种条件下性能差）
- Filter&Rank&Select：筛选、排序、选择 top-m 技能（m ≤ M^(s)）
- 只保留当前策略仍受益的技能，逐步逼迫模型内化知识

![Figure 6: 有用性动态变化](https://arxiv.org/html/2604.02268v1/x6.png)
**Figure 6**: 技能有用性在各子任务上呈"先升后降"特征。早期策略无法利用技能提示，中期学会对齐，后期课程减少技能预算强制内化，有用性回落。这验证了 ICRL 与课程学习的协同机制。

---

## 关键设计细节

1. **分离训练推断上下文**：关键创新。推断完全无技能迫使模型学习，避免过度依赖
2. **自适应压缩比**：c_t 由策略生成，使模型学会根据任务需求动态调节上下文规模
3. **有用性评估的 d-step 间隔**：平衡评估准确性与训练效率（消融选择 d=10）
4. **线性预算衰减 vs. 随机剪枝**：硬约束下的受控退火，避免分布跳跃；随机选择会导致严重性能崩溃（降 13.7%）

## 实验与效果

### Benchmark
- **ALFWorld**：6 类家务任务，3827 个实例（Pick, Look, Clean, Heat, Cool, Pick2）
- **Search-based QA**：单跳 (NQ, TriviaQA, PopQA) + 多跳 (HotpotQA, 2Wiki, MuSiQue, Bamboogle) 组合

### 主结果

| 方法 | ALFWorld (%) | Search-QA (%) | Token/步 (k) |
|-----|-------------|--------------|-----------|
| Few-Shot + Text | 78.2 | 34.2 | 2.21 |
| AgentOCR (3B) | 78.2 | 34.2 | 0.78 |
| **SKILL0 (3B)** | **87.9** | **40.8** | **0.38** |
| SKILL0 (7B) | **89.8** | **44.4** | 0.47 |
| SkillRL (有技能) | 85.5 | 43.2 | 0.87 |

**关键数字**：
- vs. AgentOCR 基线：+9.7% (ALFWorld), +6.6% (Search-QA)
- 推断时完全零 Skill，仅 0.38k token/步（SkillRL 的 5.8 倍压缩）
- 与 SkillRL (有技能增强) 的 85.5% 相比，SKILL0 在无技能情况下反超至 89.8%

### 消融与分析

**技能预算 M^(s) 的设计**（Fig. 8）：
- [6,3,0] (线性衰减) 最优：87.9% 无技能性能，+1.6% 收益（相比 SkillRL 的 -5.1%）
- [6,6,6] (固定全预算)：推断移除技能时下降 -12.3%，说明过度依赖
- [3,3,3] (固定低预算)：早期探索受限，学习不稳定

**动态课程的三步流程**（Table 2）：
- 去 Filter: 性能 -2.7%（上下文噪声）
- 去 Rank（随机选择）：性能崩溃 -13.7% → 62.9%（充分证明精准筛选的必要性）
- 完整 Filter&Rank&Select：87.9%，是唯一展现正向转移的方法

**有用性评估间隔 d**（Table 3）：
- d=5: 边际收益 +0.3%，但计算开销大
- d=10: 最优权衡选择
- d=20: 评估粗粒度，性能下降

## 结论与启发

### 论文结论
提出首个显式以 Skill 内化为训练目标的 RL 框架。通过 ICRL 与动态课程机制，成功将复杂多步行为和工具使用能力从上下文注入转移到模型参数，实现>0.5k token/步的超低推断成本同时保持性能优势（+9.7%/+6.6%）。

### 对 Skill 内化 / 沉淀方向的启发

1. **从"检索注入"到"内化习得"的范式转变**：推翻当前 skill augmentation 的必要性论证。证明通过受控课程，模型可以真正学习而非仅执行外部知识。

2. **技能生命周期管理的新视角**：Skill 不是一劳永逸的静态资源，而是训练过程中的**临时脚手架**。有用性指标与动态预算勾勒出 Skill 的"兴衰周期"——价值随训练进度衰减。

3. **精准筛选优于盲目堆积**：随机或全量 Skill 导致性能崩溃或过度依赖。基于当前策略状态的有用性评估（不是向量相似度）是 Skill 管理的核心。

4. **推断-训练分布差异的显性处理**：在推断时彻底去掉 Skill 迫使训练过程对齐，避免"提示范式"中的隐性不匹配。这对后续 Skill 进化/元学习有借鉴——不能假设推断时仍有技能可用。

5. **视觉压缩作为中间表示**：Markdown 文本 → RGB 图像 → 视觉编码的压缩链路，为长上下文问题开辟了非文本通道。可探索在 Skill 存储、检索、演化中复用此思路。

6. **课程强度指标的量化定义**：线性衰减的技能预算（每步减少量固定）比固定阈值或随机策略更稳定。暗示 Skill 内化的平稳性取决于上下文变化速率，对 Skill 库演化的增量更新有指导意义。

---

## 补充：与 SkillRL 的比较视角
- **SkillRL**：递归进化，Skill 库不断增长，推断时仍需检索
- **SKILL0**：固定库，通过 RL 内化，推断完全独立
- **选择场景**：SkillRL 适于高度模块化、技能复用率高的任务；SKILL0 适于追求低延迟、自主性强的部署环境
