## 针对的问题

### 痛点

1. **现有 benchmark 仅测"能否用 Skill"，不测"能否造 Skill"**：SkillsBench 等工作验证了外部 Skill 能显著提升模型性能，但均预设了一个静态的已有 Skill 库，未检验 Agent 能否从自身经验中发现、提炼、修复 Skill 并在后续任务中复用。
2. **缺乏 lifelong 视角**：静态评测将每次任务孤立看待，忽略了 Skill 应随任务推进而迭代演化的本质属性——失败经验应被转化为修复补丁，成功经验应被抽象为跨任务的程序性知识。
3. **任务间缺乏共享工作流结构**：已有 benchmark 的任务往往彼此异质，导致 Skill 迁移信号混杂，难以区分是"Skill 泛化能力"在提升还是仅是任务相似性带来的捷径。
4. **Skill 生成与 Skill 修复的能力缺口未被量化**：写出 Skill 与正确修复错误 Skill 是两种本质不同的能力，前者容易，后者是真正瓶颈，但现有工作均未区分评测。

### 核心差距（现有 skill benchmark 都是静态评测，缺 lifelong 视角）

SkillsBench 测"Skill 能否帮助模型"，SWE-Skills-Bench 测"Skill 在软件工程中的实际价值"，SkillUsageWild 测"野外环境下的 Skill 使用行为"——三者均为**静态快照**：Skill 事先提供，任务孤立执行，无跨任务的知识积累机制。SkillFlow 引入**终身序列化协议**，让 Agent 从零开始，在同一任务族内逐步自主构建并迭代 Skill 库，将"Skill 的整个生命周期"（发现→使用→修复→迁移）纳入统一评测框架。

---

## 核心思路

> 构建 166 个任务、20 个工作流族的终身 Skill 评测基准 SkillFlow，通过 Domain-Agnostic Execution Flow（DAEF）保证族内任务共享抽象工作流结构，并设计轨迹驱动与 Rubric 驱动两种 Skill Patch 机制，在顺序执行中测量 Agent 自主发现、修复、积累 Skill 的真实能力。

## 方法

### 评测协议总览

![Figure 1: SkillFlow 概念对比图——静态评测 vs. 终身评测](https://arxiv.org/html/2604.17308v1/x1.png)

**Figure 1** 对比了传统静态 Skill 评测（预设固定 Skill 库，孤立评测每次任务）与 SkillFlow 的终身设定（Agent 从空 Skill 库出发，在任务族内顺序解题，通过 Patch 逐步外化经验，跨任务传递更新后的 Skill 库）。

### 任务族（Task Family）设计

**Benchmark 规模**：20 个任务族，166 个可运行任务，覆盖五大领域——Finance & Economics、Operations & Supply Chain、Healthcare & Life Sciences、Governance & Strategy、Data & Document Intelligence。每族 8-9 个任务，任务间按难度排序。

**任务来源**：从 SkillsBench（18 个）和 GDPval（46 个）中遴选 64 个种子任务，过滤掉过窄、过简或难以自动验证的候选项。

**Skill 池构建**：从四个公开仓库（Anthropic/skills、ComposioHQ/awesome-claude-skills、VoltAgent/awesome-openclaw-skills、SkillsMP 等）收集超过 8,000 个开源 Skill，经安全性、格式与领域相关性过滤，保留 2,318 个用于任务构建期间的参考配对。

### Domain-Agnostic Execution Flow（DAEF）

DAEF 是每个任务族共享的**抽象工作流骨架**，形式化为有向图：

- 节点 $V$：领域无关的操作类型（read / retrieve / compute / detect / output 等受控词汇）
- 边 $E$：操作间的依赖关系
- $\lambda(v)$：领域无关操作类型标签
- $\gamma(v)$：任务特定的具体实体（文件名、字段、业务对象）

DAEF = $\phi(\mathcal{T})$，即去掉 $\gamma$ 后保留的抽象图。同一 DAEF 可实例化为不同领域的多个具体任务，是族内技能迁移的结构性保证。

![Figure 3: DAEF 跨领域对应示意](https://arxiv.org/html/2604.17308v1/x3.png)

**Figure 3** 展示相同 DAEF（抽象工作流节点与依赖结构相同）如何在金融分析、医疗合规、运营规划等不同领域被实例化，从而验证跨领域工作流迁移的可行性。

**任务构建四步流水线**（见 Figure 2）：

![Figure 2: SkillFlow 任务构建流水线](https://arxiv.org/html/2604.17308v1/x2.png)

**Figure 2** 显示构建过程：Step 1（种子任务收集 + Skill 池整理）→ Step 2（嵌入式语义匹配，将相关 Skill 与每个种子配对）→ Step 3（双 Agent 迭代扩展：GPT-5.3-Codex 担任 Architect Agent 生成任务，Claude Opus 4.6 担任 Critic Agent 在真实 Docker 环境中验证并提供修订反馈，循环直至通过）→ Step 4（人工审查：指令泄露、逻辑合理性、环境正确性、难度梯度）。最终 20 个任务族全部通过审查，无致命环境错误。

### Agentic Lifelong Learning 评测协议

**初始状态**：每族评测开始时，Agent Skill 库为空 $\mathcal{S}_0 = \emptyset$。

**第一任务**：Agent 在无任何 Skill 辅助下完成 $T_1$，产生执行轨迹 $\tau_1$，之后接收验证器派生的 Rubric $r_1$（以自然语言描述缺失或错误内容）。

**Skill Patch 生成**：对第一任务，模型从零引导生成初始 Skill：

$$\Delta_1 = \text{Model}_g(\emptyset, \tau_1, r_1), \quad \mathcal{S}_1 = \text{Apply}(\Delta_1, \emptyset)$$

对后续任务 $t > 1$，Agent 携带 $\mathcal{S}_{t-1}$ 执行 $T_t$，获得 $(\tau_t, r_t)$ 后增量更新：

$$\Delta_t = \text{Model}_g(\mathcal{S}_{t-1}, \tau_t, r_t), \quad \mathcal{S}_t = \text{Apply}(\Delta_t, \mathcal{S}_{t-1})$$

**族间重置**：评测目标是族内 lifelong 学习（同一类任务的技能积累），而非跨异质工作流的通用适配，因此每个任务族开始时重置 Skill 库，避免不同工作流信号相互干扰。

### Skill Patch 机制

Patch 使用最小可审计接口，包含三个字段：

| 字段 | 作用 |
|------|------|
| `summary` | 高层次经验总结，描述本轮学到了什么 |
| `upsert_files` | 创建或覆盖 SKILL.md 及辅助脚本 |
| `delete_paths` | 移除过时或低价值 Skill 文件 |

**Trajectory-driven patch（轨迹驱动）**：以 Agent 的实际执行轨迹 $\tau_t$ 为输入，从成功/失败的操作步骤中提炼程序性知识，侧重"从行为中归纳做法"。

**Rubric-driven patch（Rubric 驱动）**：以验证器输出的 Rubric $r_t$ 为输入，聚焦于"输出哪里出错了、应当如何修正"，侧重"从结果反馈中修复逻辑"。两者共同驱动 $\Delta_t$ 的生成，构成互补的知识提取机制。

**Skill 格式**：生成的 Skill 遵循标准结构（SKILL.md + 可选辅助脚本），与 SkillsBench 所定义的 Skill 规范兼容。Patch 生成解耦于 Agent 执行 harness，使用统一的单轮输出格式，保证跨模型评测的可比性。

### Skill 检测机制

任务执行复用各 harness 原生规划策略（Claude Code / Codex CLI / Qwen Coder / Kimi CLI），Skill 使用为自然行为（Agent 自主决定是否读取或调用已有 Skill）。评测只从执行轨迹中提取 Skill 文件的读/调用事件，计算 Skill 复用率（`%use`），不干预 Agent 决策。

## 关键设计细节

**DAEF 节点词汇表**：节点类型来自受控单词词汇（read、retrieve、compute、detect、output 等），保证跨任务的工作流抽象标准化，支持 DAEF 一致性校验。

**难度梯度约束**：每族 8-9 个任务按难度升序排列，DAEF 规范中明确了"允许变化的维度"（数据规模、业务场景、实体类型等）和"不允许变化的维度"（工作流拓扑结构），避免难度差异来自工作流变化而非任务复杂度。

**对照实验设计**：论文额外设置了"历史轨迹直接拼接"控制组（将所有历史交互原文预置为上下文，不生成 Skill），Claude Opus 4.6 在此设定下仅达 51.04%，低于 Vanilla（62.65%）和完整协议（71.08%），排除了"更长上下文"对收益的解释。

**Skill 库规模监控**：通过 `#Skills`（族内最终 Skill 数量）与 `%use`（Skill 复用率）两个指标追踪 Skill 库演化质量，高使用率不等于高效益（Kimi K2.5 使用率 66.87% 但仅提升 +0.60 点）。

## 实验与效果

### Benchmark 配置

- 评测 harness：Claude Code、Codex CLI、Qwen Coder、Kimi CLI
- 模型共 11 个：Claude Sonnet/Opus 4.5/4.6、MiniMax M2.5/M2.7、GPT 5.4/GPT 5.3 Codex、Qwen-Coder-Next、Qwen3-Coder-480B、Kimi K2.5
- 指标：任务完成率 `%comp.`、交互轮数 Turns、单任务成本（USD）、输出 Token 数、Skill 数量 `#Skills`、Skill 复用率 `%use`

### 主结果（Table 1 摘要）

| Agent | 模型 | Vanilla %comp. | Evolve %comp. | Delta | #Skills | %use |
|-------|------|---------------|--------------|-------|---------|------|
| Claude Code | Claude Opus 4.6 | 62.65 | **71.08** | **+8.43** | 1.05 | 45.78 |
| Claude Code | Claude Sonnet 4.5 | 49.40 | 55.42 | +6.02 | 2.55 | 72.89 |
| Claude Code | MiniMax M2.5 | 28.31 | 34.94 | +6.63 | 2.50 | 32.53 |
| Codex CLI | GPT 5.4 | 33.13 | 36.75 | +3.62 | 1.05 | 81.33 |
| Claude Code | Claude Opus 4.5 | 58.43 | 60.84 | +2.41 | 1.50 | 60.84 |
| Kimi CLI | Kimi K2.5 | 55.42 | 56.02 | +0.60 | 1.50 | 66.87 |
| Claude Code | Claude Sonnet 4.6 | 56.63 | 56.63 | +0.00 | 2.55 | 53.01 |
| Codex CLI | GPT 5.3 Codex | 52.41 | 46.39 | -6.02 | 1.10 | 84.94 |
| Qwen Coder | Qwen-Coder-Next | 45.18 | 44.58 | -0.60 | 5.45 | 12.05 |
| Qwen Coder | Qwen3-Coder-480B | 24.70 | 24.10 | -0.60 | 5.20 | 66.87 |
| Claude Code | MiniMax M2.7 | 37.35 | 36.75 | -0.60 | 4.60 | 51.20 |

**Claude Opus 4.6 的 +8.43 点拆解**：在 166 个任务中，完成数从 104 升至 118，绝对增加 14 个任务。按领域看（Figure 6），增益广泛分布于各任务族而非集中于单一领域——Data & Document Intelligence 正迁移最为稳定，Finance & Economics 中则存在较多负迁移案例。成本与 Token 数均同步下降（Cost -7.52%、Output Token -20.33%），是唯一在效果与效率上均实现双优的模型。

![Figure 5: 完成率-成本 Pareto 前沿](https://arxiv.org/html/2604.17308v1/x5.png)

**Figure 5** 展示了 Vanilla 与 Skill Evolution 两种设定下各模型的完成率-成本 Pareto 前沿：Claude Opus 4.6 向右上方移动（更高完成率+更低成本），GPT 5.3 Codex 则明显向左下方退化。

![Figure 6: 各领域分组的 Skill Evolution 完成率增益热力图](https://arxiv.org/html/2604.17308v1/x6.png)

**Figure 6** 按行（模型）× 列（领域组：FE / OS / HLS / GS / DDI）展示增益分布。Claude Opus 4.6 在多个领域均呈绿色正增益；Qwen 系列整体偏灰甚至偏红；Data & Document Intelligence 跨模型正迁移最稳定。

### 六大核心发现

**Finding 1：Opus 4.6 最接近稳定的 Skill 自学习**。Opus 4.6 是唯一能将 Skill 修订转化为库级持续改善（而非仅堆积成功轨迹）的模型，轨迹分析中可见经历失败后对已有 Skill 的有效修复（如电子表格工作流中的缓存值处理）。

**Finding 2：错误 Skill 导致系统性下游漂移**。一旦错误的抽象写入 Skill 库，后续任务会继承同一缺陷逻辑，将局部错误放大为序列级负迁移模式。这是 Skill Evolution 的最高风险。

**Finding 3：统一的高效用 Skill 优于碎片化的任务特定 Skill**。最优 Skill 库通常围绕 1-2 个被反复修订的核心 Skill 组织；性能强的设定最终 Skill 数量往往更少（Opus 4.6 平均 1.05 个），而弱模型会堆积更多 Skill 但收益更差。

**Finding 4：Qwen 和部分 MiniMax 的主要失败模式是 Skill 膨胀（Skill Inflation）**。这些模型几乎为每个任务单独生成一条新 Skill，Skill 数量单调增长（Qwen-Coder-Next 平均 5.45 个），但并未带来对应收益——失败原因是碎片化而非 Skill 缺失。

**Finding 5：Codex 在变体任务合并上更倾向形成单一演化核心**。Codex 的 Skill 库文件数量相对紧凑，但端到端完成率不及 Opus，说明 Skill 压缩能力不等于执行能力。

**Finding 6：模型差距的核心在于修复错误 Skill，而非生成 Skill**。几乎所有模型都能在完成任务后写出某种 Skill，但能否识别错误 Skill 并通过修订让后续任务受益，才是正向迁移的真正分水岭。

### 消融：控制组对比

| 设定 | Claude Opus 4.6 完成率 |
|------|----------------------|
| Vanilla（无 Skill） | 62.65% |
| 历史轨迹直接拼接（对照组） | 51.04% |
| 完整 Lifelong Evolution 协议 | **71.08%** |

历史轨迹拼接不仅无效，反而低于 Vanilla——证明 +8.43 点的收益来自 Skill 的结构化外化与修复，而非更长的原始上下文。

## 与相关 Benchmark 的区别与互补

| 维度 | SkillsBench | SWE-Skills-Bench | SkillFlow |
|------|------------|-----------------|-----------|
| Skill 来源 | 预设 curated Skill | 预设 curated Skill | Agent 自主生成 |
| 评测模式 | 静态快照，无 Skill → 有 Skill | 静态快照，软工任务 | 终身序列，族内累积 |
| Skill 修复评测 | 无 | 无 | 有（Patch 机制） |
| 跨任务迁移 | 无 | 无 | 有（同族任务顺序传递） |
| 工作流结构约束 | 无 | 无 | DAEF 保证族内拓扑一致 |
| 主要贡献 | Skill 设计规律、curated vs. self-gen 对比 | Skill 在 SWE 场景的边际价值 | 终身 Skill 演化的评测协议与失败模式分析 |

三者互补：SkillsBench 回答"有好 Skill 时能帮多少"，SWE-Skills-Bench 聚焦软工特定场景，SkillFlow 回答"Agent 能否自主在时间中积累并修复 Skill"。

## Skill 池与 SKILL.md 规范的关系

SkillFlow 生成的 Skill 与 SkillsBench 定义的标准 Skill 结构完全兼容：每个 Skill 包含 SKILL.md（程序性知识、使用场景、核心指导）+ 可选辅助脚本与资源。Patch 接口的 `upsert_files` 字段直接操作这一文件结构，因此 SkillFlow 生成的 Skill 可直接与 SkillsBench 场景互操作。构建期间使用的 2,318 个参考 Skill 本身也来自遵循该规范的开源社区仓库（Anthropic/skills 等）。

## 结论与启发

### 论文结论

SkillFlow 表明，终身 Skill 演化带来的收益是**选择性**而非普遍的：能力更强的模型（Opus 4.6）将外化经验转化为紧凑、可修复的程序性 Skill，实现持续增益；而弱模型则因技能碎片化（Skill Inflation）、错误强化和创建-复用协调失败而出现退化。Skill 修复能力是区分强弱的核心维度，而非 Skill 生成量。

### 对 Skill 终身学习 / 评测方向的启发

1. **"Skill 生命周期"的完整评测闭环**：传统评测仅覆盖 Skill 的"使用"环节；SkillFlow 证明"发现→修复→迁移"是可以被操作化测量的，为综述"Skill 生命周期"一节提供了完整的评测维度框架（从静态使用扩展到动态演化）。

2. **Trajectory patch 与 Rubric patch 的互补性**：轨迹驱动 patch 从"How"归纳做法，Rubric 驱动 patch 从"What went wrong"修复逻辑。两者对应于"失败复用"方向的两种信号源：成功轨迹的模式提炼 vs. 验证器反馈的精准定位。设计失败复用系统时，应同时接入这两类信号，而非只依赖成功轨迹。

3. **Skill 库的紧凑性比数量更重要**：+8.43 点的最佳模型平均只保留 1.05 个 Skill，而退化模型积累了 5+ 个碎片 Skill。这对"Skill 管理"方向有直接指导意义：需要主动的 Skill 合并、去冗余与淘汰机制，而非无限积累。

4. **错误 Skill 的传播比缺少 Skill 更危险**：一旦错误抽象进入库，后续任务系统性退化——这提示在 Skill 自动生成系统中，写入前的质量校验（类似 SkillsBench 的 Leakage Audit）和写入后的在线检测机制都是必要的。

5. **高使用率 ≠ 高效用**：Kimi K2.5 的 66.87% 使用率搭配 +0.60 点增益揭示了"使用但无实质收益"的陷阱，为 Skill 评测的指标设计提供了重要反例——应将 Skill 使用率与任务成功率的相关性纳入评测指标体系。

6. **DAEF 作为 Skill 迁移的结构化脚手架**：通过显式定义工作流抽象骨架来约束任务族的构建，是一种可推广的 benchmark 设计原则——在需要评测 Skill 跨任务泛化的场景下，共享 DAEF 能避免"任务相似性带来的伪迁移"，使评测结论更可靠。
