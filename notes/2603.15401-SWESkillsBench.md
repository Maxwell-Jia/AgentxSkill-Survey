## 针对的问题

### 痛点

1. **Skill 生态爆炸式增长，但收益存疑**：过去 136 天内已有 84,192 个 skill 在公开生态中被创建，LLM agent 在 SWE 任务上注入 skill 已成常见做法，但实际收益从未被系统量化。
2. **现有 SWE benchmark 不含 skill 条件**：SWE-Bench Verified、TerminalBench 等均未设置"有 skill vs. 无 skill"配对评测；HumanEval 等代码 benchmark 仅覆盖函数级片段，脱离真实项目语境。
3. **现有 skill benchmark 不专注 SWE**：SkillsBench（2602.12670）虽是首个把 skill 作为一等公民的跨域 benchmark，但 SWE 任务仅占 84 任务中的 16 个，且不强调需求驱动的验证逻辑。

### 核心差距

现有工作既没有在**真实 GitHub 仓库 + 固定 commit** 语境下评测 skill 的边际增益，也没有建立从**需求文档 → 确定性验收测试**的完整追溯链。SWE 这一高度结构化领域是检验 skill 可用性的最佳试验场，因为：软件工程有明确的验收标准（unit test）、版本依赖（API 版本差异）和多文件上下文，恰好能暴露 skill 注入的失效边界。

---

## 核心思路

> 构建首个**需求驱动（requirement-driven）**的 SWE skill 评测基准，将 49 个公开 SWE skill 与真实 GitHub 仓库配对，在有 skill / 无 skill 两条件下用确定性单元测试进行配对评测，量化 skill 注入的边际增益与代价。

## 方法

### 评测协议总览

![Figure 3: SWE-Skills-Bench 构建 pipeline 全貌](https://arxiv.org/html/2603.15401v1/x3.png)

**Figure 3**：四阶段 pipeline。Stage 1：从 84,192 个公开 skill 中三轮过滤得到 49 个 SWE skill；Stage 2：每个 skill 配对一个真实 GitHub 仓库 + 需求文档，生成约 565 个任务实例 (R, E, P, S)；Stage 3：将需求文档中每条验收准则转化为可执行的 pytest 单元测试；Stage 4：配对运行有 skill / 无 skill 两条件，计算 ΔP、token overhead 等指标。

### Skill 池构建（49 个公开 SWE skill）

技术路线：从 mcpmarket 类别排行榜出发，经三轮过滤：

1. **类别筛选**：从九大类中保留与 SWE 流程最相关且可单元测试的六类：Developer Tools、Security & Testing、API Development、Data Science & ML、Deployment & DevOps、Analytics & Monitoring。
2. **语义过滤**：排除生成类或主观类 skill，仅保留针对具体 SWE 动作（fix / build / develop）的 skill。
3. **可行性筛选**：排除仓库体量过大或环境搭建成本过高的候选。

最终 49 个 skill 的类别分布：Deployment & DevOps（13）、Analytics & Monitoring（12）、API Development（10）、Data Science & ML（9）、Security & Testing（4）、Developer Tools（1）。

这 49 个 skill 全部来自公开仓库，均为**标准 SKILL.md 格式**（结构化 Markdown 程序性知识包），遵循 Claude Code agent skill 规范——包含 Background、Requirement、File Operations、Acceptance Criteria 四段式模板。

![Figure 2: skill 分布与任务实例分布](https://arxiv.org/html/2603.15401v1/x2.png)

**Figure 2**：(a) 49 个 skill 在六类 SWE 子域的分布；(b) 每个 skill 生成约 10 个任务实例，全量约 565 个。

### 真实 GitHub 仓库配对

每个 skill 对应**一个真实开源 GitHub 项目**，技术栈需与 skill 领域对齐，仓库 **pinned 在固定 commit** 以保证可复现性，并为每个项目构建 Docker 容器（Ubuntu 24.04，CPU-only）。任务实例四元组定义为 (R, E, P, S)：
- R：固定 commit 的 GitHub 仓库
- E：对应容器化运行环境
- P：需求文档（含 Background、Requirement、File Operations、Acceptance Criteria 四部分）
- S（可选）：skill 文档

**Skill 注入机制**：评测前清除仓库内已有的 `.claude/skills/` 目录以防干扰；需要注入时将 SKILL.md 拷入 `~/.claude/` 目录，agent 自动检测并整合；无 skill 条件下该文件缺席。**需求文档 P 中从不引用 skill S**，确保 agent 的行为完全由 skill 文件的物理存在驱动，而非显式指令。

### 需求驱动的验证体系

核心理念：将每条验收准则（Acceptance Criteria）转化为确定性 pytest 测试，从需求到测试可全链路追溯。具体做法：以固定的"专业测试工程师"提示词模板为驱动，要求模型枚举可测行为、构造典型及边界场景、编写含强判别力的 pytest 测试文件（运行实际代码并验证具体输出/结构，而非关键词启发式）。全程不依赖 LLM-as-judge，确保评测确定性。

![Figure 4: 任务实例生成 pipeline](https://arxiv.org/html/2603.15401v1/x4.png)

**Figure 4**：三步生成流程——项目匹配（技术栈对齐 + fixed commit）→ 需求撰写（四部分模板）→ Skill 放置（文件级注入控制）。

### 评测条件对照

| 条件 | Skill 文件 | 描述 |
|------|-----------|------|
| With Skill | ~/.claude/SKILL.md 存在 | Agent 自动发现并使用 |
| No Skill | 无 ~/.claude/SKILL.md | Agent 仅凭基础能力 |

Agent 配置：Claude Code + Claude Haiku 4.5（单一配置，多模型扩展为后续工作）。

## 关键设计细节

### 评测指标体系

论文定义了四个互补指标：

- **Pass Rate**（Pass+, Pass-）：有/无 skill 条件下的任务通过率（二值平均）
- **Skill Utility Delta（ΔP）**：= Pass+ − Pass-，正表示 skill 有效，零表示无关，负表示干扰
- **Token Overhead Ratio（ρ）**：= (C+ − C-) / C-，量化 skill 注入带来的推理成本变化
- **Cost Efficiency（CE）**：= ΔP / ρ，每单位 token 增加换来的通过率提升；负值意味着 skill 要么降性能要么成本远超收益

这套指标将**正确性增益**与**推理效率**解耦分析，是本文方法论设计的核心亮点。

### Skill 样例：risk-metrics-calculation（最优 skill）

表现最佳的 skill 将金融风险计算公式（VaR、CVaR、Sharpe Ratio 等）以程序性知识形式结构化编写，提供具体计算步骤而非通用建议。结果：ΔP = +30.0%，同时 ρ = −34.8%——既提升正确率，又减少 token 消耗。这是"精炼程序知识 + 领域专用 + 无版本冲突"的理想范本。

### Context Interference 的失效机制（以 linkerd-patterns 为例）

![Figure 5: linkerd-patterns skill 导致 context interference 的详细分析](https://arxiv.org/html/2603.15401v1/x5.png)

**Figure 5**：任务要求生成 Server CRD + ServerAuthorization CRD 以强制 gRPC 服务的 mTLS 身份验证。该 skill 包含七个覆盖完整 Linkerd 栈的模板，其中 Template 5 近似匹配任务，但使用 API version v1beta1（任务需要 v1beta3）且混合了 meshTLS / unauthenticated / CIDR 等多种授权模式。

注入 skill 后失效的三阶段链式反应：
1. **Surface Anchoring（表面锚定）**：Agent 照抄模板的 v1beta1 和 HTTP/1，而非适配任务所需的 gRPC context
2. **Hallucination（幻觉）**：Agent 试图调和模板的授权模式与任务需求，杜撰了 Linkerd CRD 中不存在的 `rules/metricsServers` 字段
3. **Concept Bleed（概念渗漏）**：模板的 NetworkPolicy 示例导致 agent 将 Linkerd 层授权与 Kubernetes 原生 NetworkPolicy 混淆，在解决方案中附加了任务未要求的资源

**无 skill 时**：Agent 从第一性原理出发，正确选择 v1beta3 + gRPC + 标准 meshTLS.serviceAccounts 配置。

核心洞察：客观上相关的 skill 内容也可以**降低**性能——当 skill 的覆盖范围（holistic）与任务需求（focused）之间存在粒度错配时，丰富的模板反而扩大了决策空间、占用上下文窗口，并将 agent 锚定在与任务无关的具体参数值上。

## 实验与效果

### Benchmark 配置

- **Skill 数量**：49 个（来自公开仓库，SKILL.md 格式）
- **任务实例**：约 565 个（每个 skill 约 10 个）
- **SWE 子域**：6 个（Deployment & DevOps / Analytics & Monitoring / API Development / Data Science & ML / Security & Testing / Developer Tools）
- **Agent 配置**：Claude Code + Claude Haiku 4.5（CPU-only Docker，Ubuntu 24.04）
- **验证器**：确定性 pytest 单元测试，全程无 LLM-as-judge

### 主结果（聚合层面）

| 指标 | 数值 |
|------|------|
| 全局 Pass-（无 skill） | 89.8% |
| 全局 Pass+（有 skill） | 91.0% |
| 平均 ΔP | **+1.2%** |
| 平均 token overhead（ρ） | +10.5% |
| ΔP = 0 的 skill 数 | **39 / 49（约 80%）** |
| ΔP > 0 的 skill 数（有意义增益） | **7 / 49** |
| ΔP < 0 的 skill 数（性能下降） | **3 / 49** |

### 按 skill 类型分解的五大 Finding

**Finding 1：Skill 注入对通过率的边际增益极为有限（39/49 ΔP = 0）**

39 个 ΔP = 0 的 skill 中：
- **24 个**：Pass+ = Pass- = 100%，即基础模型已完全掌握，skill 冗余
- **15 个**：Pass+ = Pass-，但两者均未达到 100%（如 xlsx 36.4%、turborepo 50.0%），瓶颈在于复杂多步推理、陌生 API 面，而非领域知识缺失——注入更多 skill 无法解决能力本身的短板

**Finding 2：Token Overhead 与性能增益完全解耦**

即使在 ΔP = 0 的 24 个满分 skill 中，ρ 的范围跨度极大：
- 最低：python-resilience，ρ = **−77.6%**（skill 引导 agent 走更直接路径，大幅节省 token）
- 最高：service-mesh-observability，ρ = **+450.8%**（skill 显著拉长推理，但最终结果不变）

ρ 与 ΔP 之间**无系统性相关**：risk-metrics-calculation 同时实现 ΔP = +30%、ρ = −34.8%（理想情形）；而大量 ΔP = 0 的 skill 带来数百percent token 开销。这意味着 skill 改变的是 agent 的推理路径，而非必然改变推理结果。

**Finding 3：仅 7 个 skill 带来有意义的增益（+7.1% 至 +30.0%）**

| Skill 代表 | ΔP | ρ | CE | 特征 |
|-----------|----|----|-----|------|
| risk-metrics-calculation | **+30.0%** | −34.8% | 高 | 金融风险公式，专业程序知识，无版本冲突 |
| tdd-workflow | +7.1% | +78.6% | 0.09（低） | 作为清单强制 agent 覆盖容易遗漏的边界验收步骤，但成本高 |

有效 skill 的共同特征：编码了**高度专业化的程序性知识**（金融风险公式、云原生流量管理、GitLab CI 模式），且知识是 agent 预训练中覆盖不足的领域。tdd-workflow 的机制是"清单效应"——强迫 agent 逐一检查容易遗漏的交付物，代价是 token 消耗大幅提升。

**Finding 4：3 个 skill 因 context interference 主动降低性能（−9.1% 至 −10.0%）**

| Skill | ΔP |
|-------|-----|
| springboot-tdd | **−10.0%** |
| linkerd-patterns | −9.1% |
| django-patterns | −9.1% |

失效根因：每个 skill 均作为对应技术域的**综合参考文档**编写（覆盖架构、惯例、测试策略、错误处理等），当任务只需要其中一小部分知识时，冗余内容导致：(a) 决策空间不必要扩大；(b) 生产级模板将 agent 锚定于不符合任务实际需求的具体参数；(c) skill 文本占用上下文窗口、排挤任务描述与代码库理解。

**含义**：Skill 设计应优先使用**抽象指导模式**，而非带硬编码参数值的具体模板——后者在版本差异或场景差异下会触发 context interference。

### 失败模式总结

| 失败类型 | 典型表现 | 根因 |
|---------|---------|------|
| 能力瓶颈（≠ 知识缺失） | ΔP = 0 且 Pass- 仍低 | 问题在多步推理/API 理解，skill 无法弥补 |
| Skill 冗余 | ΔP = 0 且 Pass- = 100% | 基础模型已内化该领域知识 |
| Context Interference | ΔP < 0 | Skill 覆盖范围与任务粒度错配，触发 surface anchoring / hallucination / concept bleed |

## 结论与启发

### 论文结论

1. **Skill 注入不是通用性能提升器**：80% 的 skill 带来零增益，平均增益仅 +1.2%，远低于生态快速采用所暗示的收益预期。
2. **Skill 的价值高度领域特异性（domain-specific）**：仅在编码了高专业性程序知识、且与目标项目无版本/范围冲突的场景下，skill 才有意义——最高可达 +30%。
3. **Token 开销与正确性完全解耦**：skill 改变推理路径但不必然改变结果；部分 skill 在不提升通过率的同时带来 451% token 增加，性价比极差。
4. **Context Interference 是 skill 注入的结构性风险**：版本错配、范围过宽的综合性 skill 会通过表面锚定、幻觉、概念渗漏三种机制主动降低性能。

### 对 Skill 评测 / 检索 / 自动合成方向的启发

1. **SWE 场景对"Skill 可用性"的强力反事实验证**：本文是目前针对高度结构化专业领域（软件工程）最严格的 skill 增益评测。39/49 = 80% zero gain 的结论，与 SkillsBench（2602.12670）在通用领域 curated skill 上 +16.2pp 的正面结果形成鲜明对比——意味着 skill 的有效性强烈依赖于领域覆盖稀缺度：通用领域（预训练覆盖少）收益大，SWE 等模型已大量见过的领域收益趋近于零。

2. **Skill 自动合成必须关注"粒度-范围对齐"问题**：本文发现的 context interference 揭示了一个 skill 合成的关键约束——如果自动生成的 skill 覆盖范围过宽（综合参考文档模式），在单任务评测时极易因粒度错配导致负收益。合成系统应内嵌任务范围感知（task-scoped），而非仅优化知识覆盖量。

3. **与 SkillsBench（2602.12670）的互补关系**：SkillsBench 是**跨域、多模型、三条件（no/curated/self-gen）**的横向 benchmark，强调 Skill 设计规律（2-3 个最优、精简优于全面文档）；SWE-Skills-Bench 是**单域深挖、需求驱动、配对验证**的纵向 benchmark，强调 skill 注入的边际增益与失效机制。两者在"skill 对于模型已掌握领域收益微弱"这一结论上高度吻合（SkillsBench 软工增益 +4.5pp，本文全局 +1.2%），互为支撑。

4. **与 SkillUsageWild（2604.04323）的互补关系**：SkillUsageWild 关注的是**检索阶段**失效（从 34k 现实 skill 库中检索时性能大幅下降），本文关注的是**注入阶段**失效（即使 skill 已放置在正确位置，版本错配仍导致负收益）。两者合起来描述了 skill 全链路失效的完整图谱：检索漂移 → 注入干扰 → 能力瓶颈。

5. **Skill 评测需要"deterministic + requirement-driven"双保证**：本文的方法论贡献在于将需求验收标准与 pytest 测试全链路绑定，避免了 LLM-as-judge 的不确定性。这一设计对综述中"评测基建"子方向有直接参考价值——未来 skill 评测框架应向需求驱动靠拢，而非依赖主观评分。

6. **Skill 设计原则的实操指导**：本文为 skill 作者提供了一条清晰的负面规则——**不要用生产级综合模板（opinionated, hard-coded values）作为 skill 内容**，应优先使用抽象指导模式。这与 SkillsBench 发现的"精简 Skill 优于全面文档"（Detailed > Comprehensive）一致，从不同维度共同支持"精炼优于全面"的 skill 设计原则。
