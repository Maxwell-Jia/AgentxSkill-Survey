## 针对的问题（Motivation）

**核心矛盾**：现代科学生态系统中存在大量程序性知识（repositories、APIs、脚本、notebooks、文档、数据库、论文），但这些知识高度碎片化、存在于异构资源中，LLM agent 无法直接检索、执行、验证或组合成可靠工作流。

现有痛点：
- 大多数 agent skill 仍是**手工构建**的，覆盖面窄、难以扩展到多样科学领域
- 工具层（tools / MCP）只暴露可调用接口，缺乏协调多步决策所需的**程序性指导**（procedural know-how）
- 现有系统往往依赖自定义 wrapper 和预定义工具，难以跨域适配

**核心问题**：能否将隐藏在异构领域资源中的科学程序性知识，**自动转化**为能支撑可靠、可扩展科学 agent 的 skill？

---

## 核心思路（一句话概括）

SkillFoundry 以"领域知识树"为控制结构，驱动"采矿→提取运营契约→编译验证→迭代精炼"的**闭环自演化 pipeline**，将异构科学资源持续转化为结构化、可执行、经过验证的 agent skill 库。

## 方法（Framework）

### 整体架构图

![Figure 1: SkillFoundry 框架概览。从领域知识树出发，框架挖掘与分支相关的资源，提取结构化候选 skill，通过多级测试验证，扩展树并剪除冗余或低价值叶节点。](https://arxiv.org/html/2604.03964v1/fig-main/SkillFoundry_concept.png)

### 自演化 Pipeline（6 步闭环）

SkillFoundry 的核心是一个以领域知识树为控制结构的闭环过程：

```
Step 1: 领域知识树（Domain Knowledge Tree）
    ↓
Step 2: 树引导的资源挖掘（Tree-Guided Resource Mining）
    ↓
Step 3: Skill 提取（Extract Operational Contract → Compile Skill Package）
    ↓
Step 4: 多级 Skill 测试（Execution / System / Synthetic-data Testing）
    ↓
Step 5: 树扩展（通过验证的 skill 作为新叶节点加入）
    ↓
Step 6: 树精炼（冗余/低价值 skill 合并或剪除）
    ↓ 循环回 Step 1
```

### 各步骤详解

**Step 1 - 领域知识树构建**

- 将目标领域表示为有根树 T=(V,E)，内部节点表示领域/子领域，叶节点表示可操作的 skill 目标
- 树不是静态分类学，而是**自适应挖掘的控制结构**——每个节点存储轻量状态（关联资源、已有 skill、验证状态）
- 系统优先选择"资源丰富但已验证 skill 覆盖薄弱"的分支，而非均匀搜索
- 初始树手工构建（覆盖计算生物学及相邻科学软件领域），随库增长持续修订

**Step 2 - 树引导的资源挖掘**

- 按分支优先级检索权威资料：官方文档、维护良好的 repositories、包参考文档、工作流、notebooks、方法论文
- 实现为 staged loop：`tree_check → resource_search → skill_build → skill_test → refresh`

**Step 3 - Operational Contract 提取与 Skill 编译**

从检索到的资源中提取**运营契约（operational contract）**，并编译成可复用的 skill package。

## 关键设计细节

### Operational Contract（运营契约）是什么？

每个 skill 的核心规格，包含以下字段：

| 字段 | 含义 |
|------|------|
| **Task Scope** | 该 capability 完成什么任务 |
| **Inputs / Outputs** | 预期的输入输出格式 |
| **Dependencies** | 环境与软件依赖 |
| **Execution Steps** | 执行步骤 |
| **Provenance** | 来源链接（原始资源） |
| **Follow-up Guidance** | 后续使用指导 |
| **Example Invocations / Tests** | 示例调用或测试命令 |

编译后的 skill package 不只是文本描述，还包含：
- 人类可读的 skill 规格
- 机器可读的元数据
- 来源记录
- 可执行脚本
- 示例资产
- 仓库级测试（当相关资产可解析时）

### 如何验证（Multi-stage Validation）？

**三层互补测试**：

| 测试类型 | 适用场景 | 检查内容 |
|----------|----------|----------|
| **Execution Testing** | 所有 skill | 在声明的契约下能否运行 |
| **System Testing** | 依赖 SLURM 等集群基础设施的 skill | 真实部署环境下的可行性 |
| **Synthetic-data Testing** | 真实资源不可用/昂贵/不稳定时 | 契约完整性（contract completeness）+ 行为稳定性（behavioral stability） |

验证不通过 → 进入**修复循环（repair loop）**：诊断失败、修订实现/测试面、重新运行。

**两层更深的质量检查**（appendix 详述）：
- Layer 1（正确性导向）：执行测试，确保最低可执行性
- Layer 2（实用性导向）：skill vs. no-skill 比较，若优势薄弱则进入优化循环

### 新颖性审查与树更新

- 同时对比本地库和外部 skill 生态（SkillHub、SkillSMP）
- 基于**主题定位、来源、范围**判断，不仅仅是词汇相似度
- 结果处理：
  - 全新能力 → 接受为新叶节点
  - 与已有 skill 互补 → 合并
  - 完全重复 → 丢弃
  - 反复失败 → 分支标记为部分覆盖，可触发剪枝

### 并行与受控状态变更

- 分支级探索可并行，但共享全局状态（注册表、报告、站点摘要）**只由 refresh 阶段写入**，防止并发 worker 写入不一致状态

### 实现细节（模型分配）

| 阶段 | 模型 | 推理力度 |
|------|------|----------|
| tree_check、skill_build、refresh、design_skill | GPT-5.4 | medium |
| resource_search | GPT-5.4 | **high**（最推理密集的阶段） |
| skill_test、repair、benchmark、novelty_check | GPT-5.4-mini | medium |

## 实验与效果

### Skill 库概览

- 当前库快照：**286 个 skill**，覆盖 **27 个领域**、**254 个子领域**，来自 **394 个资源**
- 所有保留的 skill 均通过必要的验证测试
- **新颖性**：71.1% 的 mined skill 与 SkillHub、SkillSMP 中已有 skill 不同，属于全新 skill

![Figure 2: Skill 库组成与 pipeline 运行时间。(a) 286 个 mined skill 在领域和子领域上的分布。(b) 固定资源挖掘预算下各主要 pipeline 阶段的平均运行时间。](https://arxiv.org/html/2604.03964v1/x1.png)

运行时间分析：Skill 提取（extraction）是最大开销，其次是资源挖掘（resource mining），测试和树更新开销相对较小。

### MoSciBench 基准测试

**基准设置**：MoSciBench 包含 6 个数据集，涵盖气候科学、生物医学工程、化学信息学、健康心理学、群体基因组学、地球科学。同一 coding agent，有/无 SkillFoundry skills 对比。

**主结果表格（Table 1 重建）**：

| 数据集 | 无 Skill Repo-Acc | 有 Skill Repo-Acc | 无 Skill Paper-Acc | 有 Skill Paper-Acc | Exec |
|--------|-------------------|-------------------|--------------------|--------------------|----|
| health_spa | — | ↑ 显著提升 | — | ↑ 显著提升 | 100% |
| pop_genetics | — | ↑ 显著提升 | — | ↑ 显著提升 | 100% |
| cyclone | — | ↑ 显著提升 | — | ↑ 显著提升 | 100% |
| massspecgym | — | ↑ 提升 | — | ↑ 提升 | 100% |
| terra | — | ↑ 提升 | — | ↑ 提升 | 100% |
| nurse_stress | — | 持平 | — | 持平 | 100% |
| **平均** | **61.19%** | **66.73%** | **43.85%** | **53.05%** | **100%** |

> 注：nurse_stress 数据集基线本已很强，故无提升；代码执行成功率在所有设置下始终保持 100%，说明收益来自更好的任务完成和科学推理，而非更容易执行。

**关键数字**：
- Repo-Acc：61.19% → **66.73%**（+5.54 pp）
- Paper-Acc：43.85% → **53.05%**（+9.20 pp）
- 6 个数据集中 5 个提升，1 个持平

### Cell Type Annotation（细胞类型注释）

**任务**：空间转录组数据的细胞类型注释（MERFISH 数据集，228,635 个细胞，238 个基因，3 个样本）。

**模式**：SkillFoundry 接受任务 prompt，检索资源、构建候选 skill、测试，然后评估有/无该 skill 的 Codex，以及 SpatialAgent（领域专用 agent）。

方法对比：
- **Vanilla Codex**：PCA + k-means 聚类，marker-set 打分
- **Codex + SkillFoundry Skill**：marker-signature pipeline（PCA + Leiden 聚类 + 质心相似标注 + 聚类级规则精炼 + 局部邻域验证）
- **SpatialAgent**：从 CZI 数据库检索外部 scRNA-seq 参考数据集，用 Harmony 迁移标签

![Figure 3: 各方法细胞类型注释 UMAP。对比真实标签、Vanilla Codex、Codex+SkillFoundry、SpatialAgent。Coverage = 被分配到 8 大类之一的细胞比例；Accuracy = 预测标签与真实标签匹配的比例。](https://arxiv.org/html/2604.03964v1/fig-main/umap_broad_celltype_comparison.png)

**结果数字**：

| 方法 | Coverage | Accuracy |
|------|----------|----------|
| Vanilla Codex | 81.1% | 68.5% |
| Codex + SkillFoundry | **99.2%** | **82.9%** |
| SpatialAgent | 100.0% | 87.1% |

> SpatialAgent 略优，但它使用了外部参考数据集；Codex+SkillFoundry 不依赖外部参考，性能大幅超越 Vanilla Codex。

### scDRS 工作流

**任务**：将 GWAS summary statistics 与 scRNA-seq 数据结合，识别与疾病相关的细胞/细胞类型（多步骤统计遗传学工作流）。

**设置**：Biomni（通用生物医学 agent）有/无 SkillFoundry skill，各跑 3 个 replicate，由统计遗传学专家盲评。

- **定性评分**（0-7 分，7 项评估标准各 1 分）：细胞级/细胞类型级/基因级分析、FDR 校正、识别 chondrocytes 为最相关细胞类型、细胞类型内异质性分析、输出文件完整性
- **定量指标**：细胞级疾病评分与专家参考结果的 RMSE

![Figure 4: Biomni 在 scDRS 工作流上的定量（RMSE）和定性评估，有/无 SkillFoundry skills。(a) 3 个 replicate 的 RMSE。SkillFoundry 整体降低误差，2 个 run 与专家输出完全匹配。(b) 同 replicate 的专家定性评分，7 分为满分。SkillFoundry 产生唯一满足所有定性标准的 run。](https://arxiv.org/html/2604.03964v1/x2.png)

**关键数字**：
- 平均 RMSE：0.11 → **0.02**（降低约 5 倍）
- 定性：仅 Biomni+SkillFoundry 的某个 run 达到满分 7/7
- 2/3 个 skill 增强 run 与专家完全匹配（vs. Biomni 单独的 1/3）

## 消融与分析

论文未设置传统消融实验，但通过以下分析支撑设计选择：

**Skill 库内部验证**：
- 71.1% 新颖性率验证了从异构资源挖掘 skill 的必要性（手工库无法覆盖）
- 所有保留 skill 均通过 execution + synthetic-data testing，证明验证流程有效筛选

**任务级对比设计**：
- MoSciBench：同一 agent，仅变量为是否提供 SkillFoundry skills
- Cell type annotation：Codex vs. Codex+Skill vs. 领域专用 agent（SpatialAgent）
- scDRS：Biomni vs. Biomni+Skill，3 次 replicate，盲评

**局限性**（作者自述）：
1. 当前覆盖仍有限，只是演化库的快照
2. 大多数 skill 仅经内部测试验证，未经广泛下游评估
3. 评估范围集中在少数领域和任务

## 结论与启发

### 论文结论

SkillFoundry 证明了：
1. **自动挖掘**异构科学资源，可产生大规模、高度新颖、内部有效的 skill 库（71.1% 新颖）
2. Mined skills 在标准 benchmark（MoSciBench 5/6 数据集）和真实科学工作流（细胞注释、scDRS）上均可提升 agent 性能
3. 同一框架可按需设计**任务特定 skill**，并将其转移给外部 agent（Biomni），无需修改 agent 本身

### 对 Skill 自动合成/沉淀方向的启发

**1. 领域资源→Skill 的转化路径已经可行**

论文验证了"领域知识树 + 树引导挖掘 + 运营契约提取 + 闭环验证"这一完整 pipeline 的可行性。关键不在于有多少 skill，而在于是否有**可信的验证机制**——execution testing + synthetic-data testing 提供了可落地的最低执行性标准。

**2. Operational Contract 是 skill 质量的核心抽象**

Contract 把"任务范围、输入输出、依赖、执行步骤、来源、示例"打包为可被 agent 发现和执行的结构化单元。这比 raw tool（只暴露接口）更富语义，适合复杂多步工作流。

**3. 树结构是扩展性的关键**

用领域知识树同时作为**搜索先验**和**库状态摘要**，使系统能优先填补"资源丰富但 skill 薄弱"的缺口，避免随机漫游和重复采集。这对大规模自演化特别重要。

**4. 新颖性审查不可跳过**

28.9% 的候选 skill 与已有库重叠，若不做去重，库会被冗余污染。基于"主题定位+范围"的语义去重（而非仅词汇相似）是维护库质量的必要步骤。

**5. Task-specific Skill 合成的迁移价值**

SkillFoundry 生成的 skill 可直接作为外部资源注入其他 agent（Biomni），无需修改 agent 架构。这种"skill 作为可移植知识胶囊"的设计，对构建通用科学 agent 基础设施有重要参考意义。

**6. 与 Claude Agent Skills 生态的关联**

论文明确参考了 Anthropic 的 Claude Agent Skills 框架（skill.md 格式），SkillFoundry 的 skill 概念与之对齐。这意味着自动挖掘的 skill 可直接接入 Claude 生态，为领域 agent 快速注入专业程序性知识。
