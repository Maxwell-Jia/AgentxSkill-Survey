## 针对的问题（Motivation）

**如何在不更新模型参数的前提下，从 Agent 执行轨迹中自动蒸馏出可迁移的 Skill？**

现有方案面临两类缺陷：

1. **纯参数知识生成 Skill（Parametric）**：仅凭 LLM 预训练知识写出的 Skill 不含任务域特有的陷阱信息，效果基本等同于"无 Skill"。

2. **在线顺序更新（Online Sequential）**：已有并发工作（AutoSkill、XSkill、EvoSkill 等）将 Skill 随传入轨迹逐条更新，存在两大问题：
   - **Skill 碎片化**：为了放置每条轨迹的局部经验，持续创建细粒度新 Skill，导致检索困难；
   - **顺序过拟合（Sequential Drift）**：后续每次更新依赖前一版本 Skill，早期低质量更新会扭曲整体方向，且每条轨迹视野有限，难以归纳通用规律。

此外，人工编写的高质量 Skill（如 Anthropic 官方 xlsx skill）虽对大模型效果显著，却存在**跨模型尺度不可移植**的问题——同一份 Skill 对 122B 模型提升 +20 pp，对 35B 模型却产生 -9 pp 的负效果。

---

## 核心思路（One-liner）

**像人类专家写作 SOP 一样**：先并行分析大量执行经验，再通过归纳推理将局部轨迹知识层次化地整合为单一、全面、无冲突的 Skill 目录——无需参数更新，无需检索模块，跨模型、跨任务域可直接迁移。

## 方法（Framework）

![Figure 2: Trace2Skill 三阶段 Pipeline 概览](https://arxiv.org/html/2603.25158v1/figures/trace2skill_framwork.png)

**Figure 2**：Stage 1 用初始 Skill S₀ 让冻结 Agent πθ 并行生成成功/失败轨迹池；Stage 2 并行多 Agent 独立分析各条轨迹并提出 Patch；Stage 3 将全部 Patch 层次化归并为最终 Skill S*。

### Stage 1：轨迹生成（Trajectory Generation）

- 以 ReAct 为 Agent 框架，将初始 Skill S₀ 预置到系统提示中
- 让冻结 Agent πθ 在 **演化集** D_evolve 上并行运行，产生轨迹集合 T = {τ₁, ..., τN}
- 按正确性划分：T⁻（失败轨迹）和 T⁺（成功轨迹）
- 实践规模：200 条轨迹、50+ 步/条，122B 模型 < 2 GPU 小时

### Stage 2：并行多 Agent Patch 提案（Parallel Multi-Agent Patch Proposal）

每条轨迹分配给一个独立分析 Agent，读取 S₀ 的**冻结副本**，独立输出一个技能补丁 pᵢ：

```
p_i = A⁻(S₀, τᵢ)   if τᵢ ∈ T⁻  （Error Analyst）
p_i = A⁺(S₀, τᵢ)   if τᵢ ∈ T⁺  （Success Analyst）
```

**Error Analyst（A⁻）**：
- 实现为多轮 ReAct 风格的 Agentic Loop
- 可以检查完整执行 trace、读取输入/输出文件、对比 Agent 答案与 Ground Truth
- 迭代缩小根因范围后才提出 Patch；若无法验证根因则该轨迹被排除
- **质量门控**：每个 Patch 都锚定在经过验证的失败原因上

**Success Analyst（A⁺）**：
- 单轮调用：清理轨迹 → 识别可泛化成功模式 → 提出 Patch
- 成功轨迹无需交互式诊断，单次调用足够高效

**关键设计**：所有分析 Agent 对其他 Agent 的 Patch 无可见性，保留轨迹级别观察的完整多样性，避免过早收敛。

### Stage 3：无冲突 Patch 整合（Conflict-Free Patch Consolidation）

对 Patch 池 P = P⁻ ∪ P⁺ 执行**层次化归并**：

```
p^(ℓ+1) = M(πθ, S₀, {p₁^(ℓ), ..., p_B^(ℓ)})
```

- 归并层数 L = ⌈log_{B_merge}|P|⌉，批次大小 B_merge = 32
- 每层 Merge Operator M 执行：去重、冲突解决、保留独特洞察
- 整个 Pipeline 自包含：同一 LLM 负责轨迹生成、Patch 提案、Merge

**三重确定性防护**：
1. 引用不存在文件的 Patch 直接拒绝
2. 对同一文件同一行范围的编辑标记为冲突并搁置
3. 最终 Skill 经格式校验

**归并即归纳推理**：Merge Operator 被明确指示识别"跨多个独立 Patch 反复出现的模式"——出现频率低的 Patch 视为特例丢弃，高频 Patch 被提升为通用 SOP，这是 Skill 获得可迁移性的核心机制。

## 关键设计细节

### Trajectory-Local vs. Transferable 的区分

| 维度 | Trajectory-Local（局部）| Transferable（可迁移）|
|------|------------------------|----------------------|
| 来源 | 单条轨迹的观察 | 跨多条轨迹的归纳模式 |
| 特征 | 可能含模型特有 quirks、任务特有上下文 | 反映系统性任务属性 |
| 处理 | 由各 Analyst 独立提出 Patch | 由层次 Merge 过滤提升为 SOP |
| 存放 | pᵢ（单条 Patch） | S*（统一 Skill 目录）|

高频模式 → 主 SKILL.md；低频 niche quirks → references/ 子目录（自动复现人类专家的渐进披露层次）。

### Skill 数据结构

```
S = (M, R)
  M: SKILL.md          # 自然语言过程性知识：何时用、步骤、已知陷阱
  R = {scripts,         # 可执行脚本（确定性子任务）
       references/,     # 上下文/领域参考（低频 edge cases）
       assets}          # 其他资源
```

### 两种演化模式

| 模式 | 初始化 S₀ | 目标 |
|------|-----------|------|
| **Skill Deepening（深化）** | 人类专家编写的 Skill | 在已有高质量基础上进一步精炼 |
| **Skill Creation（从零创建）** | LLM 参数知识生成的草稿 | 从性能中性出发，完全由轨迹驱动构建有效 Skill |

## 实验与效果

### Benchmark 与基线

- **主域**：SpreadsheetBench-Verified（400 样本，200 演化 / 200 测试）
  - 指标：Vrf（精确验证）、Soft（子问题通过率）、Hard（全通过率）
- **OOD**：WikiTableQuestions（Wikipedia 表格 QA，格式转为 xlsx）
- **泛化域**：DAPO-Math（数学推理）、DocVQA（视觉问答）

**基线 Skill**：
- Human-Written：Anthropic 官方 xlsx skill（人类专家精心编写）
- Parametric：Qwen3.5-122B 凭参数知识生成的 xlsx-basic

**模型**：Qwen3.5-122B-A10B 和 Qwen3.5-35B-A3B（MoE 架构，开源）

## 消融与分析

### 并行整合 vs. 顺序编辑（Table 4）

| 条件 | 122B: Vrf | 122B: Soft | 122B: Hard | 35B: Vrf | 耗时 |
|------|----------|-----------|-----------|---------|------|
| Seq-B=4 | 59.00 | 40.63 | 20.63 | 26.17 | ~15 min |
| Seq-B=1 | 61.83 | 44.40 | 25.40 | 26.00 | ~60 min |
| **Parallel（ours）** | **65.83** | **46.60** | **27.43** | **27.00** | **~3 min** |

并行整合对 122B 模型在所有指标均优于顺序更新，速度快 **20×**（vs Seq-B=1）。  
根本优势：所有 Patch 来自同一冻结初始 Skill，避免顺序漂移；Merge 同时对全量轨迹归纳，而非对最近更新的归纳。

### Trace2Skill vs. 检索记忆库（Table 5，Deepening +Combined vs. ReasoningBank）

| 条件 | 122B: Vrf | 122B: Soft | 122B: Hard | 35B: Vrf |
|------|----------|-----------|-----------|---------|
| ReasoningBank | 56.00 | 40.10 | 21.30 | 20.50 |
| **+Combined（ours）** | **69.83** | **47.17** | **29.53** | **29.67** |

+Combined 全面领先 ReasoningBank：+13.8 pp Vrf（122B）、+9.2 pp Vrf（35B）。  
原因：检索质量依赖 query 与 memory key 的表面相似度；蒸馏的 Skill 无需检索、直接在系统提示中集成、已经过冗余过滤和泛化抽象。

### Agentic Error Analysis vs. 单次 LLM 调用（Table 6）

| 条件 | 122B Deep. Avg | 122B Creat. Avg | 35B Deep. Avg | 35B Creat. Avg |
|------|---------------|----------------|--------------|----------------|
| +Error（agentic，ours）| **40.75** | **27.84** | **36.04** | **39.06** |
| +Error LLM | 28.58 | 27.08 | 32.83 | 25.76 |

Agentic 分析在 4 组中均领先，Avg 差距 +0.8 ~ +13.3 pp。  
根本原因：单次 LLM 调用无法访问文件和 GT，在 57% 含解析错误的案例中误将解析失败归为根因（Agentic 仅 14%），且存在幻觉失败原因的情况。

### 学到的可泛化 SOP（322 条 Patch 分析）

| SOP 主题 | 来源 Patch 数 |
|----------|-------------|
| 公式重算与写回验证（recalc.py + data_only=True）| 178/323 |
| 工具选择：openpyxl 优于 pandas.to_excel() | 177/323 |
| 显式读回验证（写入后重开文件确认值）| 138/323 |
| 结构编辑安全（降序删行、写前备份）| 53/323 |
| 低频边缘案例 → references/ 子目录 | 13 个参考文件 |

系统**自动**复现了人类专家分层撰写 SOP 的惯例：通用规则进 SKILL.md，特例进 references/。

## 结论与启发

### 核心结论

1. **经验可蒸馏为可迁移 Skill**：轨迹局部知识经过并行归纳后，能跨模型尺度（35B 写的 Skill 帮助 122B 提升 57.65 pp）和跨任务域（电子表格 Skill 迁移到 WikiTQ）发挥作用。

2. **全局并行优于局部顺序**：先积累广泛经验、再统一归纳（类似人类专家工作流），比在线逐条更新的效果更好、速度更快。

3. **单一紧凑 Skill 优于检索记忆库**：不需要检索模块，推理时无检索损耗，且归纳过滤后的 Skill 比原始 episode memory 更抽象、更通用。

4. **开源小模型足够**：35B 开源模型可以完成完整的 Skill 演化流程，不依赖专有大模型。

5. **任务执行能力 ≠ Skill 创作能力**：后者依赖对失败模式的归纳推理，是独立维度的能力。

### 对 Skill 自动沉淀方向的启发

- **批量离线 > 在线单条**：单条轨迹的信噪比低，批量并行分析后归纳才能将"局部经验"升华为"通用规律"；
- **Declarative Skill as Knowledge Artifact**：将 Skill 定义为可读、可共享的声明式文档（非检索索引、非 LoRA 权重），使其天然可跨 Agent 框架移植；
- **Agentic 根因分析是瓶颈**：Error Analyst 的质量直接决定 Patch 的可迁移性，值得重点投入；
- **层次化 Skill 结构**：SKILL.md（通用 SOP）+ references/（特例）的两层设计值得在其他 Skill 系统中采纳；
- **局限**：当前无法量化单条 Patch 的因果贡献，也无法追踪推理时各 Skill 段落的使用频率——这是下一步 Skill 自动裁剪与优化的关键入口。
