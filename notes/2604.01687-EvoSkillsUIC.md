## 针对的问题

### 痛点

1. **Skill 生成依赖人工编写，成本高且质量不稳定**：当前 Agent Skills 绝大多数由领域专家手动撰写，耗时耗力，难以规模化，且无法系统保证质量。
2. **人机认知错位（human-machine cognitive misalignment）**：人类专家按照对自身直觉的理解编写工作流程，但 LLM Agent 在处理上下文、推理和执行约束时遵循不同的认知逻辑。SkillsBench 实验证实，在 Natural Science 等领域，人工 Skill 甚至导致 Agent 性能下降。
3. **一次生成多文件 Skill bundle 天然不可靠**：工具（Tool）是单一自包含函数，而 Skill 是包含 SKILL.md 指令、可执行脚本、领域参考资料的多文件结构包，一次性生成的覆盖率和逻辑连贯性难以保证。
4. **缺乏可用的密集反馈信号**：在真实世界中，Ground-Truth Oracle 只返回不透明的通过/失败二值信号，不包含任何诊断细节，无法指导 Skill 的定向修复。

### 核心差距

现有自进化方法（Voyager、Live-SWE-Agent、SkillCraft 等）本质上针对单一函数/API 的生成，存在"工具-技能鸿沟"（tool-skill gap）：既不能构建多文件 Skill 结构包，也无法在没有 Ground-Truth 反馈的情况下有效进化。EvoSkill（单数版，Alzubi 等）则依赖 Ground-Truth 失败诊断，在真实世界信号稀缺场景下适用性受限。

### 与已有 EvoSkill（单数）的根本区别

| 维度 | EvoSkill（单数）2603.02766，Sentient + Virginia Tech | EvoSkills（复数）2604.01687，UIC + MBZUAI + McGill + Columbia |
|------|------|------|
| 核心架构 | Executor / Proposer / Skill-Builder 三 Agent + Pareto 前沿治理 | Skill Generator / Surrogate Verifier 两方协同演化 |
| 进化信号 | 来自有标注训练集的失败样本（需要 Ground-Truth 标注） | Surrogate Verifier 自主合成测试断言，无需 Ground-Truth 内容 |
| 反馈类型 | 失败轨迹分析 → Proposer 提案 → Skill-Builder 物化 | 代理奖励信号 + 结构化诊断反馈，失败时 Generator 定向修复 |
| Skill 产物 | SKILL.md + 可选 Python/TS 脚本（结构相近） | SKILL.md + 可执行脚本 + 领域参考（强调多文件 bundle 的整体可靠性） |
| 防过拟合机制 | Pareto 前沿 + 验证集过滤 | Oracle 仅返回不透明 Pass/Fail 位，Verifier 信息隔离防作弊 |
| Benchmark | OfficeQA / SealQA / BrowseComp（问答类） | SkillsBench（87 任务 / 11 领域，确定性验证器） |
| 核心结论 | +7.3 pp（OfficeQA 精确匹配），+12.1 pp（SealQA） | +40.5 pp pass rate，并超越人工 Skill +17.6 pp |

技术路线完全不同：单数版是"失败驱动 + Pareto 前沿"，复数版是"Generator-Verifier 双方协同演化 + 代理验证器"。

---

## 核心思路

> 将 Skill Generator 与 Surrogate Verifier 置于协同演化循环中：Generator 迭代生成多文件 Skill bundle，Verifier 在信息隔离条件下独立合成测试断言并提供结构化诊断反馈，二者交替优化，在不依赖 Ground-Truth 内容的前提下实现 Skill 质量的持续提升。

## 方法

### 整体框架

![Figure 3: EvoSkills 协同演化框架总览](https://arxiv.org/html/2604.01687v1/x2.png)

*Figure 2：5 轮演化过程中 Skill 质量提升曲线。EvoSkills 在 5 轮内超越人工 Skill。*

![Figure 3: 协同演化框架结构](https://arxiv.org/html/2604.01687v1/x3.png)

*Figure 3：EvoSkills 协同演化框架总览。Skill Generator 与 Surrogate Verifier 通过迭代细化协同演化。Verifier 提供结构化失败反馈驱动 Skill 改进，Ground-Truth Oracle 仅返回不透明 Pass/Fail 信号，触发测试升级并确保严格信息隔离。*

任务环境被形式化为 POMDP（M = ⟨X, A, T, O, Ω, R⟩），Agent 目标是发现最优 Skill S* = argmax_S J(S)，其中 J(S) 是在 Skill S 指导下运行策略 π_θ 到达终态时的期望奖励。由于 Ground-Truth 奖励 R 仅返回不透明信号，EvoSkills 引入代理奖励 R̃(x, V)（代理 Verifier 测试通过率）作为可优化的代理目标，通过交替优化（Skill 细化 + 测试升级）逼近真实目标。

### Skill Generator

Generator 使用持久对话上下文 C，初始化为（任务指令 I，领域无关的元技能 S_meta），S_meta 是一个"如何创建 Skill"的引导器。每次迭代，Generator 读取当前 Skill S(i) 与累积 Verifier 诊断 F(i,j)，生成改进版本：

S(i+1) ~ π_θ(· | S(i), C(i+1))，其中 C(i+1) = C(i) ⊕ F(i,j)

Generator 执行 Skill 并产出输出文件 x(i) = Φ(S(i), E)，传递给 Surrogate Verifier 评估。上下文容量上限 β = 0.7（防止 LLM 上下文溢出），超限则停止当前演化。

### Surrogate Verifier

Surrogate Verifier 的核心设计目标：在没有 Ground-Truth 测试内容的情况下，为 Generator 提供密集、可操作的诊断反馈。

**为什么用 Surrogate 而非实际 Oracle 验证**：Ground-Truth Oracle 每次调用成本高昂，且仅返回不透明二值信号，无法指导 Generator 的定向修复。若每次迭代都调用 Oracle，反馈稀疏且演化效率极低；Surrogate 在"代价低、信息丰富"这两个维度上互补于 Oracle。

**信息隔离防确认偏差**：Verifier 运行于完全独立的 LLM 会话 π_θ^V，仅接收任务指令 I 和输出文件 x(i)，不继承 Generator 的推理、代码或 Skill 内容。这保证了 Verifier 生成的测试断言与 Generator 内部状态条件独立，防止 Verifier "知道答案后合谋放行"。

**代理奖励公式**：R̃(x, V) = (1/|V|) Σ 1[e_k(x)] ∈ [0, 1]，其中测试套件 V = {e_1, ..., e_|V|} 由 Verifier 自主生成的确定性断言组成。

当 R̃ < 1 时，Verifier 生成结构化诊断 F(i,j)（包含失败断言明细、根因分析、修复建议），反馈给 Generator。

### 协同演化循环

协同演化通过两条反馈通路实现交替优化：

**通路 1：Skill 细化**（当 Surrogate 测试失败时）
- 固定测试套件 V(j)，将诊断 F(i,j) 追加至 Generator 上下文，触发 Skill 修复
- Verifier 的测试套件在此阶段保持锁定，不参与更新

**通路 2：测试升级**（当 Surrogate 通过但 Oracle 失败时）
- Oracle 返回不透明 Pass/Fail 位（无测试内容），追加至 Generator 上下文
- Verifier 独立地在现有输出 x(i) 和旧测试 V(j) 基础上强化测试套件：V(j+1) ~ π_θ^V(· | I, x(i), V(j))
- 升级策略包括：生成更多样、更全面、更严格的测试用例

**防作弊机制**：Oracle 的二值信号仅告知"是否通过"，不暴露任何测试内容，防止 Generator 直接针对 Oracle 测试过拟合。Verifier 在独立会话中强化测试，也无法访问 Oracle 的测试细节。

**终止条件**：最多 N = 5 次 Oracle 评估轮次，M = 15 次 Surrogate Verifier 重试；Oracle 满分（reward = 1.0）时提前退出。保留历史最优 Skill 快照。

**实际演化效率**：每任务平均 4.1 次 Verifier 介入，2.4 次 Oracle 评估；约 40% 的迭代由 Surrogate Verifier 独自吸收（无需调用 Oracle）；超过 60% 的任务在 2 次 Oracle 轮次内收敛。

## 关键设计细节

**Skill bundle 格式**：遵循 Anthropic Agent Skills 规范，每个 Skill 是文件系统目录，包含：
- `SKILL.md`：触发元数据 + 9 步流程指令（含前置条件、终止条件）
- 可执行 Python 脚本（带完整函数实现，避免 Agent 每次执行时重新从头实现）
- 可选领域参考资料

以外行星凌日周期检测任务为例：人工 Skill 为 5 个文件、合计 1,096 行文档；EvoSkills 演化出 1 个统一 Skill，包含 64 行流程文档 + 142 行可执行 Python 函数，经 5 次独立测试全部通过（100%）。

**对比人工 Skill 的关键改进点**：
1. 两阶段周期搜索（宽扫描 + 窄精化）被实现为带参数的两个独立函数，而非人工文档中埋在 246 行内的两行提示；
2. 算法被明确规定（TLS，而非 BLS/TLS/Lomb-Scargle 三选一），附有从三次失败 BLS 尝试中得出的具体理由；
3. 可执行函数替代散文说明，消除每次执行时 Agent 重新实现引入的精度 Bug。

**Surrogate-GT 差距（案例分析揭示的局限）**：Surrogate Verifier 无法复制 Oracle 的精确精度要求（如 5 位小数匹配），也无法区分自身估算误差与 Agent 答案的误差；Oracle 作为权威仲裁者的角色不可替代。

## 实验与效果

### Benchmark（SkillsBench）

SkillsBench 包含 87 个任务，跨越约 20 个专业领域（含 Software Engineering、Natural Science、Finance、Cybersecurity、Energy、Robotics 等 11 个主要领域），每个任务配有确定性验证器，以 pass rate（奖励 = 1.0 的任务比例）为主指标，具有二值确定性，无主观偏差。

### 主结果

![Figure 4: SkillsBench 上各方法对比](https://arxiv.org/html/2604.01687v1/x3.png)

*Figure 4：SkillsBench 上各基线与 EvoSkills 的质量对比（Claude Opus 4.6 + Claude-Code）。误差棒为 5 次独立运行的 ±1 std。*

| 方法 | Pass Rate | 备注 |
|------|-----------|------|
| No-Skill Baseline | 30.6% | 无 Skill 直接执行 |
| Self-Generated Skills（SkillsBench 原版） | 32.0% ± 3.1% | 单次生成，无验证 |
| CoT-Guided Self-Generation | 30.7% ± 5.2% | 5 步 CoT 引导，仍无外部验证 |
| Skill-Creator（Anthropic 官方，自动化版） | 34.1% | 迭代草稿自测但仍是单会话 |
| Human Curated Skills | 53.5% | SkillsBench 配套人工 Skill |
| **EvoSkills（本文）** | **71.1%** | **+40.5 pp over no-skill，+17.6 pp over human** |

关键发现：一次性生成策略无论多精细的提示工程均无法突破 35% 上限；EvoSkills 的提升根本来源是协同演化验证循环，而非 Skill 生成提示本身。

### 领域拆解

![Figure 6: 各领域 pass rate 分布](https://arxiv.org/html/2604.01687v1/x5.png)

*Figure 6：SkillsBench 11 个专业领域上三种条件（无 Skill / 人工 Skill / EvoSkills）的 pass rate 对比。箭头标注 Natural Science 领域——人工 Skill 导致性能下降，EvoSkills 则取得显著提升，印证了人机认知错位。*

- **Finance**：EvoSkills 相较人工 Skill 提升 +56.9 pp（最大增益领域）
- **Cybersecurity**：+23.2 pp
- **Natural Science**：人工 Skill 性能下降（认知错位典型案例），EvoSkills 反而取得显著增益
- **Energy / Robotics**：人工 Skill 已表现良好，演化收益边际递减
- EvoSkills 在 11 个领域中 9 个超越人工 Skill

### 跨模型迁移性

![Figure 5: 跨模型迁移结果](https://arxiv.org/html/2604.01687v1/x4.png)

*Figure 5：Skill 跨模型迁移结果。Claude Opus 4.6 演化的 Skill 迁移至 6 个不同提供商的模型，所有模型提升 +36 至 +44 pp，证明演化 Skill 编码了可复用任务结构而非模型特定制件。*

- **主模型**：Claude Opus 4.6（+40.5 pp）、GPT-5.2（+40.2 pp）
- **迁移模型**（均接受 Opus-演化的 Skill）：Claude Sonnet 4.5 / Claude Haiku 4.5 / Qwen3-Coder-480B / DeepSeek V3-671B / Mistral Large 3-675B，全部提升 +36 至 +44 pp
- 模型自身演化（GPT-5.2 自演化：69.8%）略优于接受迁移 Skill（65.0%），差距 4.8 pp

### 消融

| 消融设置 | Pass Rate | 与 Full 的差距 |
|---------|-----------|-------------|
| EvoSkills（完整） | 71.1% | — |
| 去除 Surrogate Verifier（仅用 Oracle 不透明信号） | 41.1% | -30.0 pp |
| 去除 Skill 演化（仅背景上下文 + 直接执行） | 48.6% | -22.5 pp |
| No-Skill Baseline | 30.6% | -40.5 pp |

核心结论：Surrogate Verifier 贡献 -30.0 pp（最大单项贡献），表明没有结构化诊断反馈，Generator 无法进行定向修复，演化效率极低。背景知识上下文对性能有一定帮助（+18.0 pp over no-skill），但远不足以达到完整框架的水平（差距 -22.5 pp）。

## 结论与启发

### 论文结论

1. Agent 自动演化的 Skill 在 SkillsBench 上超越人工精心编写的 Skill（71.1% vs 53.5%），证实了"机器发现的 Skill 比人类设计的更适合 Agent"这一命题。
2. 演化 Skill 对不同模型家族具有强迁移性（+36 至 +44 pp），编码的是可复用的任务结构而非模型特定制件。
3. 协同演化循环（而非生成提示本身）是性能提升的根本驱动力；5 轮内即可收敛，演化成本可控。

### 对 Skill 自动合成 / 自进化方向的启发

1. **Generator-Verifier 架构 vs 单数版 Proposer-Builder 架构**：两种架构代表了不同的无监督 Skill 演化路线——单数版利用有标注失败样本驱动 Proposer 的提案式进化，复数版彻底摒弃 Ground-Truth 标注依赖，改用 Surrogate Verifier 自主合成反馈信号。复数版的适用范围更广，但对 Verifier 本身的质量依赖更高。

2. **Surrogate Verifier 作为 Skill 自动合成的通用信号源**：在 Skill 自动合成场景中，Ground-Truth 反馈通常稀缺或昂贵；以独立 LLM 会话合成测试断言作为代理奖励，是绕开这一瓶颈的可推广范式，值得在其他 Skill 生成任务中探索。

3. **+40.5 pp 对"自动合成 Skill 真的有效"的强力反驳**：SkillsBench 原论文发现自生成 Skill 基本无效（32.0%，与无 Skill 基线 30.6% 几乎持平），EvoSkills 将该结论反转至 71.1%，关键差异在于迭代验证，而非生成策略本身。这提示领域：Skill 质量不能靠一次生成保证，需要闭环验证机制。

4. **多文件 Skill bundle 的可靠生成需要迭代验证**：单次生成多文件结构包存在覆盖缺口和逻辑错误，不可避免。持久上下文 + 累积诊断反馈是当前解决这一问题的有效路径。

5. **认知错位是 Skill 领域的系统性问题**：Finance / Natural Science 等领域人工 Skill 表现的大幅差异（最差甚至性能下降），证明人类直觉与 LLM 执行逻辑之间存在结构性鸿沟；自动演化 Skill 通过实际执行反馈来弥合这一鸿沟，比人类主观判断更可靠。

6. **Skill 迁移性表明演化产物编码了任务抽象而非模型偏好**：跨 5 家提供商 6 个模型均获得 +36 至 +44 pp 提升，支持"Skill 即可迁移的任务知识"这一定位，为未来构建跨模型 Skill 共享市场（skill marketplace）提供了实验依据。
