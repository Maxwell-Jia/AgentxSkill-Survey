## 针对的问题

### 痛点

在实际 LLM 应用中，用户会反复表达相同的偏好与要求——例如"避免幻觉"、"遵循机构写作规范"、"减少技术术语"——但这些交互经验从未被系统性地沉淀为可复用知识。结果是：

- **会话间无积累**：每次新会话，用户偏好和任务规范都需要从头重申。
- **记忆系统的局限**：现有记忆增强方案（MemoryBank、MemGPT、Mem0 等）仅将过去对话当作可检索的文本片段，而非可操作化的行为能力。
- **参数更新代价高**：Self-evolution、RLHF 等参数更新方法难以支持频繁、细粒度的个性化需求，且缺乏透明度。
- **技能隐式存储**：Voyager、Reflexion 等 Agent 框架中的技能通常隐含于 prompt、轨迹或策略中，缺乏统一的检视、编辑、迁移与长期维护机制。

### 核心差距

现有方法均未能将"反复出现的交互经验"转化为**显式、可复用、可维护的技能单元**。

---

## 核心思路

> AutoSkill 将用户交互经验抽象为带版本号的显式技能制品（SKILL.md），通过"技能提取→管理→检索→注入"闭环实现无需参数更新的终身个性化学习。

---

## 方法

### 整体框架

![Figure 1: AutoSkill 框架由两个耦合循环组成](https://arxiv.org/html/2603.01145v2/x1.png)

**Figure 1**：AutoSkill 框架由两个紧密耦合的过程组成。右侧循环（*技能演化*）将交互经验转化为显式技能；左侧循环（*技能增强响应生成*）通过查询重写、技能检索、上下文注入来支持当前响应生成。系统通过显式记忆增长持续提升，而非模型微调。

---

### 形式化定义

对于用户 $u$，完整对话历史为 $\mathcal{X}_u = \{x_1, x_2, \ldots, x_T\}$，其中 $x_t = (q_t, r_t)$（查询+响应）。

**技能表示**：每个技能 $s$ 定义为七元组：

$$s = (n, d, p, \tau, \gamma, \xi, v)$$

| 字段 | 含义 |
|------|------|
| $n$ | 技能名称（name） |
| $d$ | 描述（description） |
| $p$ | 可执行指令 prompt（executable instruction prompt） |
| $\tau$ | 触发条件集合（trigger set） |
| $\gamma$ | 标签集合（tag set） |
| $\xi$ | 示例集合（example set） |
| $v$ | 版本号（version） |

技能以 **SKILL.md** 文件形式持久化，可人工检视、编辑、导入导出。

---

### Pipeline 各模块

系统由五类 Prompt 驱动模块组成，均无需专项训练：

$$\mathcal{M} = \{M_{\mathrm{rw}}, M_{\mathrm{chat}}, M_{\mathrm{ext}}, M_{\mathrm{judge}}, M_{\mathrm{merge}}, M_{\mathrm{emb}}\}$$

#### Loop 1：技能增强响应生成（在线推理路径）

**步骤 1：查询重写**

$$\tilde{q}_t = M_{\mathrm{rw}}(P_{\mathrm{rw}}, q_t, h_t)$$

将当前查询改写为检索导向的形式，解析上下文依赖，暴露格式、风格、领域等检索关键约束。

**步骤 2：混合技能检索**

对每个技能 $s \in \mathcal{B}_u^t$，计算混合相关性分数：

$$\mathrm{Rel}(q_t, s) = \lambda \hat{d}(q_t, s) + (1-\lambda) \hat{b}(q_t, s)$$

其中 $\hat{d}$ 为归一化向量语义相似度，$\hat{b}$ 为归一化 BM25 词汇匹配分数，$\lambda$ 控制二者权重。

只保留分数超过阈值 $\eta$ 的 Top-K 个技能：

$$\mathcal{H}_t = \{s \in \mathrm{TopK}(\mathcal{B}_u^t) \mid \mathrm{Rel}(q_t, s) \geq \eta\}$$

**步骤 3：技能条件响应生成**

检索到的技能渲染为紧凑上下文块 $C_t$，注入到对话模型：

$$r_t = M_{\mathrm{chat}}(P_{\mathrm{chat}}, q_t, h_t, C_t)$$

---

#### Loop 2：实时技能演化（后台学习路径）

**步骤 1：技能提取**

仅使用用户查询序列（不含模型回复）提取候选技能，以捕获稳定的用户需求而非模型生成内容：

$$z_t = M_{\mathrm{ext}}(P_{\mathrm{ext}}, \mathcal{Q}_u^t)$$

候选技能 $z_t = (n, d, p, \tau, \gamma, \xi, c)$，含置信度分数 $c$。提取目标是**持久偏好、可复用流程、输出约束、反复修正**等，而非一次性请求。

**步骤 2：检索辅助技能管理**

新候选不直接写入技能库，而是先检索最相似的现有技能作为局部证据：

$$\mathrm{Rel}_{\mathrm{m}}(z_t, s) = \alpha \hat{d}(z_t, s) + (1-\alpha) \hat{b}(z_t, s)$$

取最近邻 $s_t^* = \arg\max_{s \in \mathcal{N}_t} \mathrm{Rel}_{\mathrm{m}}(z_t, s)$，由判断模块决策：

$$a_t = M_{\mathrm{judge}}(P_{\mathrm{judge}}, z_t, s_t^*), \quad a_t \in \{\texttt{add}, \texttt{merge}, \texttt{discard}\}$$

**步骤 3：版本化技能合并**

若决策为 merge，合并模块执行版本化演化（而非简单文本拼接）：

$$s_t' = M_{\mathrm{merge}}(P_{\mathrm{merge}}, s_t^*, z_t), \quad v(s_t') = \mathrm{Bump}(v(s_t^*))$$

技能库更新规则：

$$\mathcal{B}_u^{t+1} = \begin{cases} \mathcal{B}_u^t \cup \{z_t\}, & a_t = \texttt{add} \\ (\mathcal{B}_u^t \setminus \{s_t^*\}) \cup \{s_t'\}, & a_t = \texttt{merge} \\ \mathcal{B}_u^t, & a_t = \texttt{discard} \end{cases}$$

## 关键设计细节

### 技能生命周期（四阶段）

| 阶段 | 内容 |
|------|------|
| **Stage 1：经验摄入** | 摄入对话消息与行为事件轨迹作为学习信号 |
| **Stage 2：技能提取** | 从摄入证据中提取可复用能力候选，过滤一次性请求 |
| **Stage 3：技能维护与版本控制** | add / merge / discard 三选一，合并时更新版本号 |
| **Stage 4：技能复用** | 向量索引检索相关技能，渲染为上下文后注入 LLM 请求 |

进化原则：**通过精炼而非复制** 演化 SkillBank——同一技能的后续反馈直接迭代更新该技能，而不产生重复 prompt 片段。

### SKILL.md 制品结构

每个技能以 SKILL.md 为核心，可附带脚本、参考资料等。文件格式包含：名称、描述、指令 prompt、触发条件、标签、示例、版本号。存储布局：
- `SkillBank/Users/<user_id>/...`：用户私有技能
- `SkillBank/Common/...`：共享技能
- `SkillBank/vectors/...`：向量缓存

### 系统部署模式

| 模式 | 说明 |
|------|------|
| Python SDK | `ingest` / `search` / `render_context` 接口，支持自定义工作流 |
| Web UI | 在线对话，前台检索注入，后台异步演化 |
| OpenAI 兼容反向代理 | 暴露 `/v1/chat/completions` 等标准端点，对现有 LLM 客户端透明增强 |
| 离线引导 | 批量导入历史对话、文档、Agent 轨迹初始化 SkillBank |

### 核心优势

- **无需训练**：全程不更新模型参数，改进来自显式技能构建、检索与精炼
- **模型无关**：作为插件层与任何 LLM 兼容
- **人工可控**：SKILL.md 制品可直接检视、编辑、导入导出
- **并发延迟分离**：技能提取在后台异步进行，不阻塞用户请求延迟

## 实验与效果

### 数据集与协议

- 数据来源：**WildChat-1M**（真实用户与 ChatGPT 交互日志，含 100 万条多语言对话）
- 过滤条件：保留超过 8 轮的对话（保证足够上下文以稳定提取技能）
- 构建四个 SkillBank 子集：Chinese GPT-3.5 / English GPT-3.5 / Chinese GPT-4 / English GPT-4

### 版本演化的质性证据（案例分析）

| 技能 | 语言 | 版本号 | 含义 |
|------|------|--------|------|
| `professional_text_rewrite` | 英文 | 0.1.34 | 经过 34 轮迭代精炼，捕获严格的专业改写规范（禁止额外说明、不可省略细节等） |
| `顶级心理咨询师` | 中文 | 0.1.0 | 接近初始提取形态，编码温暖/共情/非评判性的对话支持风格 |

这对比例证明：版本化维护机制能支持持续精炼而非简单技能复制；高频使用的生产力技能会被反复合并精炼，而更专业或低频触发的技能可能长期保持早期版本。

## 消融与分析

本文以**描述性/分析性**为主，未进行标准消融实验。关键设计选择的合理性通过以下方面论证：

1. **仅用用户查询提取**（不含模型回复）：确保提取到的是稳定用户需求，而非模型生成偏差
2. **最近邻局部判断**（而非全库比较）：减少 judge/merge 模块的输入规模，使决策更聚焦、更可扩展
3. **混合检索（Dense + BM25）**：语义匹配覆盖概念相关技能，词汇匹配捕获关键词精确触发
4. **阈值过滤**（$\eta$ + Top-K）：若无相关技能则不注入，避免噪声干扰

## 结论与启发

### 论文结论

AutoSkill 提供了一条实际可行的路径，将 LLM Agent 的终身学习问题从"隐式记忆积累"转化为"显式技能演化"：

- **单元迁移**：积累单元从原始记忆记录提升为显式行为知识
- **无参数改进**：通过外部技能记忆的构建、检索、精炼实现持续提升
- **透明可控**：SKILL.md 制品可被人工审查和修订，适合需要长期对齐用户偏好的场景

### 对 Skill 自动合成/沉淀方向的启发

1. **技能即制品（Skill-as-Artifact）**：将技能显式化为可版本管理的文件，比隐藏在 prompt 或策略中有更高的可维护性——这一范式值得在更多 Agent 框架中推广。

2. **仅从用户侧信号学习**：提取阶段只用用户查询而非模型回复，是避免学习到模型偏差的巧妙设计；对于需要对齐真实用户意图的系统尤为重要。

3. **版本化合并替代重复积累**：传统 RAG 系统倾向于堆积相似 chunk，AutoSkill 通过 merge+version 机制保持技能库的一致性，有效控制库规模与质量——可借鉴于其他知识管理系统。

4. **异步演化解耦延迟**：将技能提取放在后台异步执行，与在线响应生成解耦，是面向生产部署的关键工程决策。

5. **跨用户/任务迁移潜力**：SKILL.md 的标准化表示使得技能在用户间、Agent 间共享成为可能，这是构建"技能市场"或组织级技能知识库的基础。

6. **局限性**：本文实验以描述性分析为主，缺乏与基线方法的量化对比；技能质量评估依赖案例分析，系统化的评估 benchmark（如配套论文 StuLife）仍需进一步验证 AutoSkill 的端到端效果。
