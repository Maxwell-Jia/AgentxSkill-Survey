## 针对的问题

### 痛点

1. **缺乏系统化评测框架**：Agent Skills 生态迅速增长（GitHub/Claude Code 生态合计 47K+ Skills），但没有标准方法衡量 Skill 是否真正帮助。
2. **设计原则未知**：不清楚哪些 Skill 设计（指令结构、代码例、资源数量）驱动性能提升，何时失效。
3. **自动生成 Skill 的可行性未验证**：LLM 能否自动合成有效的 Skill？手写 curated Skill 的必要性程度如何？
4. **跨任务/模型的一致性缺失**：Skill 效果是否因任务域、模型规模而波动？

### 核心差距

现有 Agent benchmark（SWE-Bench、WebArena 等）仅评测"模型在无辅助条件下的能力"，而非"Skill 的边际贡献"。需要三条件对比评测框架：无 Skill vs. curated Skill vs. self-generated Skill。

---

## 核心思路

> 构建首个**以 Skill 为一等公民的 evaluation framework**，在 86 个跨 11 个领域的任务上，三条件对比测试 7 个 LLM-Agent 组合，量化 Skill 的实际效能与设计规律。

---

## 方法

### Skill 规范与定义

论文明确定义 Skill 为满足四项标准的结构化包：

| 标准 | 具体要求 |
|------|---------|
| 程序内容（Procedural） | 包含"如何做"的指导（流程、SOP），非事实知识检索 |
| 任务类通用性 | 应用于一类问题，非单个实例 |
| 结构化组件 | 必须包含 SKILL.md + 可选资源（脚本、模板、例子） |
| 可移植性 | 基于文件系统，易于编辑、版本控制、跨 harness 使用 |

**显式排除**：系统提示词 / Few-shot 例 / RAG 检索 / 工具文档。

Skill 结构：`environment/skills/<skill-name>/` 包含 `SKILL.md` + 可选的 `/scripts` 与 `/references`。

### Benchmark 构建（84 任务）

**数据来源**：
- 105 位贡献者提交 322 候选任务
- 最终通过 86 个任务（26.7% 录取率）
- 覆盖 11 个领域：软件开发、数据分析、DevOps、写作/文档等

**任务规范**：
每个任务包含四部分——指令、Docker 容器化环境、参考解、确定性验证器。使用 pytest + CTRF 格式，避免 LLM-as-judge 的不确定性。按人工完成时间分层（易/中/难）。

### 三条件评测协议

![Figure 1: 三条件下各配置的 pass rate](https://arxiv.org/html/2602.12670v1/x1.png)

**Figure 1**: Agent 架构堆栈与 84 任务上 7 个配置的通过率。Curated Skill（米色）平均提升 +16.2pp；Self-generated Skill（琥珀色）效果近零甚至负效。

评测条件：
1. **No Skill**：仅提供 instruction.md，环境无 Skill
2. **With Curated Skill**：完整的 environment/skills/ 目录（例、代码片、资源）
3. **Self-Generated Skill**：无预设 Skill，智能体被要求自行生成程序知识后求解

### 质量保证机制

**自动化检查**：
- 结构验证（文件完整性、TOML 语法）
- Oracle 执行（参考解必须 100% 通过测试）
- AI 检测（GPTZero 确保指令人工编写）

**Leakage Audit**：使用 Claude Agent 在 CI 中检测 Skill 是否包含任务特定解法（禁止：任务标识符、具体文件路径、测试常量、精确命令序列）。

**人工审查**：维护者评估 5 项标准——数据真实性、任务现实性、Oracle 质量、Skill 质量、防作弊措施。审查者实际运行 benchmark 多次验证 Skill 的必要性与有效性。

### 实验配置

![Figure 2: Benchmark 三阶段 pipeline](https://arxiv.org/html/2602.12670v1/x2.png)

**Figure 2**: 数据聚合（47.15K 个 Skill 来自开源 + Claude Code 生态 + 企业合作）→ 质量过滤（自动检查 + 人工审查）→ 评测执行（三条件 × 7 配置 × 5 trials）。

**智能体与模型**：
- 3 个商业 agent harness：Claude Code、Gemini CLI、Codex CLI
- 7 个前沿模型：GPT-5.2、Claude Opus 4.5/4.6、Claude Sonnet 4.5、Claude Haiku 4.5、Gemini 3 Pro/Flash
- 总计 7,308 条轨迹，5 次重复试验

**指标**：
- 主指标：Pass Rate（84 个任务的二值奖励平均）
- 增益衡量：Normalized Gain = (pass_skill - pass_vanilla) / (1 - pass_vanilla)

## 实验与效果

### 主要发现

![Figure 4: Pareto 前沿（Pass rate vs. 成本）](https://arxiv.org/html/2602.12670v1/x4.png)

**Figure 4**: Skill 将 Pareto 前沿上移；Gemini 3 Flash 与 Claude Opus 主导带 Skill 的前沿。Flash 虽输入 token 数 2.3× 更多（1.08M vs. 0.47M），但 4× 更低的单 token 成本使其总成本便宜 44%（$0.55 vs. $0.98/任务）。

### Finding 1: Curated Skill 收益显著但差异大

| 配置 | No Skill | With Skill | 增益 | 增益率(g) |
|------|----------|-----------|------|---------|
| Gemini CLI + Flash | 33.8% | 48.7% | +14.9pp | 22.5% |
| Claude Code + Opus 4.5 | 25.1% | 48.4% | +23.3pp | 31.1% |
| Claude Code + Opus 4.6 | 34.8% | 48.8% | +13.9pp | 21.4% |
| **平均** | **31.2%** | **47.4%** | **+16.2pp** | - |

收益范围 +13.6pp 至 +23.3pp，表明 Skill 效能高度依赖于特定 LLM-Agent 组合。

### Finding 2: Self-Generated Skill 几乎无效

Self-generated Skill 相比无 Skill 基线：**平均 –1.3pp**。
- Opus 4.6：+1.4pp（仅有微弱收益）
- GPT-5.2：–5.6pp（显著负效）
- 其他模型：持平或负

**轨迹分析**表明两种失败模式：
1. 模型识别到需要领域知识但生成的程序不精确（如"用 pandas 处理数据"而无具体 API 模式）
2. 对高领域知识任务（制造业、金融），模型常未能认识到需要专用 Skill，试图用通用方法

→ 这是对"Skill 自动合成"方向的**重要反例证据**。

### 按领域分解（Finding 3）

![Domain 增益差异（假设为 Table 4）](https://arxiv.org/html/2602.12670v1/x4.png)

| 领域 | 增益 (pp) | 特征 |
|------|---------|------|
| Healthcare | +51.9 | 高度专业化工作流，预训练覆盖少 |
| Manufacturing | +41.9 | 工业流程标准化 |
| Finance | 中等 | 已有预训练知识，Skill 补充空间有限 |
| Software Eng. | +4.5 | 最小增益，模型已掌握常见开发流程 |
| Mathematics | +6.0 | 最小增益 |

**规律**：预训练覆盖不足的领域（临床数据协调、制造工作流）收益最大；覆盖充分的领域（数学、软工）收益微弱。

### 任务级失败分析（Finding 4）

**最大受益任务**（+75pp 以上）：
- mario-coin-counting：+85.7pp（2.9% → 88.6%）
- sales-pivot-analysis：+85.7pp
- flood-risk-analysis：+77.1pp
- sec-financial-report：+74.3pp

共同特点：涉及预训练覆盖稀少的专业程序知识。

**负效任务**（16 of 84 任务）：
- taxonomy-tree-merge：–39.3pp
- energy-ac-optimal-power-flow：–14.3pp
- trend-anomaly-causal-inference：–12.9pp

假说：Skill 在模型已掌握的任务上引入冲突指导或不必要复杂度。

### Skill 设计因子分析

#### Finding 5: 2–3 个 Skill 最优

| Skill 数量 | 增益 (pp) |
|-----------|---------|
| 1 个 | +13.2 |
| 2–3 个 | **+18.6** |
| 4+ 个 | +5.9 |

结论：过多 Skill 产生认知开销或冲突指导，边际效益递减。

#### Finding 6: 精简 Skill 优于全面文档

| 复杂度类别 | 增益 (pp) | 特征 |
|----------|---------|------|
| Detailed（精简 + 例） | +18.8 | 最优 |
| Compact（简略） | +17.1 | 高效 |
| Comprehensive（详尽） | **–2.9** | 负效 |

解释：冗长文档会消耗上下文预算而难以提取关键信息；虽然提供了更多细节，但反而增加了提取相关内容的难度。模型需要**专注的程序指导**而非穷举文档。

### Finding 7: 小模型 + Skill 可匹敌大模型 + 无 Skill

| 配置 | Pass Rate | 对比 |
|------|-----------|------|
| Haiku 4.5 + Skill | 27.7% | |
| Haiku 4.5 无 Skill | 11.0% | +16.7pp |
| Opus 4.5 无 Skill | 22.0% | 接近 |

Skill 能部分弥补模型容量不足，在程序化任务上表现显著。

## 关键设计细节

### Harness 对 Skill 利用的差异

- **Claude Code**：Skill 利用率最高，提升 +13.9pp 至 +23.3pp，所有 Claude 模型一致受益
- **Gemini CLI**：原始性能最高（Flash 48.7%），改进 +13.6pp 至 +17.4pp
- **Codex CLI**：虽然竞争力强（44.7%），但**频繁忽视** Skill，智能体常承认 Skill 存在但独立实现解决方案

→ Skill 效能不仅取决于质量，也取决于 harness 的 Skill 集成设计。

### Skill 为何失败的两个深层机制

1. **上下文管理**：结构化界面可能导致长轨迹失败（如格式漂移），削弱早期注入的 Skill 影响力
2. **模型学习偏差**：Self-generated 实验证明，模型即使有充足上下文预算，也无法可靠地生成高效的程序知识

## 启发与意义

### 对 Skill 自动合成方向的启示

1. **Self-generated Skill 的失效是重要反证**：平均 –1.3pp 的结果表明，仅通过提示词让 LLM 生成 Skill 是不可行的。Skill 合成需要：
   - 人工演示或反馈引导
   - 从多个成功轨迹中归纳而来的 inductive bias
   - 结合执行反馈的迭代细化

2. **Skill 沉淀需要精心策划**：
   - 不是所有"运行成功的代码"都能成为好 Skill
   - Focused guidance（2–3 模块、精简但带例）优于全面文档
   - 防 leakage 与质量审核不可省

3. **领域与模型间的异质性强**：
   - 不能用"通用 Skill 合成算法"处理所有领域
   - 需要 per-domain 或 per-task-family 的定制化方法
   - Harness 的 Skill 集成能力是关键瓶颈

4. **Skill 评测的关键设计**：
   - 必须三条件对比（无、curated、self-gen）才能隔离 Skill 结构的价值
   - 单纯增加上下文（如无差别垫料）不足以验证 Skill 有效性
   - 跨多个 harness 的评估揭示了实现细节的影响

5. **Skill 管理与成本优化**：
   - 小模型 + 精心设计的 Skill 可以逼近大模型性能
   - 2–3 个高质量 Skill 优于数量多但重复的 Skill pack
   - 提示模型所有者优先投入"Skill 质量"而非数量

### 对综述的定位

这是首个系统化量化 Skill 跨任务效能的研究，为："skill 自动合成 / 自进化 / 沉淀 / 评测"方向提供：
- **基准与度量**：86 任务 benchmark、标准化评测协议
- **反事实证据**：Self-generated Skill 无效，打破"更多自动化即更好"的假设
- **实用指导**：2–3 精简 Skill 的设计规律

但也启示后续工作需要在"如何有系统地从演示、失败或专家反馈中自动提取 Skill"方向突破。

---

## 论文局限与未来方向

1. **评测域限制**：聚焦终端命令行任务（容器化、可复现），结果可能不直接转移至 GUI agent、多智能体协作、超长期任务
2. **因果属性归因**：Skill 注入增加上下文长度，收益可能部分来自"更多上下文"而非"程序结构"
   - Self-generated 实验部分回应了这一点（等长上下文但无效），但需要更强的长度匹配基线
3. **模型与 harness 覆盖**：评测模型、harness 行为会随时间变化
4. **多模态缘空白**：缺乏视觉-语言 agent 在 GUI 环境中的 Skill 评估
