## 针对的问题

### 痛点
现有 skill 评测基于**过度理想化的假设**：
- 在 Skill Augmented Agents 的论文中，通常直接把**手工精选（hand-crafted）的、针对特定任务的 skill** 灌给 LLM，假设 skill 总是存在且完美匹配
- 现实部署完全不同：Agent 需要从**大规模 skill 库**（数万个通用 skill）中**自行检索**，而检索到的 skill 往往质量参差、不够精炼
- 现有 benchmark（如 SkillsBench）因此高估了 skill 的实际收益

### 核心差距
**Handcrafted skill 下的收益 vs. 真实检索场景中的收益之间存在巨大鸿沟**
- 当 skill 必须被检索时，性能从 55.4% 下降到 38.4%（仅比无 skill 基线 35.4% 高 3 个百分点）
- 这反映了两个交织的失效：**检索漂移**（retrieval drift）和 **skill 不适配**（mismatch）

---

## 核心思路
> 第一个系统性的、**progressive 的**评测框架：量化 skill 在从理想到现实、从有到无的各个约束条件下的真实效能，并找出能弥合差距的精炼策略。

---

## 方法

### 评测协议与三档对照条件

论文设计了**4 个渐进式的实验条件**，从最简单到最现实：

1. **Curated Skill + Task-Specific**（上界）：直接给定手工选中的、针对该任务的 skill（仅用于估计上界，实际评测不用）

2. **No Skill Baseline**（下界）：纯 LLM 无 skill

3. **Retrieved Skill from 34k Collection**（现实条件 1）：Agent 从 34k 真实 skill（来自 skillhub.club、skills.sh 等生态）检索后使用

4. **No Skill Provided / Agent 自行检索**（最难条件 2）：Agent 既要找 skill 又要用 skill

关键指标：Pass Rate（通过率）、Recall@k（检索召回率）、Coverage Score（skill 质量打分）。

### Skill 池与数据

- **34k 现实 skill**：从开源 skill hub 收集，涵盖多个领域（code, knowledge, tool use 等）
- **SkillsBench 基准**：包含多个任务类别
- **Terminal-Bench 2.0**：扩展到终端交互场景

### 三类 Skill 不适配失效的诊断

论文通过消融发现，性能下降来自三个链式缺陷：

| 缺陷阶段 | 现象 | 具体数字 |
|---------|------|--------|
| **Skill 选择**（Selection） | Agent 即使有相关 skill 可用，也只在 49% 的情况下加载 | - |
| **Skill 检索**（Retrieval） | 从 34k 库中检索到相关 skill 的成功率只有 65.5%（Recall@5） | 37% → 38.4% |
| **Skill 适配**（Adaptation） | 通用 skill 往往误导 Agent 而非帮助（negative transfer） | 38.4% → 35.4%（无skill） |

### 精炼策略

论文探索了两类 skill refinement 方法：

#### Query-Specific Refinement（任务感知的精炼）
- Agent 基于当前任务描述，动态修改/筛选 skill 内容
- 效果最强：Claude Haiku 从 40.1% 恢复到 48.2%
- 前提：初始检索的 skill 质量足够好（Coverage Score ≥ 3.83）

#### Query-Agnostic Refinement（通用精炼）
- 对所有 skill 统一应用精炼规则（如压缩、简化）
- 效果有限：约 +2% 改进（42.0% vs. 40%）
- 作用：当初始 skill 质量不足时作为兜底

## 关键设计细节

### 为什么会有"skill degradation"

1. **Skill 量大 + 相关性差**：从 34k 库检索时，BM25 或向量检索都不能保证高精度
   - 对比：semantic search Recall@3 仅 38.1% vs. agentic hybrid search 57.3%
   
2. **Skill 不是万能的**：通用 skill 往往是面向不同任务开发，不能"开箱即用"
   - Coarse-grained skill 中存在大量 false positives
   
3. **弱模型对干扰更敏感**：GPT-3.5 / Claude Haiku 在面对不适配 skill 时性能下降明显，而 Claude Opus 4.6 能更好地"忽视"无关内容

### Agentic Hybrid Search 的优势

论文展示了 Agent **迭代式查询和评估**（iterative query formulation + candidate evaluation）相比直接语义检索的效果：
- Agentic hybrid search（Agent 自己尝试多个查询、评估候选）：57.3% Recall@3
- 直接 semantic search：38.1% Recall@3
- 这说明 Agent 的**主动性**在 skill 检索中是关键的

## 实验与效果

### 主要 Benchmark

- **SkillsBench**：包含多个 task 和 domain
- **Terminal-Bench 2.0**：真实终端命令执行场景，模拟更复杂的多步骤任务

### 核心结果对比

| 条件 | Claude Opus 4.6 | Claude Haiku | GPT-3.5 |
|------|--------|-----------|---------|
| No Skill Baseline | 35.4% | 28.1% | 22.0% |
| Retrieved Skill（无精炼） | 38.4% | 40.1% | 24.5% |
| Retrieved + Query-Specific Refinement | 48.2% | 48.2% | 35.2% |
| Terminal-Bench 2.0（无精炼） | 57.7% | - | - |
| Terminal-Bench 2.0（+精炼） | **65.5%** | - | - |

**关键观察**：
- 仅检索不精炼时，性能改进微小甚至负面（尤其弱模型）
- Query-specific refinement 是关键"乘数"：能恢复 5-10 个百分点
- 在 Terminal-Bench 2.0 上精炼能带来 +7.8% 的稳定收益

### 分任务 / 分领域分解

论文进一步按 skill 类别（code, knowledge, tool, etc.）和任务难度分解了性能，发现：
- **Code-related skills** 的检索精度最高，因为代码相似性易被向量模型捕捉
- **Knowledge skills** 检索困难，往往需要精细的语义理解
- **弱模型更容易被误导**：性能随 skill 数量增加反而下降

### Skill 质量的显著影响

论文通过 Coverage Score（衡量 skill 与任务的相关度）的消融显示：
- 当 Coverage Score ≥ 3.83 时，精炼策略能有效恢复
- 当 Coverage Score < 3.0 时，精炼也无法挽救

## 结论与启发

### 论文结论
1. **Skill 收益是脆弱的（fragile）**：现实约束（检索、不精炼）下，skill 的优势大幅缩水，有时甚至为负
2. **检索+精炼是必要的**：单纯给 LLM 一堆 skill 不如精心设计精炼流程
3. **现有 benchmark 高估了 skill 的实际效能**：当前发表的许多 skill-augmented 方法实际部署时会显著退化

### 对综述的启发

#### 1. **Skill 自进化 / 沉淀的必要性**
- 不是简单地**累积 skill**，而是要建立**质量控制**和**精炼机制**
- Skill 需要根据真实检索场景不断优化（query-specific refinement 的上游）

#### 2. **检索与评测的联动**
- 现有 skill 检索方案（如 BM25 + embedding）在 34k 规模上已经失效（65.5% Recall@5）
- 需要**更强的检索模型**或**更聪明的 hybrid 策略**（agentic search 显示有潜力）

#### 3. **精炼策略的分层**
- Query-specific refinement（任务定制）作为"第一防线"，但前提是初始 skill 质量可接受
- Query-agnostic 精炼作为"兜底"，对低质 skill 帮助有限
- 启发：精炼本身不生成知识，只是"乘以质量系数"

#### 4. **模型能力的二阶效应**
- 强模型（Claude Opus）对无关 skill 的抗性强，弱模型易被干扰
- 在多模型部署场景下，Skill 管理不能一刀切，需要按模型能力调整

#### 5. **评测框架的现实化**
- 未来 Skill 论文应该至少包含三档对照（手工 / 检索无精炼 / 检索+精炼），而非只秀手工版本
- 建议新增"Skill Degradation Ratio"这样的指标，衡量理想-现实之间的gap

#### 6. **Skill 管理生态的设想**
- 如果要让 skill 真正有用，需要：（a）高质量的 skill 池（不是随便累积 34k）；（b）智能检索（agentic search）；（c）自动精炼（query-aware adaptation）
- 这三层若缺一，skill 收益就会大幅折扣
