## 针对的问题

### Motivation：为什么 metadata-only 不够？

随着 LLM Agent 生态成熟，社区 skill 库（工具/插件）已达数万乃至十万量级。面对如此规模，把所有 skill 塞进 context 不可行——需要**skill routing**：给定用户任务，从大规模 skill 池中自动检索最相关的 skill。

**关键难点：**

1. **功能同质性（Functional Overlap）**：社区贡献的 skill 库中大量 skill 名称相近、目的相近，但实现细节各异，形成高度同质的检索池，极难区分。
2. **隐式假设被打破（Hidden-body asymmetry）**：现有 agent 架构采用"渐进披露"设计——agent 消费端只能看到 skill 的 name + description（受 context 窗口限制），routing 层却可访问完整的 skill body（实现代码/详细规范）。业界默认 metadata（name + description）足以用于 routing，**但这一假设从未被系统验证过**。
3. **现有 benchmark 不研究上游检索**：SkillBench、ToolBench、MetaTool 等评测的是"agent 如何使用已经给定的工具"，没有研究"如何从大池中找到正确工具"。

**三个核心研究问题：**
- RQ1：~80K 规模 skill 池中，路由是否是真正困难的检索问题？
- RQ2：skill 的哪个字段（name、description、body）才是精准选择的关键信号？
- RQ3：紧凑的 retrieve-and-rerank pipeline 能否有效利用这些信号？

---

## 核心思路

> **一句话概括：** skill body（完整实现文本）才是决定性信号（而非 metadata），因此 routing 系统必须在检索和重排两阶段都访问 body，SkillRouter 用 0.6B encoder + 0.6B reranker 构成 1.2B 的轻量两阶段 retrieve-and-rerank pipeline，在 ~80K skill 池上达到 74.0% Hit@1。

## 方法

### Pipeline 架构总览

![SkillRouter pipeline architecture. Stage 1: bi-encoder retrieval reduces ~80K skills to top-20 candidates. Stage 2: cross-encoder reranking produces the final ranking. Both stages use full skill text (name + description + body).](https://arxiv.org/html/2603.22455v1/x2.png)

**Figure 2：SkillRouter 两阶段架构。** 第一阶段 bi-encoder 将 ~80K skills 压缩到 top-20 候选；第二阶段 cross-encoder 对候选重排，输出最终排序。两阶段均使用完整 skill 文本（name + description + body）。

---

### 阶段一：Bi-Encoder 检索（SR-Emb-0.6B）

- **基座**：Qwen3-Emb-0.6B（decoder-based embedding model）
- **输入格式**：
  - Query 侧：`Instruct: Given a task description, retrieve the most relevant skill document... \n Query: {query_text}`
  - Skill 侧：`{name} | {description} | {body}`（description 截断至 300 token，body 截断至 2500 token）
- **训练数据**：37,979 个（query, skill）对，query 由 GPT-4o-mini 基于 skill 内容生成（不得包含 skill 名称），确保反映功能需求而非 skill 身份。训练数据与评测 benchmark **完全不相交**。
- **Hard negative mining**：多源策略，每个 query 挖掘 10 个负例：
  - 语义负例（4个）：基于 Qwen3-Emb-0.6B 预计算 embedding，从 top-50 相似 skill 中采样非正例
  - 词汇负例（3个）：BM25 scoring over full skill text
  - 分类负例（2个）：同 category 不同 name 的 skill
  - 随机负例（1个）：不同 category 的随机 skill
- **假负例过滤（三层）**：
  1. Name deduplication（名称相同的移除）
  2. Body trigram Jaccard similarity > 0.6 的移除
  3. Embedding cosine similarity > 0.92 的移除
  - 约过滤 10% 的负例对
- **损失函数**：in-batch InfoNCE（温度 τ=0.05）
- **训练配置**：单 GPU，batch size 8，梯度累积 4（等效 batch 32），学习率 2e-5 cosine schedule，BF16，训练 1 epoch

---

### 阶段二：Cross-Encoder 重排（SR-Rank-0.6B）

- **基座**：Qwen3-Reranker-0.6B（decoder-based cross-encoder）
- **输入格式**（flat-full）：
  ```
  <Instruct>: Given a task description, judge whether the skill document is relevant...
  <Query>: {query_text}
  <Document>: {name} | {description} | {body}
  ```
  （description 截断至 500 token，body 截断至 3000 token）
- **训练数据**：32,283 个 query，每个 query 取 SR-Emb-0.6B 检索的 top-20 候选，构成 20 个 skill 的 binary 标注列表（同样施加三层假负例过滤）
- **损失函数**：Listwise cross-entropy（对候选列表做 softmax + log-prob of positive）
  - 关键发现：listwise CE 比 pointwise BCE **高出 30.7pp Hit@1**（详见消融）
- **训练配置**：batch size 1（每次 forward 一个 20-skill 候选列表），梯度累积 16 步

## 关键设计细节

### Skill Body 才是决定性信号

![Skill body is the decisive signal. Left: Removing body (nd) causes 29–44pp Hit@1 drops across all methods. Right: Cross-encoder attention distribution shows 91.7% of attention on body, 7.3% on name, 1.0% on description.](https://arxiv.org/html/2603.22455v1/x1.png)

**Figure 1：左图：** 移除 body（只用 name+description，即 nd）后所有方法的 Hit@1 均出现灾难性下降（29–44 pp）。**右图：** cross-encoder 注意力分析显示 91.7% 的注意力集中在 body 字段。

**关键发现：**

| 方法 | 仅 nd（无 body） | 全文（含 body） | 差值 Δ |
|------|------------------|-----------------|--------|
| BM25 | 0.000 | 0.347 | +34.7 pp |
| Qwen3-Emb-0.6B | 0.227 | 0.587 | +36.0 pp |
| Qwen3-Emb-8B | 0.307 | 0.653 | +34.6 pp |
| Reranking（Avg Easy+Hard）Qwen3-Emb-8B × Qwen3-Rank-8B | 0.240 | 0.680 | +44.0 pp |
| Reranking Qwen3-Emb-8B × GPT-5.4-mini | 0.307 | 0.594 | +28.7 pp |

**"仅 nd"的 reranker 甚至是有害的：** nd reranker 将正确排序破坏至低于 encoder-only baseline（例如 Qwen3-Emb-0.6B × Qwen3-Rank-8B-nd：30.7% vs 单用 encoder 的 56.0%，降低 25.3pp）。

**注意力的层间演化（body→name→body 模式）：**
- Early layers (0–6)：几乎全部关注 body（97.3%）——token 级内容理解
- Middle layers (7–20)：name 的关注度上升至 peak 26.3%（语义匹配）
- Late layers (21–27)：回归 body 主导（91.7%）——最终相关性判断
- description 在所有层的注意力均 ≤ 1.4%，信息完全被 body 所包含

### Hidden-Body Asymmetry（信息不对称设计约束）

这是 skill routing 区别于普通文档检索的核心特征：
- **消费者（agent）**：只能看到 skill 的 name + description
- **路由器（retriever/reranker）**：可以访问完整的 body

这种不对称**不是设计便利，而是必要约束**——routing 层**必须**有 body 访问权限，才能实现准确选择。

## 实验与效果

### 评测 Benchmark

- **规模**：~80K skills，75 条专家验证的 query（24 单技能 + 51 多技能）
- **两个难度层级**：
  - Easy（78,361 skills）：标准条件
  - Hard（79,141 skills）：在 Easy 基础上添加 780 个 LLM 生成的 distractor skills（相关但功能不同）
- **主要指标**：Hit@1（首要指标）、MRR@10、nDCG@10、Recall@10/20/50、FC@10

### Encoder 检索结果

**Table 4：各 encoder 在 ~80K skill 池上的检索结果（全文输入）**

| 模型 | 类型 | 参数量 | Easy Hit@1 | Hard Hit@1 | Avg Hit@1 | Avg R@20 |
|------|------|--------|------------|------------|-----------|----------|
| BM25 | Sparse | — | .347 | .280 | .314 | .365 |
| gemini-emb-001 | Proprietary | — | .613 | .560 | .587 | .687 |
| text-emb-3-large | Proprietary | — | .640 | .600 | .620 | .664 |
| E5-Large-v2 | Encoder | 335M | .507 | .493 | .500 | .594 |
| BGE-Large-v1.5 | Encoder | 335M | .613 | .587 | .600 | .668 |
| GTE-Large-v1.5 | Encoder | 434M | .573 | .520 | .547 | .696 |
| Qwen3-Emb-0.6B | Decoder | 0.6B | .587 | .533 | .560 | .637 |
| NV-Embed-v2 | Decoder | 7B | .440 | .413 | .427 | .562 |
| Qwen3-Emb-8B | Decoder | 8B | .653 | .627 | .640 | .726 |
| **SR-Emb-0.6B（ours）** | Decoder | 0.6B | **.667** | **.640** | **.654** | **.754** |
| SR-Emb-8B（ours，scale） | Decoder | 8B | .693 | .667 | .680 | .777 |

**核心发现：** SR-Emb-0.6B（0.6B 微调）以更低参数量超过了 Qwen3-Emb-8B（8B 零样本），证明任务特定微调比单纯扩大规模更有效。

### 端到端 Pipeline 结果

**Table 5（摘要版）：Encoder × Reranker 端到端结果（top-20 候选，平均 Easy+Hard）**

| Encoder | Reranker | Rank Input | Avg Hit@1 | Avg MRR@10 |
|---------|----------|------------|-----------|------------|
| Qwen3-Emb-0.6B | — | — | .560 | .638 |
| Qwen3-Emb-8B | — | — | .640 | .698 |
| SR-Emb-0.6B | — | — | .654 | .723 |
| Qwen3-Emb-8B | Qwen3-Rank-8B | nd（无 body） | .240 | .392 |
| Qwen3-Emb-8B | GPT-5.4-mini | nd | .307 | — |
| Qwen3-Emb-8B | GPT-4o-mini | full | .647 | — |
| Qwen3-Emb-8B | Qwen3-Rank-8B | full | .680 | .745 |

| SR-Emb-0.6B | Qwen3-Rank-8B | full | .714 | .776 |
| **SR-Emb-0.6B × SR-Rank-0.6B（ours †）** | — | **full** | **.740** | **.791** |
| SR-Emb-0.6B × SR-Rank-8B | — | full | .747 | .804 |
| SR-Emb-8B × SR-Rank-8B（scale） | — | full | .760 | .808 |

**主要结论：**
- SkillRouter（SR-Emb-0.6B × SR-Rank-0.6B，1.2B 参数）达到 **74.0% Hit@1**，比最强零样本 8B pipeline（Qwen3-Emb-8B × Qwen3-Rank-8B，68.0%）高 **+6.0 pp**
- 比 encoder-only 基线（SR-Emb-0.6B）提升 **+8.6 pp**
- 可部署于消费级硬件（laptop CPU），无需云 API

### 规模扩展

| Pipeline | 总参数量 | Avg Hit@1 |
|----------|----------|-----------|
| SR-Emb-0.6B × SR-Rank-0.6B（primary） | 1.2B | 74.0% |
| SR-Emb-0.6B × SR-Rank-8B | ~8.6B | 74.7% |
| SR-Emb-8B × SR-Rank-8B | ~16B | 76.0% |

Encoder 质量对最终性能的贡献大于 reranker 规模：升级 encoder 0.6B→8B 贡献 +1.3pp，单独升级 reranker 仅贡献 +0.7pp。

## 消融与分析

### Metadata-Only vs Implementation-Aware 对比

无 body 的 reranker 不仅无效，甚至有害（主动破坏排序）。最佳 nd 组合（GPT-5.4-mini-nd）仅有 33.3% Hit@1，比对应 encoder-only 低 22.7pp。

### 消融一：假负例过滤的影响

**Table 7：假负例过滤消融（encoder 训练）**

| Tier | 过滤策略 | Hit@1 | MRR | nDCG | R@10 | R@50 |
|------|----------|-------|-----|------|------|------|
| Easy | 清洁负例 | .667 | .735 | .630 | .696 | .814 |
| Easy | 原始负例 | .640 | .710 | .607 | .676 | .810 |
| Easy | Δ | **-2.7** | -2.6 | -2.3 | -1.9 | -0.4 |
| Hard | 清洁负例 | .640 | .710 | .607 | .680 | .814 |
| Hard | 原始负例 | .587 | .675 | .573 | .667 | .803 |
| Hard | Δ | **-5.3** | -3.4 | -3.4 | -1.3 | -1.1 |
| Avg | 清洁负例 | .653 | .722 | .619 | .688 | .814 |
| Avg | 原始负例 | .613 | .692 | .590 | .672 | .807 |
| Avg | Δ | **-4.0** | -3.0 | -2.8 | -1.6 | -0.7 |

- 移除过滤导致平均 Hit@1 降低 **4.0pp**，Hard tier 降低更明显（-5.3pp）
- R@50 几乎不受影响（-0.7pp），说明假负例主要破坏精细排序能力，而非整体召回

### 消融二：Listwise vs Pointwise 损失

**Table 8：reranker 损失函数消融（Encoder = SR-Emb-0.6B）**

| Reranker | Loss | Easy Hit@1 | Hard Hit@1 | Avg Hit@1 | FC@10 |
|----------|------|------------|------------|-----------|-------|
| SR-Rank-0.6B | **LW（listwise）** | **.760** | **.720** | **.740** | .520 |
| Qwen3-Rank-0.6B | base（零样本） | .720 | .693 | .707 | .527 |
| Qwen3-Rank-8B | base（零样本） | .720 | .707 | .714 | .527 |
| FT-PW | PW（pointwise） | .453 | .413 | .433 | .320 |
| Encoder-only | — | .667 | .640 | .654 | .480 |

- **Listwise CE vs Pointwise BCE：差距高达 +30.7pp Hit@1**
- Pointwise 微调后甚至低于 encoder-only baseline（43.3% vs 65.4%）
- 原因：skill 池高度同质，pointwise sigmoid 对近似候选输出汇聚至窄带，排序接近随机；listwise softmax 直接建模候选间相对顺序，正是同质池中精选正确 skill 所需的比较机制

### Encoder vs Reranker 贡献分解

**Table 9：逐 query Hit@1 分解（encoder-only vs 完整 pipeline，共 150 条 query）**

| 类别 | 数量 | 比例 |
|------|------|------|
| 两者均正确 | 92 | 61.3% |
| Reranker 修复（encoder 错，pipeline 对） | 19 | 12.7% |
| Reranker 降级（encoder 对，pipeline 错） | 6 | 4.0% |
| 两者均错 | 33 | 22.0% |
| Encoder Hit@1 | 98 | 65.3% |
| Pipeline Hit@1 | 111 | 74.0% |

![Reranker contribution decomposition. The reranker rescues 12.7% of queries where the encoder fails (green), while degrading only 4.0% (red). Net contribution: +8.7pp Hit@1.](https://arxiv.org/html/2603.22455v1/x3.png)

**Figure 3：** Reranker 在 12.7% 的 query 上修复了 encoder 的错误，仅在 4.0% 上出现降级，净贡献 +8.7pp。

22% 的 query 被两阶段都错过，这是 encoder top-20 窗口召回不足或语义鸿沟需要多跳推理导致的系统上限。

## 结论与启发

### 论文总结

1. **Skill routing 是真正困难的检索问题**：最强 8B base encoder 也只有 64.0% Hit@1
2. **Skill body 是决定性信号**：移除 body 造成 29–44pp 灾难性下降，attention 分析证实 91.7% 集中在 body——推翻了"name + description 已足够"的业界假设
3. **SkillRouter（1.2B）有效且紧凑**：74.0% Hit@1，优于更大的零样本基线，可在消费级硬件上部署；同方法扩展到 8B 达到 76.0%

### 对 Skill 管理与检索方向的启发

1. **skill 文档质量至关重要**：body（实现细节/详细规范）的质量直接决定检索精度，社区应重视 skill body 的信息丰富性，而非只关注简短的 name/description
2. **routing 层架构设计**：大规模 agent 系统应将 routing 组件与 agent 消费端分离，routing 层必须有 body 访问权，agent 消费端保持 context 节省
3. **假负例过滤是同质池训练的必要步骤**：社区 skill 库功能重叠严重，不过滤假负例会严重损害 contrastive learning 效果
4. **Listwise ranking loss 对同质候选池是关键**：当候选池高度相似时，必须建模候选间相对顺序，pointwise 方案会失效
5. **微调优于单纯扩参数**：0.6B 任务特定微调模型超越 8B 零样本基线，data/training 对此任务比 scale 更重要
6. **两阶段 pipeline 的互补价值**：encoder 提供语义召回广度（R@20=75.4%），reranker 基于 body 细粒度 cross-attention 提升精度（Hit@1 净+8.7pp），两者缺一不可
7. **未来方向**：多跳推理与检索集成（当前 22% query 两阶段均失败）、更大规模 benchmark、端到端 agent 任务完成评估
