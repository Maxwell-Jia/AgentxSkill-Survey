## 针对的问题

现有 LLM 智能体在面对**新环境**时存在严重适应障碍：每次执行任务都从零开始，无法利用过去交互中积累的经验。Fine-tuning 虽然能解决适应问题，但成本高昂、对闭源模型不可行、且不支持持续学习。

已有工作（ExpeL、AutoGuide）的主要缺陷：
- **ExpeL**：需要多次重试构建成功/失败对比轨迹；生成的 insights 无差别地注入所有测试提示，随经验积累扩展性差。
- **AutoGuide**：同样需要多次执行获取对比对；每个 agent 步骤都要检索上下文，开销大；若无匹配上下文则无法提供指引。

**核心 Motivation**：在实际部署中任务通常无法重试，需要一种能从**单次执行轨迹**中高效学习的机制。

---

## 核心思路

> ERL（Experiential Reflective Learning）通过对单次任务轨迹进行反思来生成可复用的启发式规则（heuristics），并在执行新任务时利用 LLM 按相关性检索注入上下文，实现无参数更新的快速环境适应。

## 方法

### 框架图

![ERL 框架：左侧为经验积累阶段（agent 反思任务结果生成启发式规则存入池中），右侧为测试阶段（检索相关启发式规则注入上下文）](https://arxiv.org/html/2603.24639v1/x1.png)

*Figure 1: The ERL framework. During experience accumulation (left), the agent reflects on task outcomes to generate heuristics stored in a persistent pool. At test time (right), relevant heuristics are retrieved and injected into context for new tasks.*

ERL 由两个核心组件构成：

### 1. 启发式规则生成（Heuristic Generation）

**触发条件**：智能体执行任务后获得二值成功/失败反馈。

**输入**：任务描述 + 执行轨迹（推理步骤、工具调用、输出）+ 结果信号（成功/失败）。

**输出**：结构化启发式规则，包含两部分：
1. **分析**：识别成功或失败的原因
2. **指南**：带有明确触发条件和推荐行动的可泛化规则

示例规则：
> "当向日历参会者发送邮件时，应先通过 Contacts 工具将姓名解析为邮箱地址，再调用 email API。"

所有规则存入持久化的**启发式规则池（heuristic pool）**。

### 2. 检索增强执行（Retrieval-Augmented Execution）

面对新任务时：
1. LLM 分析新任务，将其分解为潜在子任务和行动步骤
2. 对规则池中每条规则按相关性评分，综合考虑：
   - **相似性**：存储任务与当前任务描述的相似度
   - **多样性**：覆盖不同潜在经验教训
   - **信息量**：指南内容本身的丰富程度
3. 选取 top-k 条规则注入 agent 的 system prompt

### Experience Transfer 机制关键设计

- **单次轨迹即可生成规则**：无需多次重试，适用于无法重复的真实部署场景
- **任务级检索而非全量注入**：避免随规则池增大导致的噪声和上下文爆炸
- **LLM 担任检索排序器**：理解错误模式和规则内容，而非仅靠语义相似度匹配

## 关键设计细节

### 模型配置（Gaia2 实验）

| 角色 | 模型 |
|------|------|
| Agent（主执行） | gpt-5-mini-2025-08-07 |
| ARE LLM Judge | gpt-5-mini-2025-08-07 |
| 启发式规则生成 | gpt-5-mini-2025-08-07 |
| 启发式规则检索 | gpt-5.2-2025-12-11 |

### 检索参数

- 默认 k=20 条启发式规则
- LLM 检索优于 embedding 检索（Qwen3-Embedding-0.6B）
- 过多规则（>40~60）反而有害，呈现非单调性能曲线

### 无奖励信号降级

- 当环境无法提供 success/failure 信号时，agent 自评准确率约 70%
- ERL 性能从 56.1% 降至 51.2%，仍高于 baseline 48.3%
- 说明精确反馈对最佳性能重要，但 ERL 对噪声信号有一定鲁棒性

### 成本分析（相比 baseline）

| 阶段 | Input Tokens | Output Tokens | 成本 |
|------|-------------|---------------|------|
| Baseline 场景执行 | 103.8M（82% 缓存） | 4.3M | $15.27 |
| ERL 启发式生成 | 7.5M（8% 缓存） | 0.3M | $2.42 |
| ERL 启发式检索 | 1.0M（86% 缓存） | 0.1M | $1.38 |
| ERL 场景执行 | 192.0M（88% 缓存） | 3.9M | $17.60 |
| **ERL 总计** | **200.5M（85% 缓存）** | **4.3M** | **$21.40** |

ERL 整体 API 成本增加约 **40%**，场景执行本身仅增加 15%（得益于更高缓存命中率）。

## 实验与效果

### 实验设置

- **Benchmark**：Gaia2（Agents Research Environments 平台），模拟移动设备环境，含 12 个应用、101 个工具
- **评估分割**：Search 分割（信息检索）和 Execution 分割（多步执行）
- **训练/测试划分**：8 个宇宙积累经验（112 执行 + 132 搜索任务），2 个保留测试宇宙（48 + 28 任务）
- **Baseline**：默认 ReAct 智能体框架

### 主结果

![成功率对比图（Gaia2，3次运行平均）](https://arxiv.org/html/2603.24639v1/x2.png)

*Figure 2: Success rates on the Gaia2 test universes (3 runs). Bold: best; underline: second best.*

| 方法 | Execution (%) | Search (%) | Overall (%) |
|------|--------------|-----------|-------------|
| Baseline（ReAct） | 43.1 | 53.6 | 48.3 |
| Few-shot（raw 轨迹） | 41.7 | 51.2 | 46.4 |
| ExpeL | 45.8 | 56.0 | 50.9 |
| AutoGuide | 39.6 | 61.9 | 50.8 |
| **ERL（本文）** | **51.4** | **60.7** | **56.1** |

**关键数字**：ERL 总体成功率 **56.1%**，较 ReAct baseline **提升 +7.8%**，较最强先前方法 ExpeL **提升 +5.2%**。

### 可靠性提升（pass@3 / pass^3）

![成功率、pass@3 与 pass^3 对比](https://arxiv.org/html/2603.24639v1/x3.png)

*Figure 3: Success rate, pass@3 and pass^3 comparison between baseline and ERL*

ERL 在 **pass^3**（要求 3 次运行全部成功）上提升尤为显著：
- Execution：+8.3%
- Search：+10.6%

说明 ERL 主要提升了**任务完成的稳定性**，而非解锁全新任务类型。

### τ²-Bench 辅助评估

| 指标 | 方法 | Airline | Retail | Telecom | Overall |
|------|------|---------|--------|---------|---------|
| Success Rate | Baseline | 36.7% | 43.3% | 30.0% | 36.7% |
| Success Rate | ERL | **38.3%** | **47.5%** | 28.3% | **38.0%** |
| pass^3 | Baseline | 20.0% | 12.5% | 10.0% | 14.2% |
| pass^3 | ERL | **25.0%** | **17.5%** | 0.0% | 14.2% |

Telecom 分割表现下降，原因是该领域任务组合空间大且涉及双控制交互（用户也参与决策），历史启发式规则难以匹配。

## 消融与分析

### 1. 启发式规则 vs. 原始轨迹

- Few-shot 注入原始轨迹：**-1.9%**（低于 baseline）
- 启发式规则：**+7.8%**
- 结论：蒸馏后的规则比原始轨迹提供更好的迁移学习信号；且在相同 token 预算下，规则能容纳更多经验（见 Figure 5）

![不同 token 预算下启发式规则 vs 轨迹的成功率对比](https://arxiv.org/html/2603.24639v1/x5.png)

*Figure 5: Success Rate over feedback token count for heuristic feedback and few-shot trajectories.*

### 2. 检索质量 vs. 数量

![随机选择不同数量规则的成功率曲线](https://arxiv.org/html/2603.24639v1/x4.png)

*Figure 4: Success rate vs. number of randomly selected heuristics. ERL with LLM-based retrieval of k=20 heuristics (★) outperforms all random configurations.*

- 随机选择规则：性能在 40~60 条时达峰值后下降（非单调）
- LLM 检索（k=20）：56.1%，优于 embedding 检索（53.3%）和最优随机配置（53.8%）
- **结论：检索质量比数量更重要**

Embedding vs LLM 检索对比（k=20）：

| 检索方式 | Execution (%) | Search (%) | Overall (%) |
|---------|--------------|-----------|-------------|
| 无检索（全量注入） | 44.4 | 63.1 | 53.8 |
| Embedding 检索（Qwen3） | 50.7 | 56.0 | 53.3 |
| **LLM 检索（GPT-5.2）** | **51.4** | **60.7** | **56.1** |

### 3. 失败 vs. 成功经验的贡献

| 消融配置 | Execution (%) | Search (%) | Overall (%) |
|---------|--------------|-----------|-------------|
| ERL（全部） | 51.4 | 60.7 | 56.1 |
| 仅失败经验 | 50.0 | **67.9** | **58.9** |
| 仅成功经验 | **52.1** | 47.6 | 49.9 |

- **失败规则**在 Search 任务上效果更好（通过负向约束剪枝无效策略，+14.3% vs baseline）
- **成功规则**在 Execution 任务上效果更好（强化已验证的行动序列，+9.0%）
- 实际部署中任务分布未知，混合使用是最安全的选择

### 4. 迭代 ERL（Iterative ERL）

在积累阶段同步检索已有规则（边积累边使用）：

| 方法 | Source 任务 Exec/Search | Test 任务 Overall |
|------|------------------------|-----------------|
| ERL（标准） | 42.0% / 53.8% | 56.1% |
| Iterative ERL | 44.6% / 59.9% | 50.7% |
| Iterative ERL（仅失败） | 41.1% / 56.8% | 54.3% |

迭代版本在积累阶段表现更好，但测试泛化性下降 -5.4%。原因：引导过的轨迹遇到的失败模式更窄，生成的规则覆盖面不足，泛化能力反而弱于从"天真失败"中学习。

## 结论与启发

### 论文结论

1. **启发式规则 > 原始轨迹**：蒸馏经验比直接注入轨迹更有效，且 token 效率更高
2. **LLM 检索 > embedding 检索 > 随机选择**：检索质量决定上限
3. **失败/成功经验的适用场景不同**：失败规则适合 Search（负约束），成功规则适合 Execution（正示范）
4. **单次轨迹即可**：无需多次重试，适用于实际无法重试的部署场景

### 局限性与未来工作

- **无奖励信号时性能下降**：依赖环境反馈，自评准确率仅 70%
- **规则池扩展挑战**：规则冲突解决、大规模池中的检索质量维护
- **双控制场景（含用户交互）表现较弱**
- 未来方向：合成任务自举规则积累、更紧凑的规则表示降低成本、两阶段检索（embedding 粗选 + LLM 精排）

### 启发

- **反思即学习**：无参数更新地利用 LLM 的反思能力，与 RL 范式形成互补
- **经验的可迁移抽象**：将轨迹提炼为带触发条件的规则，是知识复用的关键
- **检索是核心瓶颈**：随着经验规模增长，检索精度将成为瓶颈，值得重点投入
- **适用场景**：工具集固定但数据持续变化的场景（如企业内部 AI 助手），ERL 的部署成本（+40% API 费用）和 7.8% 性能提升是合理的工程权衡
