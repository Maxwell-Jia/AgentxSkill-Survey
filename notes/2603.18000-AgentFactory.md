## 针对的问题

### 现有方法的痛点
当前 Agent 自进化工作主要通过**文本提示、口头反思或推理轨迹**来记录成功经验（如 Reflexion、Self-Refine 等），但这种文本方案无法在复杂现实场景中可靠地保证任务高效重新执行。具体表现为：
- 文本经验无法跨系统迁移
- 仅记录"经验"而非"可执行能力"
- 无法动态适应新变体和边界情形

### 核心洞察
虽然用户任务表面多样，但其实可分解为**可重用的子任务**（如日历管理、文献检索、文件操作）。这意味着应该构建一个能够：
1. 为每个子任务生成并保存**可执行的 Python 代码**
2. 基于执行反馈**自动改进** subagent
3. 将成熟 subagent **跨框架部署**（纯 Python + 标准文档）

---

## 核心思路

> AgentFactory 通过三阶段生命周期（Install → Self-Evolve → Deploy），将成功任务解决方案保存为**可执行 Python 代码**（而非文本经验），实现 subagent 库的持续累积和自进化，并支持跨系统的能力迁移。

## 方法

### 整体框架

![Figure 1: AgentFactory 三阶段管道概览](https://arxiv.org/html/2603.18000v1/x1.png)

**Figure 1**: 展示两个边界情形。Q1：无相关 subagent 存在，Meta-Agent 从零创建并保存到库；Q2：匹配 subagent 已存在，Meta-Agent 复用并通过自进化流程修改。实际场景多数介于两者之间——Meta-Agent 复用部分 subagent，为新问题构建其他 subagent。多轮执行后，每个任务可能创建或修改多个 subagent，逐步构建富有能力的库，最终可导出供其他系统使用。

### 架构三层 Skill 体系

AgentFactory 基于统一 Skill 接口（对标 **Agent Skills 开放标准**），包含三个组件：

#### Meta-Agent 编排器
- **职责**：分解复杂问题为子问题，分配给专门 subagent
- **核心特色**：从 Skill 库动态**精选工具**分配给每个 subagent（而非暴露全部工具让 subagent 自行发现）
  - 减小搜索空间
  - 确保 subagent 聚焦在任务相关的工具集上
- **能力**：维护执行历史，基于结果迭代改进 subagent

#### Skill 系统（三层）

**1) Meta Skills（固定）**
- `create_subagent` — 创建新 subagent，生成 Python 脚本
- `modify_subagent` — 基于反馈修改已保存 subagent
- `run_subagent` — 执行 saved/新建 subagent，全生命周期处理
- `list_saved_subagents` / `view_subagent_code` — 库发现与代码检查
- `get_skill_description` — 获取 Skill 文档
- `finish` — 完成任务，可选持久化 subagent

**2) Tool Skills（固定）**
- `web_search`（Serper）
- `web_reading`（Jina）
- `browser_automation`（Playwright）
- `shell_command`（带安全检查）

**3) Subagent Skills（动态生成与改进）**
- 作为**可执行 Python 脚本**生成，配伴随 SKILL.md 文档
- 随系统遇到新任务而添加，基于执行反馈迭代改进
- 逐渐提升鲁棒性和通用性

#### Workspace Manager
- 为每个任务提供**隔离执行环境**（独立目录）
- 确保 subagent 修改/测试时，失败被隔离，不污染共享库
- 成功完成后，结果和改进 subagent 提升至持久库

### 三阶段生命周期

#### Phase 1: Install —— 从零构建 Subagents

**触发条件**：无现存 Skill 可解决新问题

**流程**：
1. **任务分析** — Meta-Agent 识别所需能力
2. **Subagent 构造** — 分解为多个子问题，对每个调用 `create_subagent` 生成专门 Python 脚本
3. **执行** — Subagent 使用分配的 Tool Skills
4. **持久化** — 成功后保存为纯 Python 代码 + SKILL.md（包含功能、参数、使用说明）

**差异点**：不同于 Reflexion 等仅记口头反思，AgentFactory 将成功执行模式**直接转化为可复用代码**。

#### Phase 2: Self-Evolve —— 基于反馈自动改进

**触发条件**：遇到与已保存 subagent 相似但有变体的新任务

**流程**：
1. **检索** — 用 `list_saved_subagents` 发现相关 Skill
2. **评估** — 运行候选 subagent，测试新需求
3. **反馈分析** — 若失败，分析具体失败模式
4. **自动修改** — 调用 `modify_subagent` 改进代码（添加错误处理、扩展边界情形处理、重构逻辑提升适应性）
5. **验证** — 在当前任务上测试改进

**核心洞察**：将 Self-Refine 的"生成-反馈-修改"循环从**单输出细化**扩展到**Agent 级别代码改进**。不同于仅优化提示词或推理策略的方法，AgentFactory 修改**真实可执行代码**，产生切实能力增长。

#### Phase 3: Deploy —— 导出跨框架部署

**产物形式**：纯 Python 代码 + SKILL.md 文档
- **独立执行** — 各 subagent 可作为独立 Python 脚本运行，无需 AgentFactory 运行时
- **框架集成** — 可集成至其他 Agent 框架（LangChain、AutoGen、Claude Code）
  - 通过提示词让外部 Agent 学习如何调用 subagent 脚本和阅读 SKILL.md 文档
  - **无需修改主框架代码**，纯提示-基础的迁移
- **问题求解** — 外部 Agent 理解库后，可选择并链接多个 subagent 解决复杂任务

**意义**：AgentFactory 不仅是自进化系统，更是为 AI 生态生成可部署 Agent 的工厂。

## 关键设计细节

### 1. Python 代码 + SKILL.md 的"双模态制品"
每个保存的 subagent 包含：
- **可执行 Python 脚本** — 确保跨平台可移植性（任何支持 Python 的系统均可运行）
- **SKILL.md 文档** — 标准化文档描述功能、参数、返回格式、使用示例

这个设计突破了"文本经验"的困境，使 Skill 成为真正的**可运行制品**而非仅记忆。

### 2. 动态 Skill 分配 vs. 全工具暴露
Meta-Agent 对每个 subagent 精选并分配相关工具，而非让 subagent 面对完整工具集。效果：
- 减少搜索空间
- 提升任务聚焦度
- 降低错配风险

### 3. Workspace 隔离与持久化分离
- **运行期**：在临时 Workspace 中修改和测试 subagent
- **成功后**：提升至持久库
- 好处：失败不破坏已有库，并发任务相互不干扰

### 4. 执行反馈驱动的迭代
改进循环的关键是**执行反馈**：
- 若 subagent 运行失败，分析失败原因（错误消息、失败堆栈）
- Meta-Agent 针对具体失败改进代码
- 验证改进效果

这与仅优化提示词的方法根本不同——代码级的改进更直接、更可验证。

## 实验与效果

### 评估设置

**任务设计**
- **Batch 1**（15 任务）：真实任务跨多领域（网络信息检索、数据可视化、浏览器自动化、音频处理）
  - 所有任务均需编写和执行 Python 代码
  - 产出如图表、分析报告
- **Batch 2**（15 任务）：结构相同但具体需求不同，用于迁移评估
  - 每个 Batch 2 任务与对应 Batch 1 任务结构相似但需求差异

**基线**
1. **ReAct**：标准 ReAct 模式，每个任务从零开始，无累积
2. **Self-Evolving Agent (文本经验)**：采纳 Reflexion/Star 思路，保存文本总结（成功/失败经验、经验教训），后续任务检索相关文本指导

**评估指标**
- **编排模型的平均输出 token 数**（每任务，不含 subagent 内部 LLM 消耗）
  - 隔离编排级别的效率增益
  - Token 更少 → 编排工作更轻（subagent 复用更充分）

### 主结果

表格：平均输出 token 数对比

| 设置 | ReAct | 文本经验 | AgentFactory |
|------|-------|---------|-----------|
| Batch 1, from scratch (Opus 4.6) | 8298 | 7642 | 4324 |
| Batch 2, w/ saved (Opus 4.6) | 9145 | 6891 | 2847 |

**关键发现**

1. **Subagent 复用显著降低编排成本**（Batch 2）
   - Batch 2 中使用已保存 subagent 时，AgentFactory 相比 ReAct 和文本经验均显著更优
   - Token 消耗仅为 ReAct 的 ~31%，对比文本经验仅 41%
   - **结论**：可执行 subagent 复用远胜于文本经验

2. **强模型在 Batch 1 就展现复用收益**（Opus 4.6 vs 其他模型）
   - Batch 1 内本无跨任务复用机会（15 任务来自不同领域，交叠有限）
   - 但 Opus 4.6 在 AgentFactory 下仍比 ReAct 减少 48% token
   - **启示**：更强的 LLM 能在同一批次内识别和抓住隐含复用机会，这是好信号——随模型改进，系统自动获益

3. **文本经验方案介于两者之间**
   - 优于 ReAct（因可查询历史）但逊于可执行 subagent
   - 体现了"代码驱动"相比"文本驱动"的本质优势

### 消融与能力增长

虽无直接消融表，但论文通过两个具体案例展示：

**案例 1：迭代细化（Figure 2）**
- 同一 README 生成 subagent 跨三轮演化
  - Run 1：路径解析硬编码（特定项目）
  - Run 2：LLM-based 动态解析，但失败时用脆弱硬编码后退
  - Run 3：Meta-Agent 分析失败，替换为正则提取，鲁棒性大幅提升
- **现象**：每次新任务出现，Meta-Agent 分析反馈，识别弱点，修改代码→更鲁棒、更通用

**案例 2：跨系统迁移（Figure 3）**
- 同一 subagent 库被不同 Agent 系统（AgentFactory → Claude Code）复用
  - Trajectory 1 & 2：AgentFactory 创建 Audio Transcriber、QQ Music Player、Document Creator subagents
  - Trajectory 3：Claude Code 系统通过阅读 SKILL.md 学习如何调用这些 Python 脚本
  - 给定"音频文件描述文档创建任务"，Claude Code 直接调用已保存 subagent，无需重新构建，成功完成
- **现象**：code + 标准文档的组合使 Skill 跨框架可移植

## 结论与启发

### 论文核心贡献
1. **三阶段自进化管道** — Install → Self-Evolve → Deploy 系统性框架
2. **可执行 Subagent 库** — 记录的不是文本而是功能代码，支持持续改进和跨系统复用
3. **实证验证** — 在真实任务上证明 subagent 复用相比文本经验和 ReAct 显著更优

### 对 Skill 沉淀与自进化方向的启发

1. **"制品优于记忆"** — Skill 应沉淀为**可执行代码 + 标准文档**，而非文本反思或提示词。代码的显式性和可验证性强于含糊的文本经验。

2. **自进化的关键是"代码改进"** — 与其优化提示或调整参数，直接修改 Agent 逻辑代码（增加错误处理、扩展分支、重构策略）产生真实能力增长。基于执行反馈的迭代周期是关键。

3. **双模态制品的跨系统价值** — Python 代码 + SKILL.md 文档的组合突破了框架壁垒：任何 Python 环境均可运行，任何 LLM Agent（通过提示学习）均可集成。这为 Skill 市场化和生态化铺路。

4. **动态 Skill 分配 vs. 全工具暴露** — 给 subagent 精选工具而非全部工具，既减搜索空间，又确保任务聚焦。这是提升 Agent 效率的实践经验。

5. **从"场景重复"到"能力积累"** — 日常用户任务看似多样，实际可分解为可复用子任务。系统一旦解决足够多初期任务，积累的 subagent 库就能加速后续问题求解，形成正反馈循环。

6. **跨框架部署的现实意义** — Skill 能否导出到 LangChain、AutoGen、Claude Code 等不同框架决定其生命力。AgentFactory 的"纯 Python + 文档"方案提供了一条实用路径：无需深度框架修改，仅通过提示词就能教会外部 Agent 使用 Skill。

## 技术亮点汇总

| 方面 | 亮点 |
|------|------|
| **记录形式** | 从文本反思 → 可执行 Python 代码 |
| **改进机制** | 从提示优化 → 代码级自动修改 |
| **复用验证** | 从假设 → 真实 token 效率对比（Batch 2 减少 ~70%） |
| **跨系统迁移** | 从框架锁定 → 纯 Python + 标准文档的跨框架可迁移 |
| **自进化信号** | 从一次改进 → 持续多轮迭代细化（如 README 案例的三轮演化） |
