Abstract

RAG（检索增强生成）让 LLM 以外部证据为基础作答，但它把模型变成了搜索结果的被动消费者——模型从不知道语料库是如何组织的、还有哪些内容没被检索到，因此难以回溯或综合散落在不同位置的证据。RAPTOR、GraphRAG 等层次方法虽然离线构建了文档树/图，但查询时仍靠 embedding 相似度检索，层次结构对 LLM 始终不可见。

本文提出 Corpus2Skill：一次性离线将文档语料库编译为可浏览的层次 skill 目录树，serving 时让 LLM agent 通过文件导航工具主动探索这棵树。编译流程迭代聚类文档、在每层生成 LLM 摘要，并将结果物化为可导航的 skill 文件树。Serving 时，agent 获得语料全景视图，沿 topic 分支逐层下钻，最终按 ID 检索完整文档。因为层次结构对 agent 明确可见，它可以推断去哪里找、从无结果的路径回溯，并跨分支综合证据。

在企业客服 RAG benchmark WixQA（6,221 篇 Wix 支持文章，200 道专家问题）上，Corpus2Skill 在全部质量指标上超越稠密检索、RAPTOR 和 Agentic RAG baseline，且 serving 时无需 embedding 索引或向量数据库。

Key Insight

Navigate, don't retrieve。不要把 LLM 当搜索结果的被动接收端——在编译期把语料组织成可浏览的层次 skill 树，serving 期让 agent 主动导航。给 agent 一张"地图"远比给它"搜索结果"更有效：agent 知道去哪里看、知道还有哪些地方没看，可以回溯、可以跨分支综合——这些都是 flat retrieval 无法提供的。

Framework Overview

Corpus2Skill 分两个阶段：编译阶段（Compile Phase）将原始语料库转换为可浏览的 skill 树；Serving 阶段由 LLM agent 导航该树，每次查询无需 embedding 索引。

Figure 2. Corpus2Skill 系统架构。编译阶段（上）：文档 embedding → 多级 K-Means 聚类 → LLM 生成摘要，根节点物化为 SKILL.md，子节点物化为 INDEX.md，叶文档存入独立 documents.json。Serving 阶段（下）：agent 持有预编译 skill 目录 + file navigation + get_document 两个工具，主动导航定位证据。

编译阶段：四步流水线

  1. 文档载入 & Embedding：Qwen3-Embedding-0.6B（1024 维），内容哈希 ID
  2. 迭代层次聚类：K-Means（分支比 $p$，最大顶层簇 $K$），每轮 LLM 生成摘要后重新 embedding，直到簇数 ≤ $K$。层次深度 $O(\log_p N)$
  3. 标签生成：2~5 词文件系统安全标签（如 wix-payments-ecosystem），仅凭标签即可路由
  4. Skill 树物化：根簇 → SKILL.md，子簇 → INDEX.md，叶文档 → documents.json（导航元数据与内容分离，每文件通常 <2 KB）

WixQA 默认配置（$p=10$，$K=7$，3层）：6 个顶层 skill，665 个导航文件，13 MB 文档存储，编译耗时 6.5 分钟。

Serving 阶段：Progressive Disclosure

层次深度公式（信息漏斗）：

$N \xrightarrow{\div p} \dfrac{N}{p} \xrightarrow{\div p} \dfrac{N}{p^2} \cdots \xrightarrow{\div p} \dfrac{N}{p^L} \approx 1$

每次查询 2~3 轮即可定位文档；agent 仅用 ~200 tokens 即可完成第一跳决策。

  • Tool 1：code_execution（浏览导航文件）
  • Tool 2：get_document(doc_id)（取完整文档）

导航文件内容轻量（<2 KB），完整文档按需加载，token 成本仅在真正取证时发生。

从过程型 Skill → 信息型 Skill

先前工作中 skill 编码"如何执行任务"（流程、代码、决策树）。Corpus2Skill 将同一机制重新诠释为编码"语料库包含什么内容"的信息型 skill。agent 走的是同样的 progressive disclosure 流程（描述 → SKILL.md → INDEX.md → 文档），但导航的是信息层次而非执行流程。

这一重新诠释是本文的中心设计洞见：skill 的形式语义 $S = (C, \pi, T, R)$ 中，适用条件 $C$ 是摘要内容，执行策略 $\pi$ 是导航工作流，终止条件 $T$ 是 agent 判断已收集足够证据，可复用接口 $R$ 是标准化文件结构。

宽树是最大杀手（F1 -21%，Factuality -44%，成本反而 +41%）：仅 2 个极宽技能覆盖 ~3000 文档，摘要太泛，agent 无法有效路由。
窄树微优（p=5，4层，+compact）：更细粒度分组 + 密集叶节点视图，成本仅多 8%。

探索预算 & 模型选择

探索预算（轮次）：从 5 到 15 轮，质量变化极小（5 轮 F1=0.453，仅低 1.5%）。层次组织足够良好，agent 很少需要大量探索。

服务模型选择：换用更便宜模型，成本减半（\$0.088 vs \$0.172），F1 降 8%，Factuality 降 12%，但 Context Recall 反而从 0.652 提升到 0.705——廉价模型导航更彻底，取回更多相关文档，尽管答案合成稍欠精炼。

核心启示：系统性能的主要驱动力是编译好的层次质量，而非服务模型的复杂度。

Case Study: 导航轨迹示例

Trace 1：直接下钻

Query: "I need to switch my business type from sole prop to LLC in order to use an EIN."

  1. view wix-commerce-monetization/SKILL.md → 10 子组，选 "wix-payments-ecosystem"（87 docs）
  2. view wix-payments-ecosystem/INDEX.md → 8 叶组，选账户管理子组
  3. view group-05/INDEX.md → 文档 d56cc79: "Changing Your Wix Payments Account Type"
  4. get_document("d56cc797742e") → 完整文章

Answer: "You cannot change your account type directly. Contact Wix Customer Care."

Trace 2：跨分支探索

Query: "I want to change the currency for my course."

  1. view SKILL.md → "wix-online-programs" 和 "wix-billing-documents" 均相关
  2. 进入 wix-online-programs → 找到 "Online Programs: Editing Your Program Settings"（课程设置提到货币是站点级设置）
  3. 进入 wix-billing-documents → 找到 "Setting Your Currency for Accepting Payments"

Answer: "Go to Settings > Language & Region > Currency. This site-level setting applies to all Wix products including Online Programs."

两种典型导航模式：直接下钻（单路径 4 步定位文档）和跨分支探索（同一 skill 内跨主题合并证据）。Agent 每步均由摘要引导决策，结构性视角是 flat retrieval 无法提供的。

Takeaways

核心结论

给 LLM agent 一张结构化的语料"地图"，比给它检索段落获得更高质量答案。agent 本身就是能干的"导航员"——关键是给对地图。invest computation at compile time to make the corpus navigable, rather than at query time to make it searchable。

局限性

  • 每查询成本较高（$0.172）：Skills API 每轮将导航文件内容载入 context，53K tokens/查询。Prompt Caching 可大幅缓解。
  • 依赖 Anthropic Skills API：8 skill/请求、200 文件/skill、30 MB/skill 限制；大型语料需 compact mode 或多级分区。
  • 硬单路径聚类：跨主题文档只归入一个分支，顶层 routing 错误是主要失败原因；soft/multi-parent assignment 是未来方向。
  • 不支持增量更新：新增文档需重新编译整个树（编译成本约 \$5~\$10 + 6.5 分钟）。

对检索 / RAG 方向的启发

  1. Navigability 是被低估的 RAG 维度——编译期组织语料，serving 期无需 embedding 索引；层次深度 $O(\log_p N)$，可扩展到更大语料。
  2. 与 RAPTOR 的本质区别：同样离线聚类+摘要，但 Corpus2Skill 把层次物化为 agent 可直接浏览的 skill 文件，而非再次放入 flat vector store——LLM 真正"看见"语料结构。
  3. Compile-then-navigate 互补 retrieve：适合高价值复杂查询（多跳、跨主题）；简单高频查询 BM25/Dense 仍有性价比优势。
  4. Progressive Disclosure 是 token 效率的关键：仅 ~200 tokens 顶层概览让 agent 做出有依据的第一跳，而非黑盒猜测。
  5. 层次质量决定系统上限：消融证明宽树是最大杀手，服务模型可降级但层次结构不能马虎。
