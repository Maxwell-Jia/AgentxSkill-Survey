Framework Overview. SkillFoundry 以"领域知识树"为控制结构， 从目标分支挖掘异构资源，提取结构化 skill 候选，经多级测试验证后扩展树并剪除冗余叶节点， 形成持续自演化的闭环。

Abstract

现代科学生态系统中存在大量程序性知识（repositories、APIs、脚本、notebooks、文档、数据库、论文）， 但这些知识高度碎片化、散布在异构资源中，LLM agent 无法直接检索、执行、验证或将其组合为可靠工作流。 现有 agent skill 大多仍依赖手工构建，覆盖面窄、难以扩展到多样科学领域； 工具层（tools / MCP）只暴露可调用接口，缺乏协调多步决策所需的程序性指导（procedural know-how）。

SkillFoundry 提出以"领域知识树"为控制结构， 驱动"采矿 → 提取运营契约 → 编译验证 → 迭代精炼"的闭环自演化 pipeline， 将异构科学资源持续转化为结构化、可执行、经过验证的 agent skill 库。 当前库快照已涵盖 286 个 skill，覆盖 27 个领域、 254 个子领域，来自 394 个资源； 71.1% 的挖掘 skill 与已有公开 skill 生态（SkillHub、SkillSMP）不重叠，属全新能力。 在 MoSciBench（6 数据集）、细胞类型注释与 scDRS 工作流三类评测中均显著提升 agent 性能。

Key Insight

将隐藏在异构领域资源中的科学程序性知识，通过领域知识树 + 闭环验证， 自动转化为可插拔、可迁移的 agent skill。 关键不在 skill 数量，而在于是否有可信的验证机制——execution testing + synthetic-data testing 提供了最低执行性保障，使 skill 开箱即用。

Framework Overview

SkillFoundry 的核心是以领域知识树 $T = (V, E)$ 为控制结构的六步闭环过程：内部节点表示领域/子领域，叶节点表示可操作的 skill 目标。 树不是静态分类学，而是自适应挖掘的控制结构—— 系统优先选择"资源丰富但已验证 skill 覆盖薄弱"的分支，而非均匀搜索。

Step 1–2：知识树 + 资源挖掘

按分支优先级检索权威资料：官方文档、维护良好的 repositories、包参考、工作流、notebooks、方法论文。 实现为 staged loop：tree_check → resource_search → skill_build → skill_test → refresh。 共享全局状态（注册表、报告、站点摘要）只由 refresh 阶段写入，防止并发 worker 写入不一致状态。

Step 3：Operational Contract 提取

每个 skill 的核心规格——运营契约（operational contract）——包含： Task Scope、Inputs/Outputs、Dependencies、Execution Steps、Provenance、Follow-up Guidance、Example Invocations / Tests。 编译后的 skill package 不只是文本，还含可执行脚本、来源记录、示例资产， 以及仓库级测试（当相关资产可解析时）。

Step 4–6：验证、树扩展与精炼

三层互补测试：Execution Testing（所有 skill）、System Testing（依赖 SLURM 等集群基础设施）、 Synthetic-data Testing（真实资源不可用时检查契约完整性与行为稳定性）。 验证不通过进入修复循环（repair loop）；通过的 skill 接入树作为新叶节点； 冗余 / 低价值 skill 合并或剪枝；反复失败的分支标记为部分覆盖。

新颖性审查与并行控制

每个候选 skill 同时对比本地库和外部 skill 生态（SkillHub、SkillSMP）， 基于主题定位、来源、范围判断（而非仅词汇相似度）： 全新能力 → 接受为新叶节点；互补 → 合并；完全重复 → 丢弃。 分支级探索可并行，但共享全局状态只由 refresh 阶段统一写入，保证一致性。

模型分配

tree_check / skill_build / refresh / design_skill 阶段使用 GPT-5.4（medium 推理力度）； resource_search 使用 GPT-5.4（high 推理力度，最推理密集）； skill_test / repair / benchmark / novelty_check 使用 GPT-5.4-mini（medium）。

Analysis

Skill 库组成与运行时间。 (a) 286 个 mined skill 在 27 个领域 / 254 个子领域的分布。 (b) 固定资源挖掘预算下各主要 pipeline 阶段的平均运行时间： Skill 提取是最大开销，其次是资源挖掘，测试和树更新相对较小。

Skill 库质量

当前库快照：286 个 skill，覆盖 27 个领域、254 个子领域，来自 394 个资源。 所有保留 skill 均通过必要的验证测试。 71.1% 的 mined skill 与 SkillHub、SkillSMP 中已有 skill 不重叠—— 验证了从异构资源挖掘 skill 的必要性，手工库无法替代。

Task-specific Skill 迁移

SkillFoundry 不仅可构建通用库，还可针对特定任务（如 scDRS 工作流）按需生成 skill， 并直接注入外部 agent（Biomni），无需修改 agent 架构。 这种"skill 作为可移植知识胶囊"的设计，对构建通用科学 agent 基础设施有重要参考意义。

树结构的扩展性优势

领域知识树同时作为搜索先验和库状态摘要， 系统优先填补"资源丰富但 skill 薄弱"的缺口，避免随机漫游和重复采集。 新颖性审查基于"主题定位 + 范围"的语义去重（而非仅词汇相似），维护库的长期质量。

与 Claude Agent Skills 生态的对齐

论文明确参考了 Anthropic 的 Claude Agent Skills 框架（skill.md 格式）， SkillFoundry 的 skill 概念与之对齐。 自动挖掘的 skill 可直接接入 Claude 生态，为领域 agent 快速注入专业程序性知识， 无需定制 wrapper 或重新训练模型。

Takeaways

核心结论

  1. 自动挖掘异构科学资源可产生大规模、高度新颖（71.1%）、内部有效的 skill 库，手工构建无法替代。
  2. Mined skill 在标准 benchmark（MoSciBench 5/6 数据集）和真实科学工作流（细胞注释、scDRS）均可提升 agent 性能。
  3. 同一框架可按需设计任务特定 skill，并将其转移给外部 agent（Biomni），无需修改 agent 本身。
  4. Operational Contract 是 skill 质量的核心抽象——把任务范围、输入输出、依赖、执行步骤、来源、示例打包为可被 agent 发现和执行的结构化单元。

局限性

  • 当前覆盖仍有限，只是演化库的一个快照。
  • 大多数 skill 仅经内部测试验证，未经广泛下游评估。
  • 评估范围集中在少数领域和任务；跨域（不同工具生态）泛化能力有待验证。

对 Skill 自动合成 / 沉淀方向的启发

  1. 领域资源 → Skill 的转化路径已经可行：闭环验证是关键，而非 skill 数量。
  2. 树结构是扩展性的关键：用领域知识树作为搜索先验，可规模化持续填补知识缺口。
  3. 新颖性审查不可跳过：28.9% 候选与已有库重叠，基于语义去重是维护库质量的必要步骤。
  4. Skill 作为可移植知识胶囊：生成的 skill 可直接注入其他 agent，构建通用科学 agent 基础设施。
