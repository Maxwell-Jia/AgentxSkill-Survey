Abstract

现有 LLM 智能体每次执行任务都从零开始，无法利用历史交互中积累的经验。Fine-tuning 成本高昂、对闭源模型不可行，且不支持持续学习。已有工作（ExpeL、AutoGuide）或需要多次重试才能构建对比轨迹，或在每个 agent 步骤都要执行检索，开销大、可扩展性差。

ERL（Experiential Reflective Learning）提出一种轻量级的无参数自我提升范式：agent 对单次任务轨迹进行反思，生成带有触发条件和推荐行动的可复用启发式规则（heuristics），统一存入持久化规则池；面对新任务时，LLM 担任检索排序器，从规则池中按相关性、多样性与信息量选取 $k=20$ 条规则注入 system prompt。在 Gaia2 benchmark 上，ERL 总体成功率达到 56.1%，较 ReAct baseline 提升 +7.8%，较最强先前方法 ExpeL 提升 +5.2%。

Key Insight

将轨迹蒸馏为带触发条件的规则，远优于直接注入原始轨迹（+9.7% vs −1.9%）；在相同 token 预算下，结构化规则能容纳更多经验。检索质量（LLM 排序器）是性能瓶颈：LLM 检索 56.1% > embedding 检索 53.3% > 最优随机配置 53.8%。

Framework Overview

ERL 由两个核心组件构成：启发式规则生成（Heuristic Generation）和检索增强执行（Retrieval-Augmented Execution）。规则以结构化格式存储：分析（识别成功/失败原因）+ 指南（带明确触发条件的可泛化行动建议），无需参数更新即可在不同任务间实现经验迁移。

启发式规则生成

触发时机：agent 完成任务并获得二值成功/失败反馈后。
输入：任务描述 + 完整执行轨迹（推理链、工具调用、输出）+ 结果信号。
输出：结构化规则——"分析"识别原因 + "指南"给出带触发条件的可泛化建议。
关键设计：单次轨迹即可生成规则，无需多次重试，适用于实际不可重复的部署场景。

检索增强执行

面对新任务时，LLM 先将任务分解为潜在子任务，再对规则池按三维标准评分：相似性（与当前任务的匹配度）、多样性（覆盖不同经验教训）、信息量（指南内容丰富程度）。选取 top-$k$（默认 $k=20$）条规则注入 system prompt。
任务级检索而非全量注入，避免规则池扩大时的上下文爆炸与噪声。

规则示例

"当向日历参会者发送邮件时，应先通过 Contacts 工具将姓名解析为邮箱地址，再调用 email API。"

此类规则包含明确触发条件（"当向日历参会者发送邮件时"）和推荐行动（"先查联系人再发邮件"），可直接指导 agent 在相似场景的决策，且无需重新执行历史轨迹。

模型配置与成本（Gaia2 实验）

主执行 agent、规则生成器均采用 gpt-5-mini-2025-08-07；检索排序器采用更强的 gpt-5.2-2025-12-11。整体 API 成本较 baseline 增加约 40%（主要来自规则生成 +$2.42 和检索 +$1.38），场景执行本身仅增加 15%（得益于更高缓存命中率：88% vs 82%）。

Analysis

检索质量 vs. 数量

随机选取规则时，性能在 40~60 条时达峰值后下降（非单调），说明噪声规则有害。 LLM 检索（$k=20$，56.1%）显著优于 embedding 检索（53.3%）和最优随机配置（53.8%）。 结论：检索质量比检索数量更关键——LLM 能理解错误模式与规则内容，而非仅靠语义向量距离匹配。

失败 vs. 成功经验的贡献

失败规则在 Search 任务上效果更佳（+14.3% vs baseline），通过负向约束剪枝无效探索策略； 成功规则在 Execution 任务上贡献更大（+9.0%），通过正向示范强化已验证行动序列。 混合使用是实际部署中最安全的选择（任务分布未知时更稳健）。

规则 vs. 原始轨迹（token 效率）

Few-shot 直接注入原始轨迹：-1.9%（低于 baseline）。ERL 启发式规则：+7.8%。 在相同 token 预算下，结构化规则能容纳更多经验（token 压缩率高），信噪比更佳，迁移学习信号更强。

迭代 ERL 的泛化悖论

积累阶段同步检索已有规则（边积累边使用）：积累阶段表现更好（Exec +2.6% / Search +6.1%），但测试泛化性下降 -5.4%。 原因：引导轨迹遇到的失败模式更窄，生成规则覆盖不足，泛化能力反而弱于从"天真失败"中学习——多样化失败经验是泛化的来源。

Figure 3. 成功率、pass@3 与 pass^3（三次全成功）对比。 ERL 在 pass^3 上提升尤为显著：Execution +8.3%、Search +10.6%—— 表明 ERL 主要提升了任务完成的稳定性，而非解锁全新任务类型。

Takeaways

核心结论

  1. 规则蒸馏 > 原始轨迹：将轨迹提炼为带触发条件的结构化规则，既压缩了 token 成本，又提升了迁移学习信号质量。
  2. LLM 检索 > embedding 检索：LLM 能理解错误模式与规则语义，检索质量是 ERL 性能的关键瓶颈。
  3. 单次轨迹即可学习：无需多次重试，适用于实际不可重复的生产部署场景。
  4. 失败/成功经验的适用场景不同：失败规则适合 Search（负约束剪枝），成功规则适合 Execution（正向示范），混合使用最稳健。
  5. 稳定性提升：ERL 主要提升 pass^3（一致性），说明其核心价值在于消除偶发错误而非扩展能力边界。

局限性

  • 依赖成功/失败反馈信号：自评准确率仅 70%，无反馈时性能从 56.1% 降至 51.2%。
  • 双控制场景（含用户交互）表现较弱：Telecom 分割下降，因任务组合空间大、历史规则难以匹配。
  • 规则池随经验积累增大后，冲突解决与检索精度维护的挑战尚未充分解决。
  • 整体 API 成本增加约 40%，生产部署需权衡性能收益与成本。

对后续研究的启发

  1. 反思即学习：无参数更新地利用 LLM 反思能力，与 RL 范式形成互补，适合闭源模型或快速迭代场景。
  2. 经验的可迁移抽象：带触发条件的规则是知识复用的有效表示；这一思路与 SkillX 的 functional skills 形成呼应——均强调经验的结构化表示优于原始记录。
  3. 检索是核心瓶颈：随规模增长，两阶段检索（embedding 粗选 + LLM 精排）值得重点投入。
  4. 多样化失败经验是泛化的来源：不要过早在积累阶段"引导"agent，保持探索多样性更有利于后期泛化。
