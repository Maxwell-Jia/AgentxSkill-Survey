Figure 1. AutoSkill 框架由两个紧密耦合的过程组成：右侧循环（技能演化）从对话历史持续提取、合并、精炼显式技能；左侧循环（技能增强响应生成）通过查询重写 → 混合检索 → 上下文注入支持当前响应。系统通过外部显式技能记忆成长，而非模型微调。

Abstract

现实 LLM 应用中，用户会反复表达相同偏好与要求——"避免幻觉"、"遵循机构写作规范"、"减少技术术语"——但这些交互经验从未被系统性沉淀为可复用知识。现有记忆增强方案（MemoryBank、MemGPT、Mem0 等）仅将过去对话当作可检索的文本片段，而非可操作化的行为能力；参数更新路线（Self-evolution、RLHF）代价高昂且缺乏透明度；Voyager、Reflexion 等 Agent 框架中的技能通常隐含于 prompt 或轨迹，缺乏统一的检视、编辑、迁移与长期维护机制。

AutoSkill 将用户交互经验抽象为带版本号的显式技能制品（SKILL.md），通过"技能提取 → 检索辅助管理 → 版本化合并 → 混合检索注入"的闭环实现无需参数更新的终身个性化学习。在 WildChat-1M 真实对话数据上，系统从超过 10,000 段对话中自动提取近 1,858 个技能，并通过案例验证了版本化演化机制的有效性——高频生产力技能历经数十次精炼迭代，版本号最高达 0.1.34。

Key Insight

将技能显式化为可版本管理的 SKILL.md 制品，比隐藏在 prompt 或策略中具有更高的可维护性与可迁移性；仅从用户查询侧提取技能（不含模型回复），确保捕获稳定用户需求而非模型生成偏差。

Framework Overview

AutoSkill 的系统模块集合定义为 $\mathcal{M} = \{M_{\mathrm{rw}}, M_{\mathrm{chat}}, M_{\mathrm{ext}}, M_{\mathrm{judge}}, M_{\mathrm{merge}}, M_{\mathrm{emb}}\}$，全部由 Prompt 驱动，无需专项训练。整体运行于两个并发循环：Loop 1 负责在线技能增强响应生成，Loop 2 负责后台实时技能演化，两者通过共享技能库 $\mathcal{B}_u^t$ 解耦。

技能七元组表示

每个技能 $s = (n, d, p, \tau, \gamma, \xi, v)$，字段分别为：名称 $n$、描述 $d$、可执行指令 prompt $p$、触发条件集 $\tau$、标签集 $\gamma$、示例集 $\xi$、版本号 $v$。技能以 SKILL.md 持久化，可人工检视、编辑、导入导出，亦可跨用户共享。

Loop 1：技能增强响应生成

步骤 1 查询重写：$\tilde{q}_t = M_{\mathrm{rw}}(P_{\mathrm{rw}}, q_t, h_t)$，解析上下文依赖、暴露格式/风格/领域等检索约束。
步骤 2 混合检索：$\mathrm{Rel}(q_t, s) = \lambda\hat{d}(q_t,s) + (1-\lambda)\hat{b}(q_t,s)$（Dense + BM25），取 Top-K 超阈值 $\eta$ 的技能集 $\mathcal{H}_t$。
步骤 3 注入生成：将 $\mathcal{H}_t$ 渲染为紧凑上下文块 $C_t$ 注入对话模型：$r_t = M_{\mathrm{chat}}(P_{\mathrm{chat}}, q_t, h_t, C_t)$。

Loop 2：实时技能演化

提取：仅用用户查询序列 $z_t = M_{\mathrm{ext}}(P_{\mathrm{ext}}, \mathcal{Q}_u^t)$，过滤一次性请求，捕获持久偏好、可复用流程、输出约束等。
管理：检索最近邻 $s_t^*$，由判断模块决策 $a_t \in \{\texttt{add}, \texttt{merge}, \texttt{discard}\}$。
合并：merge 时版本化演化 $s_t' = M_{\mathrm{merge}}(\cdot)$，$v(s_t') = \mathrm{Bump}(v(s_t^*))$，保持库一致性。

技能库更新规则：

$$\mathcal{B}_u^{t+1} = \begin{cases} \mathcal{B}_u^t \cup \{z_t\}, & a_t = \texttt{add} \\ (\mathcal{B}_u^t \setminus \{s_t^*\}) \cup \{s_t'\}, & a_t = \texttt{merge} \\ \mathcal{B}_u^t, & a_t = \texttt{discard} \end{cases}$$

技能生命周期（四阶段）：① 经验摄入——摄入对话消息与行为事件轨迹；② 技能提取——从摄入证据中提取可复用能力候选；③ 维护与版本控制——add / merge / discard 三选一，合并时更新版本号；④ 技能复用——混合检索相关技能，渲染为上下文后注入 LLM 请求。

系统部署模式支持四种接入方式：Python SDK（ingest / search / render_context）、Web UI（前台检索注入 + 后台异步演化）、OpenAI 兼容反向代理（透明增强现有客户端）、离线引导（批量导入历史对话初始化 SkillBank）。

Analysis

关键设计：仅用查询侧提取

技能提取阶段只用用户查询序列（不含模型回复）：$z_t = M_{\mathrm{ext}}(P_{\mathrm{ext}}, \mathcal{Q}_u^t)$。这确保提取到的是稳定的用户需求，而非模型生成偏差。对需要对齐真实用户意图的系统尤为重要——若从模型回复中学习，系统可能逐渐强化模型本身的倾向而非用户偏好。

最近邻局部管理 vs. 全库比较

候选技能不直接入库，而是先检索最相似的现有技能作为局部证据，仅在该局部范围内由 judge/merge 模块决策。相比全库遍历，这显著减少输入规模、使决策更聚焦且更可扩展。混合检索（Dense + BM25）则确保语义相关与关键词精确触发两者兼顾：阈值 $\eta$ 过滤无关技能，避免噪声干扰响应生成。

版本化合并 vs. 重复积累

传统 RAG 系统倾向于堆积相似 chunk，导致库膨胀与质量下降。AutoSkill 的 merge + version 机制在同一技能的后续反馈触发时直接迭代更新而不产生重复 prompt 片段，有效控制库规模并保持语义一致性。这是构建可长期维护技能库（乃至"技能市场"）的关键工程范式。

异步演化解耦在线延迟

技能提取在后台异步执行，与在线响应生成完全解耦：用户感知到的延迟仅来自检索注入，而非耗时的 LLM 提取与 merge 操作。这是面向生产部署的关键工程决策。系统同时支持离线引导模式——批量导入历史对话、文档或 Agent 轨迹初始化 SkillBank，无需等待实时积累即可冷启动。

Takeaways

核心结论

  1. 技能即制品（Skill-as-Artifact）：将技能显式化为可版本管理的 SKILL.md 文件，比隐藏在 prompt 或策略中具有更高的可维护性——这一范式值得在更多 Agent 框架中推广。
  2. 仅从用户侧信号学习：提取阶段只用用户查询而非模型回复，是避免学习到模型偏差的巧妙设计，对需要对齐真实用户意图的系统尤为重要。
  3. 版本化合并替代重复积累：merge + version 机制保持技能库的一致性，有效控制库规模与质量，可借鉴于其他知识管理系统。
  4. 异步演化解耦延迟：将技能提取放在后台异步执行，是面向生产部署的关键工程决策。
  5. 跨用户迁移潜力：SKILL.md 的标准化表示使得技能在用户间、Agent 间共享成为可能，是构建"技能市场"或组织级技能知识库的基础。

局限性

  • 实验以描述性分析为主，缺乏与基线方法的系统性量化对比。
  • 技能质量评估主要依赖案例分析，端到端效果有待配套评测 benchmark（如 StuLife）进一步验证。
  • 当前主要针对文本对话场景，对工具调用型 Agent 的覆盖有待扩展。

对后续研究的启发

  1. 将"终身学习"问题从隐式记忆积累重新表述为显式技能演化，是一个值得广泛采用的研究范式。
  2. SKILL.md 制品标准化为跨框架的"技能市场"奠定了基础——多 Agent 协作时可直接共享技能库。
  3. 版本化管理与 add/merge/discard 三路决策，是维护任何领域知识库质量的通用有效机制。
  4. 混合检索（Dense + BM25）+ 阈值过滤是技能检索的实用基线，应优先于纯语义检索。
