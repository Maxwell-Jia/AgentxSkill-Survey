AgentFactory: A Self-Evolving Framework
Through Executable Subagent Accumulation and Reuse

通过可执行子 Agent 累积与复用实现自进化的框架

Figure 1. AgentFactory 三阶段管道概览。 Q1（无相关 subagent）：Meta-Agent 从零创建并持久化到库； Q2（已有 subagent）：复用并通过自进化流程修改。 多轮执行后，subagent 库逐步积累，最终可导出至其他 Agent 框架（LangChain、AutoGen、Claude Code 等）。

Abstract

现有 Agent 自进化工作（如 Reflexion、Self-Refine）主要依赖文本提示、口头反思或推理轨迹来记录成功经验，但文本化经验无法在复杂现实场景中可靠保证任务的高效重执行，也难以跨系统迁移。

AgentFactory 提出将成功任务解决方案保存为可执行 Python 代码 + SKILL.md 文档（双模态制品），而非文本反思。通过三阶段生命周期（Install → Self-Evolve → Deploy），构建一个持续累积和自进化的 Subagent 库：强模型（Opus 4.6）驱动的 AgentFactory 相比 ReAct 基线减少约 ~70% 编排 token 消耗（Batch 2 复用场景），且 subagent 库可导出为框架无关的可移植制品，供 Claude Code 等外部系统直接调用。

Key Insight

Skill 应沉淀为可执行代码而非文本记忆。代码的显式性和可执行验证性，使 Agent 自进化从"提示优化"上升为真实的"能力增长"，并突破框架壁垒实现跨系统迁移。

Framework Overview

AgentFactory 的核心架构由 Meta-Agent 编排器、三层 Skill 体系与 Workspace Manager 构成。Meta-Agent 分解复杂问题，动态从 Skill 库精选工具分配给每个 Subagent（而非全量暴露）；Workspace Manager 提供隔离执行环境，成功后再将结果提升至持久库。

Phase 1: Install

触发：无现存 Skill 可解决新问题。
Meta-Agent 识别所需能力，分解为子问题，对每个调用 create_subagent 生成专门 Python 脚本，执行成功后保存为可执行代码 + SKILL.md。 不同于 Reflexion 的口头反思，AgentFactory 将成功模式直接转化为可复用代码。

Phase 2: Self-Evolve

触发：遇到与已保存 subagent 相似但有变体的新任务。
检索相关 Skill → 运行 → 分析失败模式 → 调用 modify_subagent 自动改进代码（增加错误处理、扩展边界情形）→ 验证。 改进的是真实可执行代码，产生切实能力增长。

Phase 3: Deploy

产物：纯 Python 脚本 + SKILL.md 文档，独立可运行，无需 AgentFactory 运行时。
可集成至 LangChain、AutoGen、Claude Code——通过提示词教会外部 Agent 读取 SKILL.md 并调用脚本，无需修改主框架代码。

三层 Skill 体系

Meta Skills（固定）

create_subagent / modify_subagent / run_subagent / list_saved_subagents / view_subagent_code / get_skill_description / finish

Tool Skills（固定）

web_search（Serper）/ web_reading（Jina）/ browser_automation（Playwright）/ shell_command（带安全检查）

Subagent Skills（动态）

运行时自动生成与改进的可执行 Python 脚本，随系统遇到新任务持续添加，基于执行反馈迭代提升鲁棒性和通用性。

Results

指标：Meta-Agent 平均输出 token 数（每任务，不含 subagent 内部 LLM 消耗）。越少代表编排更高效、复用更充分。

设置 ReAct 文本经验（Self-Evolving） AgentFactory
Batch 1，从零开始（Opus 4.6） 8298 7642 4324
Batch 2，使用已保存 subagent（Opus 4.6） 9145 6891 2847
  • Batch 2（复用场景）中，AgentFactory token 消耗仅为 ReAct 的 ~31%，相比文本经验仅 ~41%——可执行 subagent 复用远胜文本经验。
  • Batch 1 中 Opus 4.6 驱动的 AgentFactory 仍比 ReAct 减少 ~48% token，说明强模型能在同批次内识别隐含复用机会。
  • 文本经验方案介于两者之间：优于 ReAct（可查历史），但逊于可执行 subagent，体现"代码驱动"vs"文本驱动"的本质差异。

Analysis

Figure 3. 跨系统迁移案例：AgentFactory 先后创建 Audio Transcriber、QQ Music Player、Document Creator subagent；Claude Code 通过读取 SKILL.md 学习调用方式，无需重建即可复用这些 Python 脚本，成功完成"音频文件描述文档创建"任务。

迭代细化：代码级自进化

同一 README 生成 subagent 经历三轮演化（Figure 2）：
Run 1：路径解析硬编码（特定项目）
Run 2：LLM-based 动态解析，但失败时用脆弱硬编码后退
Run 3：Meta-Agent 分析失败，替换为正则提取，鲁棒性大幅提升
每次新任务出现，Meta-Agent 分析反馈、识别弱点、修改代码——每一轮产生实质能力增长。

关键设计决策

动态 Skill 分配 vs. 全工具暴露：Meta-Agent 对每个 subagent 精选相关工具而非暴露全集，减少搜索空间、提升任务聚焦度、降低错配风险。

Workspace 隔离：运行期在临时 Workspace 中修改和测试，成功后才提升至持久库，失败不破坏已有库，并发任务互不干扰。

双模态制品：Python 代码确保跨平台可移植（任何支持 Python 的系统均可运行）；SKILL.md 标准化文档使外部 LLM Agent 能够通过提示学习调用方式。

Takeaways

核心贡献

  1. 三阶段自进化管道（Install → Self-Evolve → Deploy）——系统性框架，使 Agent 从"一次性执行"走向"能力持续积累"。
  2. 可执行 Subagent 库——记录的是功能代码而非文本，支持持续改进和跨系统复用；Batch 2 复用场景下编排 token 消耗降低约 70%。
  3. 跨框架 Deploy——纯 Python + SKILL.md 使 Skill 可无缝迁移至 LangChain、AutoGen、Claude Code 等生态，无需修改主框架代码。

对 Skill 沉淀方向的启发

  1. "制品优于记忆" — Skill 沉淀为可执行代码 + 标准文档，而非文本反思或提示词。代码的显式性和可验证性强于含糊的文本经验。
  2. 自进化的关键是"代码改进" — 直接修改 Agent 逻辑代码（增加错误处理、扩展分支）产生真实能力增长；执行反馈驱动的迭代周期是关键。
  3. 从"场景重复"到"能力积累" — 用户任务虽多样，实质可分解为可复用子任务；库一旦足够丰富，正反馈循环加速后续问题求解。
  4. 跨框架部署决定 Skill 的生命力 — "纯 Python + 文档"方案提供了不依赖特定框架、仅靠提示词即可迁移的实用路径。

局限性

  • 评估规模较小（每批 15 个任务），缺少大规模 benchmark 对比。
  • 指标仅为编排 token 数，未涵盖任务成功率、代码质量等多维度评估。
  • Subagent 与具体 Python 生态绑定，纯对话型 Agent 场景（无代码执行）覆盖不足。
