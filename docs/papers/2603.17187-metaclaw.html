Abstract

LLM agent 在真实部署中面临根本矛盾：必须持续不中断地服务用户，同时其能力随任务分布漂移而逐渐过时——模型一经训练便静止不动。现有方法各有致命缺陷： Memory-based 方法（Reflexion、Mem0 等）存储原始轨迹，未提炼可复用行为知识； Skill-based 方法（Voyager、ExpeL 等）将经验压缩为 skill 指令，但 skill 库静态，从不与权重优化协调； RL-based 方法（GRPO、DAPO 等）更新模型权重，却遭受陈旧奖励污染——skill 已演化后，旧 skill 上下文收集的轨迹奖励信号已失效。

MetaClaw 将 LLM policy（权重 $\theta$）与 behavioral skill 库（$S$）联合演化： 技能驱动的即时 prompt 注入（零停机）与基于空闲时段的 RL 权重更新（opportunistic updates）相互强化，形成持续演化的正向循环。 在 MetaClaw-Bench（934 题 / 44 模拟工作日）与 AutoResearchClaw（23 阶段全自动科研 pipeline）上， Full Pipeline 将 Kimi-K2.5 的任务完成率提升 8.25×，并几乎追平更强的 GPT-5.2 baseline。

Key Insight

技能库（$S$）是天然的 meta-parameter。 自然语言行为指令跨任务可迁移，一次失败蒸馏的 skill 可泛化到结构不同的后续任务。 两种时间尺度（prompt-level 秒级 vs. weight-level 小时级）互补而非竞争—— 仅 skill 无法解锁文件执行完成率，仅 RL 缺乏结构化失败分析则更新低效，两者缺一效果均不完整。

Framework Overview

MetaClaw 的核心抽象是 meta-model $M = (\theta, S)$： $\theta$ 是基础 LLM policy 的权重参数；$S = \{s_1, s_2, \ldots, s_K\}$ 是 skill 指令库，每条 skill 是简洁、可复用的自然语言行为指令。 推理时先通过嵌入检索取出与当前任务最相关的 skill 子集，再输入给 policy：

$$a \sim \pi_\theta(\cdot \mid \tau,\, \mathrm{Retrieve}(S, \tau))$$

系统通过四个核心机制实现持续 meta-learning，覆盖"失败分析 → 技能蒸馏 → 版本控制 → 机会性训练"完整闭环。

机制一：Skill-Driven Fast Adaptation

当失败轨迹积累到阈值后，LLM evolver $\varepsilon$ 分析失败，合成新的行为指令并并入 skill 库：

$S_{g+1} = S_g \cup \varepsilon(S_g,\, D^{\mathrm{sup}}_g)$

$g$ 是 skill generation（技能代数），每次 skill 库更新时递增。 完全 gradient-free，通过 prompt 注入生效，零停机。 技能库具有双重角色：作为 meta-parameter 跨任务流积累行为知识（$S_{g+1} \supseteq S_g$）， 同时作为推理时的 adaptation basis 按任务检索。

机制二：Opportunistic Policy Optimization

在用户空闲时段，对 post-adaptation 轨迹（query data）执行 RL 权重更新：

$\theta_{t+1} = \theta_t + \alpha\, \nabla_\theta\, \mathbb{E}_{(\tau,\xi,g') \sim B}\!\left[R\!\left(\pi_\theta(\cdot \mid \tau, S_{g'})\right)\right]$

$R$ 为 Process Reward Model（PRM）评分；实现方式为 Cloud LoRA fine-tuning + GRPO，无需本地 GPU。 RL buffer $B$ 只接受 query data（post-adaptation 轨迹），support data 被排除在外。

机制三：Skill Generation Versioning

陈旧奖励污染问题：触发 skill 演化的轨迹（support data）其奖励反映的是旧 skill 下的行为， 若进入 RL buffer 会污染梯度。解决方案：每条样本打上 skill generation index $g_i$ 的戳记。

  • Support data $D^{\mathrm{sup}}_g$：触发 $S_g \to S_{g+1}$ 演化的轨迹，被 evolver 消费后从 RL buffer 中丢弃
  • Query data $D^{\mathrm{qry}}_{g+1}$：$S_{g+1}$ 生效后收集的轨迹，才可进入 RL buffer

当 skill generation 从 $g$ 推进到 $g+1$ 时，训练器自动清除 buffer 中所有版本 $\le g$ 的样本。 这是在线异步 skill 演化场景下必要的正确性保证，不可选。

机制四：Opportunistic Meta-Learning Scheduler（OMLS）

OMLS 是后台 daemon，监控三类空闲信号，只在用户不活跃时触发 RL 训练：

信号说明
Sleep window用户配置的睡眠时段（如 23:00–07:00）
System inactivity轮询 OS 输入设备空闲计时器（macOS: ioreg HIDIdleTime），默认 30 分钟
Calendar-aware查询 Google Calendar API，会议时段推断用户不可用

任意信号指示"用户离开"→ 训练窗口开启；任意信号指示"用户返回"→ 训练暂停。 支持 mid-batch checkpointing，断点续训。

Algorithm 1：MetaClaw 完整流程

输入：$M = (\theta_0, S_0)$，skill evolver $\varepsilon$，task stream $\{\tau_i\}$，PRM $R$，OMLS 初始化：$g \leftarrow 0$，RL buffer $B \leftarrow \emptyset$ for 每条任务 $\tau_i$： // 服务任务 $S_{\tau_i} \leftarrow \mathrm{Retrieve}(S_g, \tau_i)$ // 检索相关技能 $\xi_i \leftarrow \mathrm{Execute}(\pi_\theta(\cdot \mid \tau_i, S_{\tau_i}))$ // 执行并收集轨迹 $r_i \leftarrow R(\xi_i)$；给 $(\tau_i, \xi_i, r_i)$ 打上代数 $g$ if $\xi_i$ 揭示失败：加入 support set $D^{\mathrm{sup}}_g$ else：加入 RL buffer $B$（query data） // Skill-driven fast adaptation if $|D^{\mathrm{sup}}_g| \ge$ 阈值： $\Delta S \leftarrow \varepsilon(S_g, D^{\mathrm{sup}}_g)$ // 从失败中合成新技能 $S_{g+1} \leftarrow S_g \cup \Delta S$ // 演化技能库 清除 $B$ 中所有版本 $\le g$ 的样本 // 支持-查询分离 $g \leftarrow g + 1$ // Opportunistic policy optimization if OMLS 检测到空闲窗口 and $|B| \ge$ batch size： $\theta \leftarrow \theta + \alpha \nabla_\theta\, \mathbb{E}_{(\tau,\xi,r,g') \sim B}[R(\pi_\theta(\cdot \mid \tau, S_{g'}))]$ 热替换模型权重

Takeaways

核心结论

  1. 技能库是天然的 meta-parameter。 自然语言指令跨任务可迁移，一次失败蒸馏的 skill 可泛化到结构不同的后续任务——这是传统 memory 方法（存原始轨迹）做不到的。
  2. 双时间尺度协同是关键架构决策。 Prompt-level 适应（秒级）与 weight-level 优化（小时级）是互补关系而非竞争： 前者提供即时知识注入，后者将隐性规则内化到参数中。两者缺一，效果均不完整。
  3. 版本控制是正确性保证。 在在线、异步的 skill 演化场景下，support/query 分离是必要的正确性保证—— 不加版本控制会导致陈旧奖励污染，等价于在已修正错误的前提下仍惩罚 policy。
  4. 弱模型 + MetaClaw ≈ 强模型 baseline。 Kimi-K2.5（弱）+ Full Pipeline 几乎追平 GPT-5.2（强）的 baseline， 对"以经济模型服务生产规模"场景具有实际价值。
  5. 零停机 skill 注入具有跨域泛化能力。 AutoResearchClaw 证明，为 CLI 任务设计的 skill evolver 可直接迁移到多阶段科研 pipeline， 无需任何领域特定调整——skill 的自然语言行为指令形式本身就是领域无关的。

局限性

  • 空闲窗口检测依赖用户配置，未必适用所有部署环境（服务器端无用户接口、多租户场景等）。
  • Skill 库只增不减（$S_{g+1} \supseteq S_g$），长期运行后可能出现技能冗余或冲突，尚无修剪机制。
  • 当前在 OpenClaw（单用户 CLI agent）上验证，尚未在多用户并发场景下测试版本隔离的正确性。

对 Agent × Skill 自进化方向的启发

  1. 将 support/query 分离的思路推广到在线 continual learning 是通用设计模式，适用任何"技能/记忆在线演化 + 权重优化"的场景。
  2. OMLS 的"机会性训练"为需要 GPU 计算但又不能中断服务的 agent 提供了一个可工程化的落地路径。
  3. 失败轨迹的结构化分析（而非直接存储）是提炼跨任务 skill 的关键；skill evolver 的设计质量决定技能代数演化的上限。
