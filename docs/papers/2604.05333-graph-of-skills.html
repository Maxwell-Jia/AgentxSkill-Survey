Abstract

现代 Agent 系统中，skill library 正从数十个快速扩展到数千、数万个。两种主流策略各有根本缺陷：Vanilla Skills（全量加载）token 成本线性增长且存在 "Lost in the Middle" 现象；Vector Skills（向量检索）按语义相似度 top-k，但顶级语义匹配往往是高层求解器技能，真正执行所需的底层解析器、预处理、格式化工具因语义距离远而被漏掉——形成 prerequisite gap。

GoS 离线从本地技能包构建 类型化有向技能图，推理时通过混合语义–词法播种 + 反向感知 Personalized PageRank 图扩散 + 预算约束水化，输出一个依赖完备、token 高效的技能执行包。在 SkillsBench（1000 技能 / 11 领域）与 ALFWorld 上，跨 Claude Sonnet 4.5、MiniMax M2.7、GPT-5.2 Codex 三家模型：reward 相对 Vanilla 平均 +43.6%、token 平均 −37.8%，所有 6 个模型 × benchmark 组合中均取得最高 reward。

Key Insight

语义接近 ≠ 可执行完备。仅靠向量检索对长链技术任务不够——检索系统必须显式建模技能间的依赖结构，并沿依赖边反向传播相关性，才能组装可直接执行的最小完整技能包。

Framework Overview

GoS 将本地技能语料库转换为类型化有向图 $G = (V, E, w, \varphi)$，边类型 $\varphi(e) \in \{\text{dep}, \text{wf}, \text{sem}, \text{alt}\}$。给定查询 $q$ 与上下文预算 $\tau$，检索目标为找到束 $B(q) \subseteq V$ 最大化"相关性 + 依赖完备性"：

$\max_{B\subseteq V}\ \sum_{v\in B} \text{rel}(v,q) + \beta \sum_{(u,v)\in E_{\text{dep}}} \mathbb{1}[u\in B \land v\in B]\quad \text{s.t.}\ \text{cost}(B) \le \tau$

通过三阶段近似求解：混合种子检索 → 反向感知图扩散 → 预算化重排与水化。

Figure 2. GoS 总体框架——离线索引构建类型化有向图；在线检索经"混合播种 → 反向感知 PPR → 预算化水化"输出执行包。

四类边

  • dep（dependency）— I/O 兼容性确定性归纳：$v_i$ 输出与 $v_j$ 输入 overlap 超阈值。
  • wf（workflow）— 常见多步流程中的顺序关系。
  • sem（semantic）— 近似重复或主题相邻。
  • alt（alternative）— 同一子问题的可互换策略。

非依赖边通过"稀疏验证"建立：先构造候选池（词法 + 语义邻居 + I/O 扩展），池内 LLM 关系验证（高精度优先）。

反向感知类型扩散

统一转移算子：

$T = \text{RowNorm}\!\left(\sum_{r} \lambda_r (T_r^{\to} + \gamma_r T_r^{\leftarrow})\right)$

Personalized PageRank 扩散至收敛：

$s^{(\ell+1)} = \alpha \mathbf{p} + (1-\alpha) T^\top s^{(\ell)}$

dep 边的反向系数 $\gamma_{\text{dep}}$ 最大——沿依赖边反向恢复上游 prerequisite 是 GoS 的核心机制。

Analysis

库规模敏感性

GPT-5.2 Codex 在 SkillsBench 上将库规模从 200 扩展到 2000：Vanilla 的 token 从 1.93M 增长到 5.84M（约 3×），而 GoS 维持在 1.14~1.38M。200 技能小规模时 Vanilla 与 GoS reward 相当；一旦库规模增大，GoS 稳定领先，优势不随库规模增大而削弱。

Takeaways

核心结论

在大规模技能库中，结构化依赖感知检索优于语义相似度检索和全量加载，且这一优势在跨 benchmark、跨模型家族、跨库规模下均成立。

局限性

  • 依赖图质量是上限——文档不完整 / I/O schema 模糊会降低边质量。
  • 当前图是静态的，不从执行轨迹或用户反馈中动态更新。

对 Skill 管理与检索方向的启发

  1. Prerequisite gap 是实际问题——长链任务必须显式建模技能间依赖。
  2. 图 + PPR 是可落地解法——离线建图成本摊薄到检索时，inference 开销可控。
  3. 混合种子优于单一信号——词法擅长具体文件名/API，语义擅长抽象意图。
  4. 水化是最后一公里——检索结果需物化为 agent 可直接执行的负载（含源路径 + 执行注意事项）。
  5. Agent Interface 是方法的一部分——如何把检索结果"强制"传达给 agent，与检索质量同等重要。
