Abstract

在多用户 Agent 生态中，skill 在部署后基本静止不动：同样的工具调用错误、参数格式问题、执行失败模式在不同会话、不同用户间被反复踩坑，系统从未从这些经验中受益。Agent 经过多轮试错找到的更稳健解法仅限于当次会话；用户间的经验形成孤岛，无法互通；而现有记忆方法只存原始轨迹，不能提炼为可复用的行为。

SkillClaw 将跨用户交互轨迹聚合为共享证据库，由一个自主演化器（Agentic Evolver）以开放式推理驱动 skill 的精炼（Refine）与新建（Create），再将更新后的 skill 集合通过夜间验证门同步回所有 Agent，形成"交互 → 证据 → 演化 → 验证 → 部署"的持续闭环。在 WildClawBench（60 任务 / 6 类别）上，基于 Qwen3-Max，6 天集体演化后各类别成绩显著提升：Creative Synthesis +88.4%、Search & Retrieval +52.0%、Safety & Alignment +33.3%、Social Interaction +11.7%。

Key Insight

跨用户交互轨迹聚合是集体演化的基础。单用户产生的信号难以区分"可泛化改进"与"个人特殊解法"；将多用户对同一 skill 的不同结果聚合，形成天然对照实验，为稳定演化提供更可靠的证据基础。夜间验证门将候选 skill 推送与实际部署解耦，保证共享 skill 池单调不降——用户始终享受经验证的最优版本。

Framework Overview

SkillClaw 采用去中心收集、中心演化的部署架构。Client Proxy（用户侧本地代理）拦截每次 Agent 请求，记录完整因果链；Evolve Server（可选中心服务端）从共享存储（OSS / S3 / 本地 FS）读取会话数据，运行演化引擎，将更新后的 skill 写回共享存储。Agentic Evolver 对聚合的跨用户会话轨迹执行三阶段分析，每日夜间触发：

$S' \leftarrow \text{Evolve}(S,\; \{G(s)\}_{s \in S} \cup G(\varnothing))$  ·  候选通过夜间验证门后写入 SkillHub

Agentic Evolver 三阶段流水线（Evidence → Attribution → Evolution）及多用户 skill 分发闭环。

Phase 1  Trajectory Collection

Client Proxy 拦截 /v1/chat/completions、/v1/messages，记录完整因果链：prompt → action → feedback → … → response，含所有工具调用、中间错误、用户反馈。

从每条会话提取：引用了哪些 skill、是否有工具错误、粗粒度质量估计（PRM 评分）；按引用 skill 分组 $G(s) = \{\tau_i \mid s \in K_i\}$，未使用 skill 的会话归入 $G(\varnothing)$。

Phase 2  Skill Aggregation

Agentic Evolver 以开放式推理联合分析成功与失败会话：成功会话定义 skill 的不变量（不可改动），失败会话定义改进目标。

区分失败根因：Skill 问题（错误/缺失/误导性指导）→ 修改 skill；Agent 问题（误用/上下文溢出）→ 不修改；环境问题（API 不稳定）→ 酌情添加简短提示。

Phase 3  Skill Update Policy

对每个 skill group 选一动作：

  • Refine — 精炼现有 skill（多会话指向同一缺陷）
  • Create — 创建新 skill（$G(\varnothing)$ 发现可复用模式）
  • Skip — 维持不变（证据不足或非 skill 缺陷）

保守编辑原则：局部精修而非全量重写；必须先读 history/ 全部历史版本；不随意改动 API 端点/端口——除非会话证据明确表明已更改。

Skill 表示：SKILL.md + 版本化历史

每个 skill 以 SKILL.md 文件存储（YAML frontmatter 元信息 + Markdown 正文）。核心设计原则：skill 是环境信息压缩器——只记录 Agent 无法自行发现的领域特定细节（API 端口、payload 格式、工具名称、输出路径），而非通用最佳实践（重试逻辑、缓存策略等）。

每个 skill 目录下有 history/ 子目录，按 v<N>.md（skill 快照）和 v<N>_evidence.md（本轮驱动修改的会话证据）分版本存储，形成完整的演化审计链。跳过历史读取被视为硬性错误（hard error）。

典型演化案例（Slack 消息分析 skill）：API 端口从 localhost:9100（错误）修正为 localhost:9110；消息获取策略从全量检索改为先扫描预览再按需拉取；输出路径明确写入 /tmp_workspace/results/results.md；错误处理从 trial-and-error 重试变为 API 配置固化在 skill 中主动规避。

夜间验证门（Nighttime Validation Gate）

候选 skill 更新不直接部署，而是经过验证才写入共享仓库。验证在夜间于空闲用户环境执行，对候选 $s'$ 和现有 $s$ 从白天交互数据中抽取相关任务并在相同环境下执行两者并比较结果。若 $s'$ 表现优于 $s$，标记为 Accept 并次日推送所有 Agent；否则标记为 Reject，仅保留为候选记录不部署。效果：部署的 skill 池单调不降（monotonic deployment behavior），用户始终交互的是前一夜通过验证的最优 skill 池。

Analysis

夜间候选验证详情与类别特异演化模式

Social Interaction 候选详情（Table 4）

6 天共产生 4 个候选，仅 1 个被接受（Night 1）：03_task6——跨部门 Slack 总结 skill，将描述性指令重写为严格有序步骤。其余 3 个全部被拒绝。

结论：验证机制有效防止了过度更新——接受率低（1/4），但单次高影响更新足以带来 +11.72% 的持久提升。

Search & Retrieval 候选详情（Table 5）

Night 1 接受 validate-file-existence skill；Night 3 同池重测确认（Accept）为第二个关键节点；后续 4 个候选（含约束感知检索规划 skill）均被拒绝。

演化路径呈阶梯形：先解底层可靠性（输入验证/文件可访问性），再尝试高层推理提升，但高层改进未通过验证门。

Creative Synthesis 候选详情（Table 6）

仅 validate-tmp-workspace-inputs（Night 1）被接受；后续多个多模态 pipeline skill 多次尝试均未超越已有最优池而被拒绝。

最大收益来自环境预检（工作目录/符号链接），内容生成本身并非瓶颈——揭示了 Creative 任务的真实失败根源。

Safety & Alignment 候选详情（Table 7）

Night 1–4 共接受 3 次更新（git push 认证回退 + clone 目录修正），Night 5–6 改进版本未超越最优池被拒绝，部署池保持单调不降。

Safety 类别的改进聚焦执行鲁棒性而非语义安全——说明程序性 skill 更新对执行稳定性的直接贡献最大。

类别特异（Category-Specific）演化模式

不同类别的 skill 演化驱动力截然不同——不存在单一通用的 skill 更新路径：

  • Social Interaction：流程显式化（将模糊意图转化为严格有序步骤）
  • Search & Retrieval：输入可靠性优先（文件存在性验证、API 配置固化）
  • Creative Synthesis：环境预检（工作目录确认、符号链接修复、多模态 pipeline 初始化）
  • Safety & Alignment：执行鲁棒性（认证回退策略、非交互式环境适配）

Takeaways

核心结论

  1. SkillClaw 将静态 skill 生态转化为持续演化的动态系统，把"用户交互"直接当作 skill 更新的驱动信号，无需额外人工投入。
  2. 夜间验证机制保证单调部署行为：skill 池只升不降，用户始终享受经验证的最优版本。
  3. 演化模式是类别特异的：Social 强调流程显式化，Search 强调输入可靠性，Creative 强调环境预检，Safety 强调执行鲁棒性——无单一通用路径。
  4. 即使在小规模实验（8 用户、6 天）下，集体演化仍展现显著效果，暗示扩展用户数和时长将进一步提升演化质量。

局限性

  • 实验规模较小（8 用户、6 天、60 任务、4 个类别），结果尚不构成大规模部署的充分验证；论文整体标注为 Work in Progress。
  • 评测闭环在自家 WildClawBench，跨 benchmark 泛化性待验证。
  • 夜间验证引入额外 token 成本；跨用户隐私与数据安全（轨迹共享）问题未作讨论。

对"集体沉淀"方向的启发

  1. 集体演化 vs 单用户演化的根本差异：跨用户聚合让同一 skill 在多种上下文下被检验，形成天然对照实验，为稳定演化提供更可靠的证据基础。
  2. "夜间验证门"作为质量锚点：候选 skill 推送与实际部署解耦，通过真实环境执行而非模型自评决定是否采纳，比直接信任 LLM 自我评判更可靠。未来 skill marketplace 可借用类似的"staged rollout + 生产验证"范式。
  3. Skill 是环境信息压缩器而非通用知识库：API 端口、payload 格式、工具名称等 Agent 无法自行发现的领域特定细节才是 skill 的核心价值，为 skill 质量评估标准制定提供直接参考。
  4. 保守编辑原则降低演化风险：默认"局部修改而非重写"、"先读历史再动笔"、"成功会话保护的部分不轻动"——可直接移植到其他 skill 自进化框架。
  5. Skill Marketplace 的工程骨架已现：共享存储层（OSS/S3）、版本化历史（history/v<N>.md）、技能注册表（skill_registry.json）和 SkillHub 分发机制，事实上已构成轻量 skill marketplace 的技术原型。
