Abstract

Agent Skill 生态正急速膨胀——2026 年 2 月已逾 28 万个 Skill，规模增长带来三重挑战：发现困难（用户无法从数十万 Skill 中找到所需工具）、组合困境（LLM Agent 缺乏协调多 Skill 依赖关系与执行顺序的显式机制）、治理成本（分散的第三方 Skill 生态难以保证质量与安全性）。

现有"平铺式"Skill 调用（native flat invocation）在小规模时勉强可用，但面对千级、万级生态时完全失效——信息超载导致 Agent 无法有效利用庞大的 Skill 库，更无法进行跨多 Skill 的结构化编排。

AgentSkillOS 提出两阶段解法：离线用层级能力树（Capability Tree）将海量 Skill 递归组织，在线用有向无环图（DAG）编排管理多 Skill 执行。配套构造了覆盖 5 类 30 项任务的 benchmark，采用 Bradley-Terry 配对评估方案，定量验证了结构化管理的收益随生态规模增大而放大。

Key Insight

结构化组合而非单纯的 Skill 可用性，才是解锁生态潜力的关键。树形检索 + DAG 编排的组合拳不仅优于平铺调用，甚至在 Skill 选择固定的条件下（oracle 集合），有无结构化编排之间的差距依然显著。

Framework Overview

AgentSkillOS 由离线构建阶段（Manage Skills）与在线求解阶段（Solve Tasks）组成，两者通过能力树 $\mathcal{T}$ 和 DAG $G=(V,E)$ 解耦：离线一次建好树，在线动态选 Skill、生成编排计划、并行执行。

能力树构建

基于节点级递归分类算法：LLM 生成 $B \in \{7,12\}$ 个候选类别，将 Skill 分配至最合适的类别，类别不足容量阈值 $C = \lfloor 1.5B \rfloor$ 的节点直接升级为叶节点。根节点固定五大类（内容创作 / 数据处理 / 软件开发 / 自动化 / 垂直领域）。新增 Skill 时自顶向下逐层分配，沿路更新节点描述。

DAG 编排策略

三种策略对应不同优化目标：Quality-First——添加准备与精化阶段，生成深密 DAG，最大化交付质量；Efficiency-First——最大化并行度，宽且浅的 DAG，降低时延；Simplicity-First——保留最小本质 DAG，每个节点均必要，便于人工理解与调试。给定 Skill 集合 $V$ 与策略，LLM 输出 $G=(V,E)$，有向边表示数据流依赖。

超大规模适配

当生态规模达 200K 时，引入 Usage-Frequency Queue：只选最高 $K=10000$ 个高频 Skill 构造活跃树，其余 Skill 进入"休眠索引"，支持向量相似度检索补充。历史任务的 DAG 编排计划存入 recipe pool，可按向量相似度复用，跳过检索与编排两阶段，加速推理。

多 Skill 执行

拓扑排序后，同层无依赖节点并行执行，不同层顺序执行。每个节点的执行提示明确包含：子任务描述、上游产物列表（路径 + 用途）、预期输出规范，生成的文件以"制品"（artifact）形式传递给下游节点。DAG 结构使得多 Skill 间的数据流和控制流均可被显式追踪，为后续 Skill 自进化留下可读日志。

即使给予 oracle Skill 集合，缺少 DAG 编排依然大幅损害质量。编排比检索更关键——这是消融实验的核心结论。

三种 DAG 策略的结构特性

论文通过节点数、边数、最大深度（最长依赖链）、最大宽度（最大并行度）四维分布验证三种策略确实体现了设计目标：

策略深度宽度边密度
Quality-First最大中等最密
Efficiency-First最浅最宽中等
Simplicity-First中等中等最疏

三种策略不是同一个系统的微小变体，而是在编排拓扑上存在本质差异，可依场景灵活选择。

树形检索逼近 Oracle 上限

Quality-First 与 Quality-First (Oracle) 的差距在 200 / 1K / 200K 三个生态规模上均保持稳定且较小，说明树形能力树检索能够高精度地逼近人工指定的最优 Skill 集合。"Creative skill discovery"——即发现非显而易见但功能相关的 Skill——是纯向量检索做不到的。

定性案例对比

三个代表性任务的 Vanilla vs Quality-First 对比：
(1) 太空探索交互网页：Vanilla 产出平面低分辨率页；AgentSkillOS 调用图像生成 + 前端设计 Skill，生成高质量全屏分层滚动效果。
(2) 微积分教学包：Vanilla 仅有静态 matplotlib 图；AgentSkillOS 调用动画 Skill 生成 Manim 动态演进，配套精美 PDF 讲义。
(3) 量子计算演示文稿：Vanilla 文字堆砌无设计；AgentSkillOS 生成 Bloch 球可视化、布局统一的专业幻灯片。
核心观察：性能差异来自 Skill 发现与调用的结构化程度，而非 LLM 底层能力。

Takeaways

核心结论

  1. 能力树 + DAG 是生态规模管理的必要条件：平铺式调用在小规模（200）时尚可应对，但在千级、万级生态中完全失效。
  2. 编排比检索更关键：即使 Skill 选择完美（oracle），缺乏结构化编排也会严重损害质量。
  3. 三种策略各有所长：Quality-First 最大化质量但成本高；Efficiency-First 降低延迟；Simplicity-First 便于人理解——用户可根据场景权衡。
  4. 规模越大，收益越明显：这是 AgentSkillOS 相对于 flat invocation 最核心的竞争优势。

局限性

  • 当前假设 Skill 已被收集到生态库，未涉及自动 Skill 发现与集成（从开源社区发现新 Skill）。
  • 系统侧重一次编排执行，自动修正失败、逐步改进的自进化机制尚未完整实现。
  • Bradley-Terry 评估依赖 LLM judge，可能继承 judge 自身的偏差。

对 Skill 管理与生态发展的启发

  1. 层级分类是基础设施：构建全局能力树是长期必要投资，不应依赖每个用户自行用语义检索摸索。
  2. 跨 Skill 编排的标准化：定义清晰的 Skill 输入输出规范、依赖声明格式，让 LLM 可靠地推理编排计划。
  3. 质量与规模的权衡：Usage-Frequency Queue 动态维护"活跃 Skill 集"，将低频 Skill 隔离为休眠索引，兼顾生态开放性与噪声控制。
  4. 编排计划的复用与演进：历史 DAG 方案（recipe pool）可加速推理，长期可积累"最佳实践 DAG 模板库"。
  5. 可解释评测框架：Bradley-Terry 配对评估给出细粒度排序，避免任意绝对评分，为迭代优化提供清晰信号。
  6. 自进化的入口：Skill 以可读制品形式存储，失败的编排执行日志可被用于自动改进 Skill 定义——为 Skill 沉淀与自进化奠定基础。
