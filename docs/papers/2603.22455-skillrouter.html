SkillRouter: Retrieve-and-Rerank
Skill Selection for LLM Agents at Scale

Baohua Dong, Hangcheng Zhu, et al.
Alibaba Group，杭州

Abstract

随着 LLM Agent 生态成熟，社区 skill 库（工具 / 插件）已达数万乃至十万量级。把所有 skill 塞进 context 不可行——需要 skill routing：给定用户任务，从大规模 skill 池中自动检索最相关的 skill。然而，该问题存在三个核心难点：功能同质性（大量 skill 名称相近但实现细节各异，极难区分）；信息不对称假设被打破（现有系统默认 name + description 已足够，但这一假设从未被系统验证）；现有 benchmark 空白（SkillBench / ToolBench 等评测的是"如何用给定工具"，而非"如何从大池中找到正确工具"）。

本文系统研究上述问题，构建了 ~80K skill、75 条专家验证 query 的评测 benchmark，并提出 SkillRouter：以 SR-Emb-0.6B（bi-encoder）+ SR-Rank-0.6B（cross-encoder）构成 1.2B 的两阶段 retrieve-and-rerank pipeline，在 ~80K skill 池上达到 74.0% Hit@1，优于更大的零样本 8B baseline（68.0%），且可部署于消费级硬件。

Key Insight

Skill body（完整实现文本）才是决定性信号，而非 metadata。移除 body 造成 29–44 pp 灾难性下降；routing 层必须有 body 访问权限，agent 消费端保持 context 节省——这种"Hidden-Body Asymmetry"是架构设计的必要约束，不是便利选择。

Framework Overview

SkillRouter 是紧凑的两阶段 pipeline：第一阶段 bi-encoder 将 ~80K skills 压缩到 top-20 候选，第二阶段 cross-encoder 对候选进行精细重排。两阶段均使用完整 skill 文本（name + description + body）。

Figure 2. SkillRouter 两阶段架构。Stage 1：SR-Emb-0.6B（bi-encoder）将 ~80K skills 压缩到 top-20 候选；Stage 2：SR-Rank-0.6B（cross-encoder）对候选重排，输出最终排序。两阶段均使用完整 skill 文本。

Stage 1 — SR-Emb-0.6B（Bi-Encoder 检索）

基座：Qwen3-Emb-0.6B（decoder-based embedding）。输入格式：Query 侧带 instruct 前缀，Skill 侧为 name | description | body（description 截至 300 token，body 截至 2500 token）。训练数据：37,979 个（query, skill）对，query 由 GPT-4o-mini 基于 skill 内容生成，与评测 benchmark 完全不相交。Hard negative mining 从四类来源各取 4/3/2/1 个负例，并施加三层假负例过滤（名称去重 + trigram Jaccard > 0.6 + 余弦相似度 > 0.92），约过滤掉 10% 负例对。损失函数：in-batch InfoNCE（$\tau = 0.05$）。

Stage 2 — SR-Rank-0.6B（Cross-Encoder 重排）

基座：Qwen3-Reranker-0.6B（decoder-based cross-encoder）。输入格式（flat-full）：对每个 query 将其与 20 个候选 skill 的完整文本拼接输入，description 截至 500 token，body 截至 3000 token。训练数据：32,283 个 query，每个取 SR-Emb-0.6B 检索的 top-20 候选，构成 binary 标注列表（同样施加假负例过滤）。损失函数：Listwise cross-entropy（对候选列表做 softmax + log-prob of positive），比 pointwise BCE 高出 30.7 pp Hit@1。

Hidden-Body Asymmetry：信息不对称设计约束

这是 skill routing 区别于普通文档检索的核心特征。消费者（agent）：受 context 窗口限制，只能看到 skill 的 name + description。路由器（retriever / reranker）：可以访问完整的 body（实现代码 / 详细规范）。此不对称不是设计便利，而是必要约束——routing 层必须有 body 访问权限，才能实现准确选择；agent 消费端则保持轻量。注意力层间演化呈现 body → name → body 模式：早层（0–6）body 占 97.3%，中层（7–20）name 升至峰值 26.3%（语义匹配），晚层（21–27）回归 body 主导（91.7%）。

Analysis

Figure 3. Reranker 贡献分解（共 150 条 query）：Reranker 在 12.7% 的 query 上修复了 encoder 的错误（绿色），仅在 4.0% 上出现降级（红色），净贡献 +8.7 pp Hit@1。22% query 被两阶段均错过——这是 encoder top-20 窗口召回不足或语义鸿沟需要多跳推理导致的系统上限。

Takeaways

核心结论

  1. Skill routing 是真正困难的检索问题：最强 8B base encoder 也只有 64.0% Hit@1，功能同质性使得任何简单方法均难以应对。
  2. Skill body 是决定性信号：移除 body 造成 29–44 pp 灾难性下降，attention 分析证实 91.7% 集中在 body——推翻了"name + description 已足够"的业界假设。
  3. SkillRouter（1.2B）有效且紧凑：74.0% Hit@1，优于更大的零样本基线，可在消费级硬件上部署；同方法扩展到 16B 达到 76.0%。
  4. 任务特定微调优于单纯扩大规模：0.6B 微调超越 8B 零样本，data / training 对此任务比 scale 更重要。

对 Skill 管理与检索方向的启发

  1. Skill 文档质量至关重要：body（实现细节 / 详细规范）的质量直接决定检索精度，社区应重视 skill body 的信息丰富性，而非只关注简短的 name / description。
  2. Routing 层架构设计：大规模 agent 系统应将 routing 组件与 agent 消费端分离；routing 层必须有 body 访问权，agent 消费端保持 context 节省。
  3. 假负例过滤是同质池训练的必要步骤：社区 skill 库功能重叠严重，不过滤假负例会严重损害 contrastive learning 效果。
  4. Listwise ranking loss 对同质候选池是关键：当候选池高度相似时，必须建模候选间相对顺序，pointwise 方案会失效。
  5. 两阶段 pipeline 的互补价值：encoder 提供语义召回广度（R@20 = 75.4%），reranker 基于 body 细粒度 cross-attention 提升精度（Hit@1 净 +8.7 pp），两者缺一不可。

局限性与未来方向

  • 22% query 被两阶段均错过——系统上限来自 encoder top-20 窗口召回不足或语义鸿沟需要多跳推理，是未来重点突破方向。
  • 当前 benchmark 为 75 条 query，规模有限；更大规模、更多领域的评测集仍待构建。
