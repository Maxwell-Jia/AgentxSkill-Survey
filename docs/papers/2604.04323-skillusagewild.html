Abstract

现有 Skill Augmented Agents 论文通常将手工精选、任务专属的 skill 直接灌给 LLM，暗含 skill 总是可得且完美匹配的假设。然而现实部署中，Agent 需要从涵盖数万个通用 skill 的开放 skill 库（如 skillhub.club、skills.sh）中自行检索，检索到的 skill 质量参差、不够精炼。这一"理想 vs 现实"的鸿沟导致现有 benchmark 系统性地高估了 skill 的实际收益。

本文提出第一个渐进式（progressive）的 skill 评测框架，覆盖从理想到现实的四档对照条件，系统量化 skill 在各约束下的真实效能。在 SkillsBench 和 Terminal-Bench 2.0 两个 benchmark 上，作者发现仅检索不精炼时性能改进微乎其微（35.4% → 38.4%），而引入Query-Specific Refinement（任务感知精炼）后可恢复到 48.2%，在 Terminal-Bench 2.0 上更带来 +7.8% 的稳定收益。

核心发现：Skill Degradation Gap

手工精选 skill 下通过率 55.4%；从 34k 库检索后无精炼仅 38.4%，仅比无 skill 基线（35.4%）高 3 个百分点。现实检索场景下 skill 的实际收益大幅缩水。

Key Insight

Skill 的收益是脆弱的（fragile）——检索漂移（retrieval drift）与 skill 不适配（mismatch）两个失效相互叠加，使得"给 Agent 一堆 skill"不如"精心设计精炼流程"。Query-Specific Refinement 是弥合理想-现实差距的关键杠杆。

Framework Overview

论文设计了四档渐进式对照条件，从最简单（理想上界）到最现实（Agent 既要找 skill 又要用 skill），系统拆解 skill 在不同约束下的失效来源。

  1. Curated Skill + Task-Specific（理想上界）
    直接给定手工选中的、针对该任务的 skill。仅用于估计上界，真实部署不可得。通过率约 55.4%。
  2. No Skill Baseline（下界）
    纯 LLM 无 skill 辅助。Claude Opus 4.6：35.4%；Claude Haiku：28.1%；GPT-3.5：22.0%。
  3. Retrieved Skill from 34k Collection（现实条件 1）
    Agent 从包含 34k 真实 skill 的开源 skill hub 中检索后使用，无精炼。Claude Opus 4.6：38.4%，仅比无 skill 高 3 个百分点。
  4. Agent 自主检索 + 自主使用（最难条件 2）
    Agent 既要自行决定是否检索 skill、如何检索，又要自行决定是否使用。即使有相关 skill 可用，Agent 也只在约 49% 的情况下主动加载。

三类链式失效的诊断

消融实验显示性能下降来自三个阶段的链式缺陷，每一环都会进一步折损收益。

Skill 选择失效

Agent 即使面对相关 skill，也只在 49% 的情况下主动加载。
原因：弱模型难以评估 skill 的相关性；通用 skill 标题与任务描述语义距离较大，导致 Agent 错误判断"无 skill 可用"。

Skill 检索失效

从 34k 库中检索相关 skill 的成功率仅 65.5% Recall@5（Agentic Hybrid Search）；纯语义检索仅 38.1% Recall@3。
迭代查询 + 候选评估（Agentic 主动式检索）显著优于直接检索，但仍有大量漏召。

Skill 适配失效

通用 skill 往往误导 Agent 而非帮助（negative transfer）：弱模型（GPT-3.5、Claude Haiku）在面对不适配 skill 时性能反而下降；加入检索 skill 后 GPT-3.5 通过率从 22.0% 降至不稳定。
强模型（Claude Opus 4.6）对无关内容的抗性更强。

两类精炼策略

Query-Specific Refinement（任务感知精炼）

Agent 基于当前任务描述，动态修改 / 筛选 skill 内容，使通用 skill 定制化适配当前查询。
效果最强：Claude Haiku 从 40.1% 恢复到 48.2%；Terminal-Bench 2.0 上 Claude Opus 4.6 从 57.7% 提升到 65.5%（+7.8%）。
前提：初始检索的 skill 质量足够好（Coverage Score ≥ 3.83）。

Query-Agnostic Refinement（通用精炼）

对所有 skill 统一应用精炼规则（如压缩、简化），不感知具体查询。
效果有限：约 +2% 改进。
作用：当初始 skill 质量不足时作为兜底，但当 Coverage Score < 3.0 时，精炼也无法挽救。
结论：精炼本身不生成知识，只是"乘以质量系数"。

  • 仅检索不精炼时，Claude Haiku 虽从 28.1% 升至 40.1%，但 GPT-3.5 提升微弱（22.0% → 24.5%），弱模型更容易被无关 skill 干扰。

  • Terminal-Bench 2.0 上精炼带来 +7.8% 的稳定收益，说明终端交互等复杂场景中精炼同样有效。
  • 当 Coverage Score（skill 与任务的相关度）< 3.0 时，任何精炼策略都无法挽救——检索质量是硬门槛。

Agentic Hybrid Search（Agent 自己尝试多个查询、评估候选）相比直接语义检索有显著提升。Agent 的主动迭代式查询是 skill 检索的关键能力，单纯依赖 BM25 或向量检索在 34k 规模上已不足。

Coverage Score 与精炼效果

Coverage Score 衡量检索到的 skill 与当前任务的相关程度（满分 5 分）：

Coverage ≥ 3.83：Query-Specific Refinement 能有效恢复性能，弥合理想-现实 gap。

Coverage < 3.0：精炼无法挽救——初始 skill 质量是精炼的硬前提，精炼不创造知识，只放大已有 skill 的价值。

结论：精炼策略与检索质量必须协同设计，二者缺一则整体 pipeline 效能大打折扣。

Skill 类别差异

Code-related skills：检索精度最高，因为代码相似性易被向量模型捕捉，函数名 / API 名与查询词汇匹配度高。

Knowledge skills：检索困难，需要精细的语义理解；通用知识库的 skill 与具体任务往往存在语义漂移。

Tool use skills：介于两者之间，工具 API 文档的结构化程度影响检索效果。

模型能力的二阶效应

强模型（Claude Opus 4.6）：对无关 skill 的抗性强，能更好地"忽视"不适配内容，negative transfer 现象不明显。

弱模型（GPT-3.5、Claude Haiku）：面对不适配 skill 时性能明显下降，甚至低于无 skill 基线。弱模型对干扰内容更敏感，skill 管理不能一刀切。

启发：在多模型部署场景下，Skill 的筛选门槛应按模型能力动态调整。

Takeaways

核心结论

  1. Skill 收益是脆弱的（fragile）：现实约束（检索、不精炼）下，skill 的优势大幅缩水，弱模型甚至为负收益。
  2. 检索 + 精炼是必要的组合：单纯给 LLM 一堆 skill 不如精心设计精炼流程；Query-Specific Refinement 是关键乘数。
  3. 现有 benchmark 高估了 skill 的实际效能：许多 skill-augmented 方法在学术评测中表现优异，真实部署时会显著退化。
  4. Agentic 主动检索优于被动检索：Agent 迭代式查询 + 候选评估（Agentic Hybrid Search）显著优于直接语义检索。

局限性

  • 评测聚焦于 SkillsBench 和 Terminal-Bench 2.0，开放性更强的场景（如 web agent、embodied agent）覆盖不足。
  • 精炼策略需要 LLM 参与，引入额外推理开销；尚未分析精炼成本与收益的权衡曲线。
  • 34k skill 池来自特定开源 hub，不代表所有部署场景中的 skill 分布。

对综述 / 后续研究的启发

  1. 未来 Skill 论文应包含三档对照：手工精选 / 检索无精炼 / 检索+精炼，而非只秀手工版本。建议新增"Skill Degradation Ratio"指标衡量理想-现实 gap。
  2. Skill 质量控制与精炼机制缺一不可：不能只是累积 skill 数量，需要同步建立质量控制和精炼 pipeline。
  3. 检索与评测应联动设计：现有向量检索在 34k 规模上已失效，需要更强的检索模型或更聪明的 Agentic hybrid 策略。
  4. Skill 管理生态的三层架构：（a）高质量 skill 池（非随意累积）；（b）智能检索（Agentic Search）；（c）自动精炼（Query-Aware Adaptation）——三层缺一则收益大幅折扣。
  5. 与 GoS（Graph of Skills）的关联：本文揭示的检索漂移问题，可通过依赖感知图检索（GoS）缓解——结构化依赖建模有助于恢复通用 skill 库中的相关候选。
