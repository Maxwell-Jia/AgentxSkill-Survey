Abstract

Skill 生态在过去 136 天内已积累超过 84,192 个公开 skill，向 LLM agent 注入 skill 已成为 SWE 任务中的常见做法——但其实际收益从未被系统量化。现有 SWE benchmark（SWE-Bench Verified、TerminalBench）均未设置"有 skill vs. 无 skill"配对评测；而现有 skill benchmark 中 SWE 任务覆盖极少，且缺乏需求驱动的验证体系。

SWE-Skills-Bench 构建了首个需求驱动（requirement-driven）的 SWE skill 评测基准：将 49 个公开 SWE skill 与真实 GitHub 仓库配对，在有 skill / 无 skill 两条件下用确定性 pytest 单元测试进行配对评测，量化 skill 注入的边际增益（ΔP）与代价（token overhead ρ）。核心发现：约 80% 的 skill（39/49）带来零增益，全局平均 ΔP 仅 +1.2%；仅 7 个 skill 带来有意义增益，3 个 skill 因 context interference 主动降低性能。Token overhead 与正确性增益完全解耦，部分 skill 在不提升通过率的同时带来高达 +450.8% 的推理成本。

Key Insight

Skill 注入不是通用性能提升器。在高度结构化的软件工程领域，基础模型已大量内化相关知识——注入冗余 skill 收效甚微；注入覆盖范围过宽的"综合参考文档"型 skill，反而会通过表面锚定、幻觉、概念渗漏三种机制主动降低性能（Context Interference）。Skill 的价值高度领域特异性，且极度依赖粒度-范围对齐。

Benchmark Construction

SWE-Skills-Bench 的构建围绕"如何在真实 SWE 语境下进行可信的配对评测"展开，覆盖 skill 筛选、仓库配对、需求驱动验证三个核心环节。每个任务实例被定义为四元组 $(R, E, P, S)$：固定 commit 的仓库 $R$、容器化环境 $E$、需求文档 $P$（含 Background / Requirement / File Operations / Acceptance Criteria）以及可选 skill 文档 $S$。

Figure 2. (a) 49 个 skill 在六类 SWE 子域的分布；(b) 每个 skill 生成约 10 个任务实例，全量约 565 个。Deployment & DevOps（13）和 Analytics & Monitoring（12）是最大的两个类别。

Skill 池构建

从 mcpmarket 84,192 个公开 skill 出发，经三轮过滤得到 49 个 SWE skill：

  1. 类别筛选：保留六大与 SWE 流程最相关、可单元测试的类别
  2. 语义过滤：排除生成类或主观类 skill，仅保留针对具体 SWE 动作的 skill
  3. 可行性筛选：排除仓库体量过大或环境搭建成本过高的候选

全部采用标准 SKILL.md 格式（Background / Requirement / File Operations / Acceptance Criteria 四段式），遵循 Claude Code agent skill 规范。

真实仓库配对

每个 skill 对应一个真实开源 GitHub 项目，关键设计：

  • 仓库 pinned 在固定 commit，保证可复现
  • 每个项目构建 Docker 容器（Ubuntu 24.04，CPU-only）
  • 评测前清除 .claude/skills/ 目录防止干扰
  • 注入时将 SKILL.md 拷入 ~/.claude/，无 skill 条件下缺席

需求文档 P 中从不引用 skill S，确保 agent 行为完全由 skill 文件的物理存在驱动。

需求驱动验证

核心理念：将验收准则（Acceptance Criteria）转化为确定性 pytest 测试，从需求到测试全链路追溯：

  • 固定"专业测试工程师"提示词模板
  • 枚举可测行为，构造典型及边界场景
  • pytest 运行实际代码，验证具体输出/结构

全程无 LLM-as-judge，确保评测确定性。

Results

评测指标：Pass Rate（Pass+/Pass-）· Skill Utility Delta（ΔP）· Token Overhead Ratio（ρ）· Cost Efficiency（CE = ΔP / ρ）

指标 数值 说明
全局 Pass-（无 skill） 89.8% 基础模型已有较强基线能力
全局 Pass+（有 skill） 91.0% 加入 skill 后的整体通过率
平均 ΔP（增益） +1.2% 全局边际增益极为有限
平均 token overhead（ρ） +10.5% skill 注入带来的平均推理成本增加
ΔP = 0 的 skill 数 39 / 49（≈ 80%） 绝大多数 skill 带来零增益
ΔP > 0 的 skill 数 7 / 49 有意义增益，+7.1% 至 +30.0%
ΔP < 0 的 skill 数 3 / 49 性能下降，−9.1% 至 −10.0%

最优 Skill：risk-metrics-calculation

将金融风险计算公式（VaR、CVaR、Sharpe Ratio 等）以程序性知识形式结构化编写，提供具体计算步骤而非通用建议。

ΔP = +30.0% ρ = −34.8%

既提升正确率，又减少 token 消耗——"精炼程序知识 + 领域专用 + 无版本冲突"的理想范本。

有效 Skill 的共同特征

  • 编码了高度专业化的程序性知识（金融风险公式、云原生流量管理、GitLab CI 模式）
  • 知识是 agent 预训练中覆盖不足的领域
  • 无版本错配或范围冲突

tdd-workflow（ΔP = +7.1%）的机制是"清单效应"——强迫 agent 逐一检查容易遗漏的交付物，代价是 ρ = +78.6%（CE 极低）。

  • 39 个 ΔP = 0 的 skill 中，24 个是 Pass+ = Pass- = 100%（skill 完全冗余，基础模型已内化）；15 个两条件均未达 100%（瓶颈在于复杂多步推理，而非领域知识缺失）。
  • Token overhead 与正确性完全解耦：ρ 范围从 −77.6%（python-resilience）到 +450.8%（service-mesh-observability），却均对应 ΔP = 0。Skill 改变的是推理路径，而非必然改变推理结果。
  • 3 个 skill 因 context interference 主动降低性能：springboot-tdd（−10.0%）、linkerd-patterns（−9.1%）、django-patterns（−9.1%）。

Analysis

失败模式分类

失败类型ΔP 表现根因
能力瓶颈 = 0，Pass- 仍低 多步推理 / API 理解不足，skill 无法弥补
Skill 冗余 = 0，Pass- = 100% 基础模型已内化该领域知识
Context Interference < 0 Skill 覆盖范围与任务粒度错配

Context Interference 通过三种机制生效：Surface Anchoring（照抄模板错误参数）→ Hallucination（试图调和模板与任务，杜撰不存在字段）→ Concept Bleed（相邻概念污染输出）。

Context Interference 案例：linkerd-patterns

任务：生成 Server CRD + ServerAuthorization CRD 以强制 gRPC 服务的 mTLS 身份验证。Skill 含 7 个 Linkerd 模板，Template 5 近似匹配但使用 v1beta1（任务需要 v1beta3）。

  1. Surface Anchoring：照抄 v1beta1 和 HTTP/1，未适配 gRPC
  2. Hallucination：杜撰 rules/metricsServers 字段（CRD 中不存在）
  3. Concept Bleed：混入 NetworkPolicy，生成任务未要求的 K8s 资源

无 skill 时：Agent 从第一性原理出发，正确选择 v1beta3 + gRPC + 标准 meshTLS。

Figure 5. linkerd-patterns skill 导致 context interference 的详细三阶段分析：Surface Anchoring → Hallucination → Concept Bleed。展示了客观上相关的 skill 如何因粒度错配而主动降低 agent 性能。

Token Overhead 分布

ρ 与 ΔP 之间无系统性相关：

  • 最低：python-resilience，ρ = −77.6%
  • 最高：service-mesh-observability，ρ = +450.8%
  • 两者均对应 ΔP = 0

Skill 影响的是 agent 的推理路径，不必然影响最终正确性——token overhead 是 skill 固有属性，与增益正交。

与相关工作的对比

  • SkillsBench（2602.12670）：通用领域 curated skill 上 +16.2pp；SWE-Skills-Bench 全局仅 +1.2%——模型已大量见过的领域，skill 收益趋近于零
  • SkillUsageWild（2604.04323）：关注检索阶段失效；本文关注注入阶段失效——两者合描述 skill 全链路失效图谱

Takeaways

核心结论

  1. Skill 注入不是通用性能提升器：80% 的 skill 带来零增益，平均增益仅 +1.2%，远低于生态快速采用所暗示的收益预期。
  2. Skill 的价值高度领域特异性：仅在编码了高专业性程序知识、且与目标项目无版本/范围冲突时才有意义——最高可达 +30%。
  3. Token 开销与正确性完全解耦：skill 改变推理路径但不必然改变结果；部分 skill 在不提升通过率的同时带来 +451% token 增加。
  4. Context Interference 是结构性风险：版本错配、范围过宽的综合性 skill 通过表面锚定、幻觉、概念渗漏三种机制主动降低性能。

对 Skill 设计的实操指导

不要用生产级综合模板（opinionated, hard-coded values）作为 skill 内容——覆盖范围与任务粒度的错配是 context interference 的直接来源。应当使用抽象指导模式，只编码模型预训练中覆盖不足的高专业性程序知识。这与 SkillsBench 发现的"精简 Skill 优于全面文档"从不同维度共同支持"精炼优于全面"的设计原则。

局限性

  • 当前仅测试 Claude Code + Claude Haiku 4.5 单一 agent 配置，多模型扩展为后续工作。
  • 49 个 skill 来自公开生态，不一定代表最优 skill 质量上限。
  • Skill 对其他 LLM（GPT、Gemini）的边际增益差异尚未量化。

对 Skill 评测基建的启发

  1. Deterministic + requirement-driven 双保证：将验收准则与 pytest 测试全链路绑定，避免 LLM-as-judge 的不确定性——这一设计对评测基建子方向有直接参考价值。
  2. 配对实验设计（有 skill vs. 无 skill）是量化 skill 边际增益的必要条件，不应只报告加入 skill 后的绝对性能。
  3. 检索漂移（SkillUsageWild）+ 注入干扰（本文）+ 能力瓶颈（本文）= skill 全链路失效的完整图谱，为综述框架提供完整分析视角。
