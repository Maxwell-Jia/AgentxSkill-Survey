Abstract

现有 Agent Skill 方案面临两类根本缺陷：纯参数知识生成的 Skill 不含任务域特有的陷阱信息，效果基本等同于"无 Skill"；而在线顺序更新（AutoSkill、XSkill 等）会造成 Skill 碎片化与顺序漂移——早期低质量更新扭曲整体方向，且单条轨迹视野有限，难以归纳通用规律。

此外，人工编写的高质量 Skill（如 Anthropic 官方 xlsx skill）存在跨模型尺度不可移植的问题——同一份 Skill 对 122B 模型提升 +20 pp，对 35B 模型却产生 −9 pp 负效果。

Trace2Skill 提出一个三阶段无参数更新 pipeline：轨迹生成 → 并行多 Agent Patch 提案 → 无冲突层次化归并，像人类专家撰写 SOP 一样，先广泛积累执行经验，再通过归纳推理将局部知识整合为单一、全面、无冲突的 Skill 目录。35B 开源模型撰写的 Skill 可让 122B agent 在 WikiTQ 上提升 +57.65 pp，并行整合比顺序更新快 20×。

Key Insight

全局并行归纳优于局部顺序更新。先积累大量经验，再统一归纳（类似人类专家写作 SOP），而非在线逐条更新——效果更好、速度更快，且无需检索模块。任务执行能力 ≠ Skill 创作能力，归纳推理能力是独立的核心维度。

Framework Overview

Trace2Skill 不更新模型参数，通过三阶段 pipeline 将执行轨迹蒸馏为结构化 Skill 目录 $S = (\mathcal{M}, \mathcal{R})$，其中 $\mathcal{M}$ 为 SKILL.md（自然语言 SOP），$\mathcal{R}$ 包含可执行脚本、领域参考文件和辅助资产。

Figure 2. Trace2Skill 三阶段 Pipeline：Stage 1 以初始 Skill $S_0$ 让冻结 Agent 并行生成轨迹池 $\mathcal{T}$；Stage 2 并行多 Agent 独立分析各条轨迹并提出 Patch $p_i$；Stage 3 将全部 Patch 层次化归并为最终 Skill $S^*$。

Stage 1：轨迹生成

以 ReAct 为框架，将初始 Skill $S_0$ 预置到系统提示中，让冻结 Agent $\pi_\theta$ 在演化集 $D_{\text{evolve}}$ 上并行运行，产生轨迹集合 $\mathcal{T}$，按正确性划分为 $\mathcal{T}^-$（失败）与 $\mathcal{T}^+$（成功）。实践规模：200 条轨迹、50+ 步/条，122B 模型 <2 GPU 小时。

Stage 2：并行多 Agent Patch 提案

每条轨迹独立分配给一个分析 Agent，读取 $S_0$ 的冻结副本，输出 Patch $p_i$。 Error Analyst（$A^-$）：多轮 ReAct，可检查 trace、读取文件、对比 GT，迭代缩小根因后才提出 Patch——质量门控。 Success Analyst（$A^+$）：单轮调用识别可泛化成功模式。所有 Analyst 彼此不可见，保留多样性。

Stage 3：无冲突层次化归并

对 Patch 池 $P = P^- \cup P^+$ 执行层次化归并：$$p^{(\ell+1)} = M(\pi_\theta, S_0, \{p_1^{(\ell)}, \ldots, p_B^{(\ell)}\})$$ 归并层数 $L = \lceil\log_{B_{\text{merge}}}|P|\rceil$，批次大小 $B_{\text{merge}} = 32$。Merge Operator 明确识别"跨 Patch 高频模式"并提升为通用 SOP；低频特例自动归入 references/ 子目录。三重确定性防护防止幻觉。

两种演化模式

Skill Deepening（深化）：以人类专家编写的高质量 Skill 为 $S_0$，轨迹分析进一步精炼其薄弱之处。 Skill Creation（从零创建）：以 LLM 参数知识草稿为 $S_0$，完全由轨迹驱动构建有效 Skill。两种模式均在实验中验证有效，且均能跨模型尺度迁移。

归并即归纳推理：出现频率低的 Patch 视为特例丢弃，高频 Patch 被提升为通用 SOP——这是 Skill 获得可迁移性的核心机制，自动复现了人类专家分层撰写 SOP 的惯例（主文 SKILL.md + 特例 references/）。

Takeaways

核心结论

  1. 经验可蒸馏为可迁移 Skill：轨迹局部知识经过并行归纳后，能跨模型尺度（35B 写的 Skill 帮助 122B 提升 +57.65 pp）和跨任务域（电子表格 Skill → WikiTQ）发挥作用。
  2. 全局并行优于局部顺序：先积累广泛经验、再统一归纳（类似人类专家工作流），效果更好、速度快 20×。
  3. 单一紧凑 Skill 优于检索记忆库：无需检索模块，推理时无检索损耗，归纳过滤后更抽象、更通用。
  4. 开源小模型足够：35B 开源模型可完成完整的 Skill 演化流程，不依赖专有大模型。
  5. 任务执行能力 ≠ Skill 创作能力：后者依赖对失败模式的归纳推理，是独立维度的能力。

局限性

  • 无法量化单条 Patch 的因果贡献，也无法追踪推理时各 Skill 段落的使用频率——这是下一步 Skill 自动裁剪与优化的关键入口。
  • 当前 Skill 结构为声明式文档，尚不支持在执行轨迹反馈下的持续在线更新。

对 Skill 自动沉淀方向的启发

  1. 批量离线 > 在线单条：单条轨迹信噪比低，批量并行分析后归纳才能将"局部经验"升华为"通用规律"。
  2. Declarative Skill as Knowledge Artifact：将 Skill 定义为可读、可共享的声明式文档（非检索索引、非 LoRA 权重），使其天然可跨 Agent 框架移植。
  3. Agentic 根因分析是瓶颈：Error Analyst 的质量直接决定 Patch 的可迁移性，值得重点投入。
  4. 层次化 Skill 结构：SKILL.md（通用 SOP）+ references/（特例）的两层设计值得在其他 Skill 系统中采纳。
