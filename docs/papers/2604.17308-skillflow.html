Abstract

现有 Skill benchmark（SkillsBench、SWE-Skills-Bench、SkillUsageWild）均预设了一个静态的已有 Skill 库，验证"外部 Skill 能否帮助模型"，却从未检验：Agent 能否从零开始，在序列化任务执行中自主发现、提炼、修复 Skill，并在后续任务中真正复用？这正是 lifelong 视角的核心缺口。

SkillFlow 构建了包含 166 个任务、20 个工作流族的终身 Skill 评测基准，覆盖 Finance、Operations、Healthcare、Governance、Data & Document 五大领域。每个任务族共享一套 Domain-Agnostic Execution Flow（DAEF）抽象工作流骨架，保证族内技能迁移具有结构性基础。评测协议规定 Agent 从空 Skill 库出发，通过轨迹驱动（trajectory-driven）与 Rubric 驱动（rubric-driven）两种 Skill Patch 机制在任务序列中增量积累知识，实现"发现→使用→修复→迁移"全生命周期测量。

在 11 个模型、4 个 harness 的评测中，Claude Opus 4.6 在完整终身演化协议下达到 71.08% 完成率，较 Vanilla 提升 +8.43 pts，同时成本下降 7.52%、输出 Token 减少 20.33%——效果与效率双优。

Key Insight

Agents begin with no skills. SkillFlow 是目前唯一真正 lifelong 的 Skill 评测框架：每族任务从空库起步，trajectory- 与 rubric-driven skill patches 共同驱动知识外化，序列化执行使 Skill 在任务间积累与修复。"能否写 Skill"远不是瓶颈——能否识别并修复错误 Skill，才是正向迁移的真正分水岭。

Benchmark Construction

SkillFlow 的构建核心是保证族内任务共享抽象工作流结构，从而使 Skill 迁移信号可靠且可测量。任务族从 SkillsBench（18 个）和 GDPval（46 个）中遴选 64 个种子任务，通过嵌入式语义匹配与双 Agent 迭代扩展（GPT-5.3-Codex 为 Architect Agent，Claude Opus 4.6 为 Critic Agent，在真实 Docker 环境中验证），最终生成 166 个可运行任务，全部通过人工审查，无致命环境错误。

Figure 2. 任务构建四步流水线：种子收集 + Skill 池整理 → 语义匹配配对 → 双 Agent 迭代扩展与 Docker 验证 → 人工审查（指令泄露 / 逻辑合理性 / 环境正确性 / 难度梯度）。

任务族划分

  • 20 个工作流族，每族 8–9 个任务，按难度升序排列
  • 覆盖五大领域：Finance & Economics · Operations & Supply Chain · Healthcare & Life Sciences · Governance & Strategy · Data & Document Intelligence
  • 族间重置 Skill 库，避免异质工作流信号相互干扰

DAEF 工作流骨架

每族共享 Domain-Agnostic Execution Flow（DAEF）：形式化为有向图，节点为受控词汇操作类型（read / retrieve / compute / detect / output），边为依赖关系。

DAEF = $\phi(\mathcal{T})$，去掉具体实体后保留的抽象图，可在金融、医疗、运营等不同领域实例化，是族内 Skill 迁移的结构性保证。

Skill Patch 协议

Patch 包含三个最小可审计字段：

  • summary：高层次经验总结
  • upsert_files：创建或覆盖 SKILL.md 及辅助脚本
  • delete_paths：移除过时 Skill 文件

轨迹驱动从行为轨迹 $\tau_t$ 归纳做法；Rubric 驱动从验证器反馈 $r_t$ 修复逻辑——两者互补。

Figure 3. 相同 DAEF（抽象工作流节点与依赖结构相同）在金融分析、医疗合规、运营规划等不同领域的实例化示例，验证跨领域工作流迁移的可行性。

Agentic Lifelong Learning 协议

每族评测开始时，Agent Skill 库为空 $\mathcal{S}_0 = \emptyset$。第一个任务在无任何 Skill 辅助下完成，产生执行轨迹 $\tau_1$ 与 Rubric $r_1$，从零引导生成初始 Skill：

$\Delta_1 = \text{Model}_g(\emptyset,\, \tau_1,\, r_1), \quad \mathcal{S}_1 = \text{Apply}(\Delta_1,\, \emptyset)$

对后续任务 $t > 1$，Agent 携带 $\mathcal{S}_{t-1}$ 执行 $T_t$，获得 $(\tau_t, r_t)$ 后增量更新：

$\Delta_t = \text{Model}_g(\mathcal{S}_{t-1},\, \tau_t,\, r_t), \quad \mathcal{S}_t = \text{Apply}(\Delta_t,\, \mathcal{S}_{t-1})$

Skill 使用为自然行为（Agent 自主决定是否读取已有 Skill），评测从轨迹中提取读/调用事件，计算 Skill 复用率（%use），不干预 Agent 决策。

Analysis

Figure 6. 按行（模型）× 列（领域组：FE / OS / HLS / GS / DDI）展示 Skill Evolution 增益分布。Claude Opus 4.6 在多领域均呈绿色正增益；Data & Document Intelligence 跨模型正迁移最稳定；Finance & Economics 中存在较多负迁移案例。

Finding 1 & 3：紧凑 Skill 库 vs. Skill 膨胀

Opus 4.6 是唯一能将 Skill 修订转化为库级持续改善的模型，平均保留 1.05 个核心 Skill。Qwen 等弱模型几乎为每个任务单独生成新 Skill（平均 5.45 个），Skill 数量单调增长但收益更差——失败原因是碎片化而非 Skill 缺失，即 Skill Inflation 问题。

Finding 2 & 6：错误 Skill 的传播危害

一旦错误抽象写入 Skill 库，后续任务会继承同一缺陷逻辑，将局部错误放大为序列级负迁移。能否识别并修复错误 Skill 才是正向迁移的分水岭——几乎所有模型都能写出某种 Skill，但能否在后续任务中有效修订，区分了强弱模型。

Codex 在变体任务合并上更倾向形成单一演化核心（Skill 库文件数量相对紧凑），但端到端完成率不及 Opus 4.6，说明 Skill 压缩能力 ≠ 执行能力。两者是相对独立的维度，均需在 lifelong 评测中分开测量。

Takeaways

核心结论

SkillFlow 表明，终身 Skill 演化带来的收益是选择性而非普遍的：能力更强的模型（Opus 4.6）将外化经验转化为紧凑、可修复的程序性 Skill，实现持续增益；弱模型则因 Skill Inflation、错误强化和创建-复用协调失败而出现退化。Skill 修复能力是区分强弱的核心维度，而非 Skill 生成量。

对 Lifelong Skill 评测的启发

  1. 现有评测体系的唯一缺失维度——SkillsBench 回答"有好 Skill 时能帮多少"，SWE-Skills-Bench 聚焦软工场景，SkillFlow 是目前唯一回答"Agent 能否自主在时间中积累并修复 Skill"的工作，填补了 Skill 生命周期评测的最后一环。
  2. Trajectory patch 与 Rubric patch 的互补性——轨迹驱动从"How"归纳做法，Rubric 驱动从"What went wrong"修复逻辑。设计失败复用系统时，应同时接入这两类信号，而非只依赖成功轨迹。
  3. Skill 库的紧凑性比数量更重要——+8.43 pts 的最佳模型平均只保留 1.05 个 Skill。需要主动的 Skill 合并、去冗余与淘汰机制，而非无限积累。
  4. 错误 Skill 的传播比缺少 Skill 更危险——写入前的质量校验（类似 Leakage Audit）和写入后的在线检测机制都是必要的。
  5. 高使用率 ≠ 高效用——Kimi K2.5 的 66.87% 使用率搭配 +0.60 pts 增益，揭示了"使用但无实质收益"的陷阱。评测指标应将 Skill 使用率与任务成功率的相关性纳入体系。
  6. DAEF 作为 Skill 迁移的结构化脚手架——显式定义工作流抽象骨架约束任务族构建，可避免"任务相似性带来的伪迁移"，使评测结论更可靠，是 benchmark 设计的可推广原则。

局限性

  • 评测聚焦族内 lifelong 学习，每族起始重置 Skill 库，暂未评测跨异质工作流的 Skill 泛化。
  • Skill Patch 生成与 Agent 执行 harness 解耦，但 Patch 质量仍受模型指令遵循能力影响。
  • 当前协议中 Skill 库规模监控依赖 #Skills 与 %use 两个代理指标，尚无直接度量 Skill 质量演化的评测维度。
