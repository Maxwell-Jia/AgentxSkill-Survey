Abstract

以 Claude Code、OpenHands、Codex 为代表的编码 Agent 虽具备通用问题求解能力，但缺乏特定领域任务所需的专业知识。现有技能绝大多数依赖人工制作，扩展性极差；而 AlphaEvolve、GEPA 等进化方法直接在 prompt 或代码等低层制件上进行优化，与特定任务和模型强绑定，无法自然产生可跨场景复用的组件。

EvoSkill 提出在技能抽象层（而非 prompt/代码层）进行进化搜索：通过三个协作 Agent（Executor、Proposer、Skill-Builder）的迭代循环，自动分析执行失败，发现并精炼结构化可复用的 Agent 技能（遵循 agentskills.io 规范）。底层模型始终冻结，仅技能库和 Agent 元数据在迭代中积累演化。在 OfficeQA 上精确匹配准确率从 60.6% 提升至 67.9%（+7.3%）；在 SealQA 上从 26.6% 跃升至 38.7%（+12.1%）；发现的核心技能零样本迁移到 BrowseComp 仍带来 +5.3% 提升。

Key Insight

优化粒度决定可迁移性：在技能层（skill level）进化，比在 prompt 层或代码层产生更可解读、可组合、可迁移的制件——结构化技能文件夹天然携带触发条件，与具体模型和任务解耦。

Framework Overview

EvoSkill 将每个 Agent Program 定义为"系统提示 + 积累技能库"，存储为 git 分支，使性能差异可归因于技能变化。Pareto 前沿 $G$（容量 $k=3$）维护当前最优程序集合，通过验证集过滤防止进化退化。

Executor

在当前 Agent Program（含技能库）下执行任务，收集得分低于阈值的失败样本集合 $F$。对代码仓库只读，不写入任何内容。

Proposer

分析失败 trace 与累积反馈历史 $H$，执行结构化根因分析（trace review → gap analysis → 现有技能审计 → 技能识别），输出 CREATE 或 EDIT 提案 $\pi$。历史 $H$ 防止冗余提案。

Skill-Builder

将 Proposer 提案具体化为完整技能文件夹（SKILL.md + 可选 Python/TS 脚本）。通过读取元技能 skill-creator/SKILL.md 了解最佳实践，实现"技能创造技能"的自举。

迭代循环（Algorithm 1）

  1. 数据划分：LLM 将数据集分类到 $K$ 个类别，分层切分为训练集（失败检测）、验证集（前沿选择）、测试集（不参与演化）。
  2. 程序选择：从 Pareto 前沿 $G$ 以轮询方式选取父程序 $p$。
  3. 失败采集：Executor 对训练批次运行，使用多容忍度加权评分（$\tau \in \{0.00, 0.01, 0.025, 0.05, 0.10\}$，权重 $w(\tau)=\frac{1}{1+20\tau}$ 偏向严格阈值），加权分低于 0.8 的样本标记为失败集合 $F$。
  4. 提案生成：Proposer 接收 $F$ 和历史 $H$，输出提案 $\pi$（CREATE 或 EDIT）。
  5. 技能物化：Skill-Builder 将 $\pi$ 实现为候选程序 $\tilde{p}$。
  6. 前沿更新：在验证集 $V$ 评估 $\tilde{p}$；若分数超过前沿最弱成员则替换入 $G$，否则丢弃。将结果追加到历史 $H$。
  7. 早停：连续若干迭代无前沿改进时停止；$T$ 轮后返回 $G$ 中最优程序。

Analysis

训练数据量的影响

5% → 10% 训练数据：精确匹配显著提升（+2.4 pp）。10% → 15%：性能轻微下滑（64.5% < 65.8%），提示约 10% 为最优训练规模，更多数据可能导致过拟合或分布偏移。该规律在所有容忍阈值上保持一致。EvoSkill 在极小数据下仍有效，大幅降低标注成本。

Skill Merge 的互补性

多次独立运行各自针对不同失败模式发现不同技能，合并后（merge-unique）精确匹配达 68.1%，超过任何单次运行最优（65.8%）。表明多次独立进化的技能具有功能互补性，不相互覆盖——类似 ensemble 集成策略，可作为提升技能库上限的实用方法。

为何零样本迁移有效？

技能是结构化、可解读的文件夹，带有显式触发条件，与训练任务和底层模型解耦。相比之下，优化 prompt 迁移往往因模型或任务分布偏移而退化；技能作为更高抽象单位具备更好的迁移性。当前迁移实验仅覆盖 SealQA → BrowseComp 一对任务，更广泛验证留待未来工作。

Pareto 前沿治理与失败评分

前沿固定容量 $k=3$，用 git tag（前缀 frontier/）追踪；设置早停机制防止无效迭代。失败检测使用多容忍度加权评分，权重 $w(\tau)=\frac{1}{1+20\tau}$ 偏向严格阈值（$\tau=0$），确保进化方向朝精确匹配优化，避免偏向宽松容忍度下的虚假提升。

Takeaways

核心结论

  1. EvoSkill 在两个截然不同的 benchmark 上均取得显著提升，且仅使用极小训练子集（5%–15%），验证了技能层进化的高效性。
  2. 在技能层进行优化比在 prompt / 代码层优化产生更可解读、可组合、可迁移的制件。
  3. 技能进化是持续积累的过程——技能库不断扩展，Agent 能力逐步增强，底层模型不需要更新。
  4. Skill Merge 类似 ensemble 策略，多次独立运化后合并唯一技能，可进一步提升性能上限。

局限性

  • 零样本迁移目前仅覆盖一对任务（SealQA → BrowseComp），更广泛的跨域验证留待未来工作。
  • 实验底层模型固定为 Claude Code Opus 4.5，跨模型家族的泛化性尚未评估。
  • 技能格式遵循 agentskills.io 规范，与规范深度耦合，换用其他 Agent 框架需适配。

对后续研究的启发

  1. 抽象层次决定迁移性：prompt-level → code-level → skill-level，粒度越粗迁移性越强，EvoSkill 用自动化解决了"发现难度"问题。
  2. 失败驱动是有效信号：利用有标签失败样本做根因分析，比无监督自我进化更高效，类比监督学习中的误差分析。
  3. Proposer 的反馈历史 $H$：防止冗余提案、引导探索"曾部分奏效"方向——这是跨迭代知识积累的关键设计。
  4. 技能市场（Skill Marketplace）的潜力：一个任务上进化出的技能可被其他 Agent 和用户浏览、组合、复用，形成开放技能生态。
