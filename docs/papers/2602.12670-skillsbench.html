Abstract

Agent Skills 生态系统正快速增长（GitHub / Claude Code 生态合计 47K+ Skills），然而该领域缺乏系统化评测框架——既不清楚 Skill 是否真正帮助 agent 完成任务，也不清楚哪些设计决策（指令结构、资源数量、细节程度）驱动了性能提升。现有 Agent benchmark（SWE-Bench、WebArena 等）只评测"模型无辅助能力"，而非"Skill 的边际贡献"。

SkillsBench 构建了首个以 Skill 为一等公民的评测框架：在 105 位贡献者提交的 322 个候选中精选 84 个任务，覆盖 11 个领域，采用三条件（无 Skill / Curated Skill / Self-generated Skill）× 7 个 LLM-Agent 组合（含 Claude Code、Gemini CLI、Codex CLI 搭配 GPT-5.2、Claude Opus/Sonnet/Haiku 4.5、Gemini 3）进行评测，生成 7,308 条轨迹。核心发现：Curated Skill 平均带来 +16.2pp 通过率提升，而 Self-generated Skill 平均下降 –1.3pp，揭示了自动 Skill 合成的关键失效机制。

Key Insight

Skill 的价值高度依赖于质量与设计，而非数量——精选的 2–3 个精简 Skill 可带来 +18.6pp 提升，而让模型自行生成 Skill 平均无效（–1.3pp）。Skill 效能并非来自更多上下文，而来自结构化的程序性指导。

Framework Overview

SkillsBench 定义 Skill 为满足四项标准的结构化文件包（SKILL.md + 可选 /scripts / /references），并搭建覆盖"数据聚合 → 质量过滤 → 三条件评测执行"的完整 pipeline，以 Normalized Gain $g = (\text{pass}_\text{skill} - \text{pass}_\text{vanilla}) / (1 - \text{pass}_\text{vanilla})$ 作为核心效能指标。

Figure 2. 数据聚合（47.15K 个 Skill 来自开源 + Claude Code 生态 + 企业合作）→ 质量过滤（自动检查 + Leakage Audit + 人工审查）→ 评测执行（三条件 × 7 配置 × 5 次重复）。

Skill 定义四标准

程序内容（包含"如何做"的流程 / SOP）、任务类通用性（面向一类问题而非单例）、结构化组件（SKILL.md + 可选资源）、可移植性（文件系统形式，易于版本控制）。显式排除：系统提示词 / Few-shot 例 / RAG 检索 / 工具文档。

三条件评测协议

No Skill：仅提供 instruction.md，环境中无 Skill。
Curated Skill：完整的 environment/skills/ 目录（代码片、例子、资源）。
Self-Generated Skill：无预设 Skill，要求 agent 先自行生成程序知识再求解。

质量保证机制

自动化检查：结构验证 + Oracle 执行（参考解 100% 通过率）+ AI 检测（GPTZero）。
Leakage Audit：CI 中 Claude Agent 检测 Skill 是否含任务特定解法（任务标识符、精确命令序列等）。
人工审查：5 项标准多轮核查，录取率 26.7%。

Analysis

规律：预训练覆盖不足的专业领域（临床数据协调、工业制造流程）收益最大；模型已掌握常见范式的领域（数学、软工）收益微弱。

过多 Skill 引入认知开销或冲突指导；冗长文档消耗上下文预算，关键信息反而难以提取。模型需要专注的程序指导而非穷举文档。

Harness 差异

Claude Code：Skill 利用率最高，所有 Claude 模型一致受益（+13.9pp 至 +23.3pp）。
Gemini CLI：原始性能最高（Flash 48.7%），改进 +13.6pp 至 +17.4pp。
Codex CLI：竞争力强（~44.7%），但频繁忽视 Skill——agent 常承认 Skill 存在却独立实现解决方案，说明 Skill 集成设计是效能瓶颈，与 Skill 质量同等重要。

Self-Generated Skill 失效机制

轨迹分析揭示两种失败模式：
1. 程序不精确：模型识别到需要领域知识，但生成内容过于泛化（如"用 pandas 处理数据"而无具体 API 模式）。
2. 需求盲区：高领域知识任务（制造业、金融）中，模型未意识到需要专用 Skill，直接用通用方法，导致 GPT-5.2 最终 –5.6pp。
→ Self-generated 实验同时控制了上下文长度，说明 失效根源是结构而非长度。

Takeaways

核心结论

  1. 精心策划的 Curated Skill 平均带来 +16.2pp 通过率提升，在 Healthcare 等专业领域可高达 +51.9pp。
  2. Self-generated Skill 平均 –1.3pp，打破"让 LLM 自动生成 Skill"的直觉假设——Skill 合成需要人工演示或多轨迹归纳。
  3. 2–3 个精简 Skill（+18.8pp） 优于 4+ 个或冗长文档（–2.9pp）；聚焦的程序指导比穷举文档更有效。
  4. 小模型 + Skill 可接近大模型无 Skill 的水平（Haiku 4.5 + Skill 27.7% ≈ Opus 4.5 无 Skill 25.1%）。
  5. Skill 效能不仅取决于质量，还依赖 agent harness 的 Skill 集成能力——这是常被忽视的实现瓶颈。

局限性

  • 评测域限制：聚焦终端命令行 / 容器化任务，结论可能不直接转移至 GUI agent、多智能体协作或超长期任务。
  • 因果属性归因：Skill 注入增加上下文长度；虽然 Self-generated 实验部分控制了长度，仍需更强的匹配基线。
  • 覆盖范围：多模态 / 视觉-语言 agent 在 GUI 环境下的 Skill 评估仍是空白。

对后续研究的启发

  1. 三条件对比是必要的评测范式——只比较"有 Skill vs. 无 Skill"无法区分结构价值与上下文长度效应。
  2. Skill 合成需要：人工演示引导 / 从多成功轨迹归纳 / 结合执行反馈迭代细化，单纯 LLM 提示不够。
  3. Leakage Audit 与 Oracle 检查是 Skill benchmark 的必要基础设施，不可省略。
  4. 领域异质性强，通用 Skill 合成算法须考虑 per-domain / per-task-family 适配。
